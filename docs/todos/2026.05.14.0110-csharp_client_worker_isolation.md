# TODO

## Title
CSharpClient worker isolation and Roslyn API validation.

## Problem
A future CSharpClient should provide a PythonClient-like runtime for C# user code, but direct in-process Roslyn execution cannot reliably stop user-created threads, timers, tasks, static event subscriptions, native calls, or blocking work.

## Impact
Without a hard isolation boundary, a stopped C# client can leave runtime resources behind and keep the host process alive or unstable. Simple text scanning for patterns such as `new Thread` would also miss aliases, fully qualified names, indirect APIs, reflection, external libraries, and native interop.

## Suggested Fix
Use a separate C# worker process as the hard lifecycle boundary. Add Roslyn validation before compilation to reject disallowed APIs such as `System.Threading.Thread`, `ThreadPool`, direct `Task.Run`, `Task.Factory.StartNew`, direct timers, `Process.Start`, `DllImport`, `NativeLibrary`, `unsafe`, and risky reflection entry points. Provide a narrow approved runtime API for cooperative lifecycle management, for example managed tasks, managed timers, cancellation tokens, value publishing, logging, and registered disposables.

## Priority
Medium
