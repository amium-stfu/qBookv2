# TODO

## Title
Fix ItemClient echo handling for event-driven writes to published items.

## Problem
When a Monitor event action writes a value to an item that is also published through ItemClient, the ItemClient can receive an echo or stale broker state and write the old value back. This makes Monitor `SetValue` actions appear to oscillate even when the Monitor rule target is not part of the rule condition.

## Impact
Event-driven writes such as `OnActivated -> SetValue` and `OnCleared -> SetValue` are unreliable for published ItemClient items. This can cause confusing value jumps and makes Monitor actions look unstable although the Monitor transition logic itself is stable.

## Suggested Fix
Investigate ItemClient publish/write-back echo detection for locally originated event writes. Ensure the ItemClient can distinguish its own recent local writes from external broker updates and does not overwrite them with stale values. Add a focused regression test around a locally published writable item that is updated by an event action.

## Priority
High
