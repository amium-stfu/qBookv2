# DEBUG REPORT

## Problem
The `Functions` widget row state remains `Ready` when the same function is started through a Button Interaction Rule using `RunFunction`.

## Expected Behavior
When any caller starts a declarative catalog function, the matching row in an already visible `Functions` widget should change from `Ready` to `Running` and expose `Stop`.

## Actual Behavior
The row updates reliably when the function is started from the `Functions` widget itself, but may stay `Ready` when the function is started by another item through an Interaction Rule.

## Error Messages / Logs
No build or runtime error is required to explain the behavior. The issue is caused by missing state notification/polling after an external start.

## Relevant Code
- `src/HornetStudio.Editor/Models/PageItemModel.cs`
  - `RunningDeclarativeFunctions` is a static dictionary, so the running execution is not stored directly on the `Functions` widget row.
  - `TryStartRunDeclarativeFunction(...)` adds the running execution to `RunningDeclarativeFunctions`.
  - `IsCatalogFunctionRunning(...)` queries `RunningDeclarativeFunctions` by normalized function reference.
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml.cs`
  - `CreateCatalogRow(...)` and `ApplyRunningState(...)` call `item?.IsCatalogFunctionRunning(row.Reference)`.
  - `ApplyRunningState()` starts `_runningStateTimer` only when it already sees a running entry.
  - The timer is stopped when no running entry is visible.

## Attempted Fixes
- No fix attempted in this debug pass.

## Current Hypothesis
- The core running state is not truly row-local, but the `Functions` widget only refreshes that state during its own actions, refreshes, or while its timer is already running.
- If a Button Interaction Rule starts a function while the `Functions` widget timer is stopped, there is no notification that asks the widget to call `ApplyRunningState()`.
- Therefore the row can remain `Ready` until another local trigger occurs, such as manual refresh, a widget action, reattach, or a later polling cycle that only exists if the timer was already active.

## Suggested Fix
- Add a small centralized function execution state notification in `PageItemModel` or a nearby function execution helper.
- Raise the notification when a declarative function execution is added and when it is removed.
- Let `FunctionsControl` subscribe while attached and call `ApplyRunningState()` when the notification fires.
- Keep `RunningDeclarativeFunctions` keyed and queried by normalized function reference as today for the minimal fix.
- A larger structural extraction to a dedicated execution registry is optional, but not required for this specific defect.
