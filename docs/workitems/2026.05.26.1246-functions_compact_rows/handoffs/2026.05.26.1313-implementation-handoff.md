# IMPLEMENTATION HANDOFF

## Goal
Make the `Functions` widget body easier to scan by replacing the current card-like catalog entries with compact rows, displaying both `YAML` and `Python` as consistent type badges, replacing the prominent status bubble with a quieter status indicator, using one toggling `Run`/`Stop` action button, and keeping the list scrollable.

## Scope
- Included: visual row layout for the `Functions` widget catalog, display label mapping from `Declarative` to `YAML`, consistent type badges for `YAML` and `Python`, one row-level `Run`/`Stop` toggle action, scrollable catalog body, compact status handling, and matching widget documentation updates.
- Not included: changing persisted function references, changing the function registry model, changing YAML schema, changing Python registration/discovery behavior, or redesigning the function editor dialog.

## Starting Point
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml` defines `HornetStudio.Editor.Widgets.FunctionsControl`.
- The current catalog item template uses a bordered multi-line card with display name, kind badge, file name, state, source, optional status text, and `Edit`/`Delete` buttons.
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml.cs` builds `FunctionCatalogRow` from `FunctionCatalogEntry`.
- Current `KindText` is `entry.Kind.ToString()`, so declarative entries display as `Declarative`.
- The current status presentation is too visually dominant when represented as a bubble/pill and should not compete with the type badge.
- Existing docs describe rows as showing kind, source, and state.
- Run dispatch already exists through the function registry / `RunFunction` foundation according to `Functions.help.md`, but the `Functions` widget surface currently exposes only `Add Function`, `Refresh`, `Edit`, and `Delete`.

## Tasks
1. Replace the function catalog item template with a compact row that follows this structure: `[TypeBadge] [Name + Source] [Status] [RunOrStop] [Edit] [Delete]`.
2. Render both `YAML` and `Python` as visually consistent type badges.
3. Map `FunctionCatalogKind.Declarative` to the visible type badge text `YAML`; keep `Python` visible as `Python`.
4. Replace separate `Run` and `Stop` buttons with one action button whose text and behavior toggle by execution state.
5. Show `Run` when the entry is runnable and not currently running.
6. Show `Stop` when the matching entry is currently running.
7. During start/stop transitions, disable the toggle action or show a clear transitional state only if the existing execution API exposes such state cheaply.
8. Keep long names and source details from overloading the row by using single-line text trimming and tooltips where useful.
9. Add a scrollable container around only the catalog list, so the widget header/actions and explanatory text remain stable.
10. Represent `Ready`, `Running`, and `Invalid` compactly as quiet text or a small marker, not as a large status badge.
11. Ensure `Invalid` remains visibly distinguishable and exposes the validation message via tooltip or concise inline status.
12. Preserve existing edit/delete behavior and confirmation flow.
13. Update widget documentation to reflect compact rows, consistent badges, the `YAML` display label, the `Run`/`Stop` toggle, and scrollable list behavior.
14. Verify build and inspect the UI behavior manually if the application can be launched in the local environment.

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml` -> replace card item template with compact row template, add scroll viewer/list height behavior, render consistent type badges, add one `Run`/`Stop` toggle button, apply text trimming and tooltips, and remove the prominent status bubble pattern.
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml.cs` -> add display kind mapping, row action state, toggle action text, run/stop event handling, per-entry execution tracking, and compact status properties.
- `src/HornetStudio/docs/widgets/Functions.md` -> update summary documentation for compact rows, type badges, and actions.
- `src/HornetStudio/docs/widgets/help/Functions.help.md` -> update detailed help for compact rows, `YAML` display label, `Python` badge, `Run`/`Stop` toggle, and status behavior.
- `src/HornetStudio/docs/widgets/descriptions/Functions.md` -> update only if the short picker description mentions old row behavior or visible actions.

## Implementation Order
1. Inspect the existing execution API used by `RunFunction` callers and identify the minimal way for `FunctionsControl` to start and cancel a function by registry reference.
2. Extend `FunctionCatalogRow` with UI-only properties for type badge text, status text/marker, action toggle text, action enabled/visible state, and tooltip text.
3. Update `CreateCatalogRow` so declarative entries display as `YAML` while the underlying `FunctionCatalogEntry.Kind` and `Reference` remain unchanged.
4. Ensure Python rows use the same badge control/style as YAML rows instead of plain text.
5. Update the XAML template from card layout to a compact row grid with stable columns for type badge, status, action button, edit, and delete, plus a flexible name/source column.
6. Add a `ScrollViewer` around `WorkflowItemsControl` with an appropriate maximum height or available-space behavior so the function list scrolls instead of stretching the widget.
7. Add one toggle click handler, for example `OnRunStopFunctionClicked`, that dispatches to run or stop based on the row's current execution state.
8. Track running state by stable catalog reference so refreshes can preserve the visible `Stop` state for a running entry.
9. Wire cancel/stop to only the selected running function execution.
10. Keep `Ready` visually quiet and `Invalid` clearly visible, with validation details available without expanding the row.
11. Update documentation files.
12. Build and perform focused manual UI verification.

## Technical Constraints
- Answer and user-facing planning text stay German, but code, comments, UI text, file names, and technical identifiers stay English.
- Do not introduce new dependencies.
- Do not change persisted function references or schema.
- Prefer the existing registry and execution patterns over new services or wrapper abstractions.
- Keep the implementation localized to the current `FunctionsControl` unless the existing execution path requires a small focused helper in an already related class.
- User-visible validation and error messages must be English.
- Keep UI styling aligned with existing Avalonia theme bindings and avoid fixed icon/color assumptions that conflict with theme rules.
- Use `YAML` only as display text; do not rename `FunctionCatalogKind.Declarative`.

## Acceptance Criteria
- Function catalog entries render as compact rows rather than multi-line cards.
- `YAML` and `Python` both render as consistent type badges.
- Declarative entries display `YAML`, not `Declarative`.
- Python entries continue to display `Python`.
- Each row has one primary execution action that displays `Run` when idle and `Stop` when that exact entry is running.
- Separate `Run` and `Stop` buttons are not shown side by side.
- `Edit` and `Delete` remain limited to entries that support those capabilities.
- Status is not shown as a large bubble/pill competing with the type badge.
- `Ready` is visually quiet, `Running` is visible, and `Invalid` is clearly distinguishable.
- Long function names do not expand the row height or overlap action buttons.
- The function list scrolls when there are more rows than fit in the body.
- Existing add/edit/delete workflows still work.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore is available.
- If restore is required, run the repository-approved restore first.
- Manually open a page containing a `Functions` widget and verify compact rows, scrolling, `YAML` and `Python` badges, quiet status display, invalid state, and action enablement.
- Test with at least one YAML entry and, if available, one Python entry.
- Test running and stopping a long-running or delayed function so the row-level `Run`/`Stop` toggle can be observed.

## Out of Scope
- Renaming `FunctionCatalogKind.Declarative`.
- Migrating files or changing `Scripts/Functions` / `Scripts/Workflows` discovery.
- Changing function YAML schema.
- Adding new function step types.
- Redesigning the function editor dialog.
- Changing Button, Monitor, or Workflow function picker behavior unless a compile-time dependency requires a small adjustment.

## Risks / Watchouts
- The existing executor may expose run/cancel at a different layer than `FunctionsControl`; avoid duplicating execution logic if a central dispatch path already exists.
- Refreshing the list while a function runs can lose row object identity; track running state by stable catalog reference, not by row instance.
- Python function execution depends on runtime registration and may not be available during design-time/editor-only sessions; handle unavailable runtime state with a clear footer message.
- Avoid hiding invalid entries behind tooltip-only feedback; there should be a visible marker.
- The source path is currently visible on each card. Moving most of it into a tooltip improves density but must not remove access to important diagnostics.
- The toggle action must not show `Stop` for other rows when a different function is running.

## Notes
- Preferred row structure: fixed-width type badge, flexible name/source area, quiet status text/marker, one execution toggle button, then edit/delete buttons.
- `YAML` and `Python` badges should share shape and visual weight.
- `Status` should be secondary information, not another badge of equal prominence.
- `SourceText`, `FileName`, and validation details are good candidates for row tooltip content instead of always-visible multi-line text.
