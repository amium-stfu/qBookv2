# IMPLEMENTATION HANDOFF

## Goal
Make the `Functions` widget body easier to scan by replacing the current card-like catalog entries with compact one-line rows, adding visible `Run` and `Stop` actions, displaying declarative entries as `YAML`, and keeping the list scrollable.

## Scope
- Included: visual row layout for the `Functions` widget catalog, display label mapping from `Declarative` to `YAML`, row-level `Run` and `Stop` actions, scrollable catalog body, compact invalid/ready status handling, and matching widget documentation updates.
- Not included: changing persisted function references, changing the function registry model, changing YAML schema, changing Python registration/discovery behavior, or redesigning the function editor dialog.

## Starting Point
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml` defines `HornetStudio.Editor.Widgets.FunctionsControl`.
- The current catalog item template uses a bordered multi-line card with display name, kind badge, file name, state, source, optional status text, and `Edit`/`Delete` buttons.
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml.cs` builds `FunctionCatalogRow` from `FunctionCatalogEntry`.
- Current `KindText` is `entry.Kind.ToString()`, so declarative entries display as `Declarative`.
- Existing docs describe rows as showing kind and source and state.
- Run dispatch already exists through the function registry / `RunFunction` foundation according to `Functions.help.md`, but the `Functions` widget surface currently exposes only `Add Function`, `Refresh`, `Edit`, and `Delete`.

## Tasks
1. Replace the function catalog item template with a compact one-line row: `[Type] [Name] [Run] [Stop] [Edit] [Delete]`.
2. Keep long names and source details from overloading the row by using single-line text trimming and tooltips where useful.
3. Add a scrollable container around only the catalog list, so the widget header/actions and explanatory text remain stable.
4. Map `FunctionCatalogKind.Declarative` to the visible label `YAML`; keep `Python` visible as `Python`.
5. Add row properties needed by the compact template, including visible/enabled state for `Run`, `Stop`, `Edit`, and `Delete`.
6. Wire `Run` to the existing function execution path for runnable registry entries.
7. Track active execution per catalog entry reference so `Stop` is visible/enabled only for the matching running entry.
8. Wire `Stop` to cancel only the selected running function execution.
9. Preserve existing edit/delete behavior and confirmation flow.
10. Represent `Ready`/`Invalid` compactly without returning to a multi-line card layout; invalid entries must remain visibly identifiable and should expose the validation message via tooltip or concise inline status.
11. Update widget documentation to reflect the compact row layout, `YAML` label, `Run`, `Stop`, and scrollable list behavior.
12. Verify build and inspect the UI behavior manually if the application can be launched in the local environment.

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml` -> replace card item template with compact row template, add scroll viewer/list height behavior, add `Run` and `Stop` buttons, apply text trimming and tooltips.
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml.cs` -> add display kind mapping, row action state, run/stop event handlers, per-entry execution tracking, and compact status properties.
- `src/HornetStudio/docs/widgets/Functions.md` -> update summary documentation for compact rows and actions.
- `src/HornetStudio/docs/widgets/help/Functions.help.md` -> update detailed help for compact rows, `YAML` display label, `Run`/`Stop`, and status behavior.
- `src/HornetStudio/docs/widgets/descriptions/Functions.md` -> update only if the short picker description mentions old row behavior or visible actions.

## Implementation Order
1. Inspect the existing execution API used by `RunFunction` callers and identify the minimal way for `FunctionsControl` to start and cancel a function by registry reference.
2. Extend `FunctionCatalogRow` with UI-only properties for `DisplayKindText`, action availability, compact status, and tooltip text.
3. Update `CreateCatalogRow` so declarative entries display as `YAML` while the underlying `FunctionCatalogEntry.Kind` and `Reference` remain unchanged.
4. Update the XAML template from card layout to a single-row grid with stable column widths for type and action buttons and a flexible name column.
5. Add a `ScrollViewer` around `WorkflowItemsControl` with an appropriate maximum height or available-space behavior so the function list scrolls instead of stretching the widget.
6. Add `OnRunFunctionClicked` and `OnStopFunctionClicked` handlers using the existing executor/dispatch path and update row state when execution starts, completes, fails, or is canceled.
7. Ensure `RefreshFunctions()` preserves or reapplies running state where possible using the stable catalog reference.
8. Update documentation files.
9. Build and perform focused manual UI verification.

## Technical Constraints
- Answer and user-facing planning text stay German, but code, comments, UI text, file names, and technical identifiers stay English.
- Do not introduce new dependencies.
- Do not change persisted function references or schema.
- Prefer the existing registry and execution patterns over new services or wrapper abstractions.
- Keep the implementation localized to the current `FunctionsControl` unless the existing execution path requires a small focused helper in an already related class.
- User-visible validation and error messages must be English.
- Keep UI styling aligned with existing Avalonia theme bindings and avoid fixed icon/color assumptions that conflict with theme rules.

## Acceptance Criteria
- Function catalog entries render as compact one-line rows.
- Declarative entries display `YAML`, not `Declarative`.
- Python entries continue to display `Python`.
- Rows expose actions in this order: `Run`, `Stop`, `Edit`, `Delete`.
- `Run` is available only for runnable entries.
- `Stop` is available only for the matching currently running entry.
- `Edit` and `Delete` remain limited to entries that support those capabilities.
- Long function names do not expand the row height or overlap action buttons.
- The function list scrolls when there are more rows than fit in the body.
- Invalid entries remain visibly distinguishable and expose the validation message.
- Existing add/edit/delete workflows still work.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore is available.
- If restore is required, run the repository-approved restore first.
- Manually open a page containing a `Functions` widget and verify compact rows, scrolling, `YAML` label, invalid state, and action enablement.
- Test with at least one YAML entry and, if available, one Python entry.
- Test running and stopping a long-running or delayed function so the row-level `Stop` state can be observed.

## Out of Scope
- Renaming `FunctionCatalogKind.Declarative`.
- Migrating files or changing `Scripts/Functions` / `Scripts/Workflows` discovery.
- Changing function YAML schema.
- Adding new function step types.
- Redesigning the function editor dialog.
- Changing Button, Monitor, or Workflow function picker behavior unless a compile-time dependency requires a small adjustment.

## Risks / Watchouts
- The existing executor may expose run/cancel at a different layer than `FunctionsControl`; avoid duplicating execution logic if a central dispatch path already exists.
- Refreshing the list while a function runs can lose row object identity; track running state by stable catalog reference, not by row instance.
- Python function execution depends on runtime registration and may not be available during design-time/editor-only sessions; handle unavailable runtime state with a clear footer message.
- Avoid hiding invalid entries behind tooltip-only feedback; there should be a visible marker.
- The source path is currently visible on each card. Moving it into a tooltip improves density but must not remove access to important diagnostics.
