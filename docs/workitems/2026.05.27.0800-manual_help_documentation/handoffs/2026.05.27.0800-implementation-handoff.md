# IMPLEMENTATION HANDOFF

## Goal
Define and prepare a maintainable documentation model for HornetStudio that supports both in-app help and a PDF handout from shared Markdown sources.

## Scope
Included:
- Audit the existing Markdown documentation relevant to user-facing help.
- Define clear responsibilities for manual, widget description, and widget help Markdown files.
- Add an initial manual structure under the application documentation tree.
- Document the registry interaction concept at handbook level, especially the item-to-host-registry visualization/update path.
- Update repository agent rules so future changes keep the relevant documentation files in sync.

Explicitly not included:
- Implementing the Help menu UI.
- Implementing Markdown rendering inside the application.
- Implementing PDF generation.
- Writing final complete manual content.
- Adding final screenshots.
- Refactoring registries, widgets, or application architecture.

## Starting Point
- Existing user-facing documentation lives under `src/HornetStudio/docs/`.
- Widget documentation already exists in three related forms:
  - `src/HornetStudio/docs/widgets/<WidgetType>.md`
  - `src/HornetStudio/docs/widgets/descriptions/<WidgetType>.md`
  - `src/HornetStudio/docs/widgets/help/<WidgetType>.help.md`
- `agents/solution.md` already contains widget documentation maintenance rules.
- The desired model is hybrid:
  - in-app help through a Help menu
  - PDF handout generated later from shared Markdown documentation
- The registry concept to capture is not a strict hierarchy where `FunctionRegistry` belongs to `Item`.
- Core concept: items become visible and updateable only after they are submitted to the host registry path.

## Tasks
1. Audit existing Markdown documentation under `src/HornetStudio/docs/`, `src/HornetStudio.Host/Python/Integration/`, and root `docs/` for overlap, stale content, and candidate handbook material.
2. Define the documentation source model:
   - `manual/` for cross-topic handbook chapters and future PDF source.
   - `widgets/descriptions/` for short picker/selection text.
   - `widgets/help/` for detailed in-app widget help.
   - existing technical integration documents remain technical source material unless intentionally moved or referenced.
3. Add an initial manual folder structure under `src/HornetStudio/docs/manual/`.
4. Add concise skeleton pages for the manual. Keep them intentionally lightweight and avoid claiming final workflows where the UI is still evolving.
5. Add a registry interaction concept page that explains:
   - external or internal producers submit items to the host registry.
   - the host registry is the required path for visualizing and updating items.
   - UI-facing systems should observe/query the registered host state.
   - `FunctionRegistry` is an independent registry for callable functionality and may operate on items without being structurally owned by items.
6. Add or update an index page that establishes the intended manual chapter order for both in-app help and future PDF handout.
7. Update `agents/solution.md` with project-specific documentation maintenance rules for:
   - manual pages when core concepts or user-visible workflows change.
   - widget descriptions and widget help when widget behavior changes.
   - keeping in-app help and PDF handout aligned through shared Markdown sources.
8. If useful, add a short note in `CHANGELOG.md` only if the repository convention treats documentation model changes as relevant. Do not add release-version entries.

- `src/HornetStudio/docs/manual/index.md` -> define manual purpose, chapter order, and shared-source intent.
- `src/HornetStudio/docs/manual/getting-started.md` -> add lightweight skeleton for first-use guidance.
- `src/HornetStudio/docs/manual/concepts.md` -> add lightweight skeleton for core concepts and link/point to registry interaction.
- `src/HornetStudio/docs/manual/registry-interaction.md` -> document the host registry item flow and independent registry relationship.
- `src/HornetStudio/docs/manual/ui-editor.md` -> add lightweight skeleton for UI editor help.
- `src/HornetStudio/docs/manual/widgets.md` -> explain how widget descriptions, widget help, and manual content relate.
- `src/HornetStudio/docs/manual/python-integration.md` -> add lightweight skeleton and reference existing Python integration documentation.
- `src/HornetStudio/docs/manual/troubleshooting.md` -> add lightweight skeleton.
- `src/HornetStudio/docs/manual/glossary.md` -> add initial glossary skeleton with terms such as item, host registry, UI registry, function registry, widget, and producer.
- `agents/solution.md` -> add HornetStudio-specific rules for maintaining manual/help/description documentation.
- `CHANGELOG.md` -> optional documentation note only if consistent with existing changelog usage.

## Implementation Order
1. Inspect the existing Markdown files listed above and note overlaps or naming inconsistencies.
2. Create `src/HornetStudio/docs/manual/` with minimal skeleton files.
3. Write `index.md` with the manual chapter order and explain that the same Markdown source is intended for in-app help and future PDF output.
4. Write `registry-interaction.md` first, because it defines the conceptual model driving the rest of the handbook.
5. Add the remaining manual skeleton pages with short headings and TODO-style content notes only where needed.
6. Update `agents/solution.md` so future widget, workflow, registry, and concept changes mention the correct documentation files.
7. Review Markdown links and ensure all referenced relative paths are valid.
8. Run a lightweight verification by listing the new files and, if available, checking Markdown links manually or with an existing local tool.

## Technical Constraints
- Chat responses must be German; Markdown content, headings, UI text, and technical terms in repository files must be English.
- Do not introduce new dependencies.
- Do not implement PDF generation yet.
- Do not implement application UI changes yet.
- Keep the manual skeleton concise and avoid overengineering.
- Keep documentation source roles explicit to avoid duplicate long-form content.
- Follow existing repository documentation paths under `src/HornetStudio/docs/`.
- Keep `agents/solution.md` project-specific; do not move HornetStudio-specific paths into reusable modules under `agents/`.

## Acceptance Criteria
- A new manual documentation area exists under `src/HornetStudio/docs/manual/`.
- The manual has a clear index and chapter order suitable for in-app help and future PDF generation.
- The registry interaction concept is documented accurately:
  - item producers submit items to the host registry.
  - item visualization/update goes through the host registry path.
  - `FunctionRegistry` is presented as independent from item ownership.
- Widget documentation roles are clarified:
  - description files are short selection text.
  - help files are detailed context help.
  - manual files are cross-topic handbook chapters.
- `agents/solution.md` tells future agents which documentation files to maintain when relevant code or concepts change.
- No runtime behavior changes are made.

## Verification
- Confirm all new files are under `src/HornetStudio/docs/manual/`.
- Confirm all Markdown links added in the manual resolve to existing repository-relative paths or intentionally planned pages.
- Confirm `agents/solution.md` still keeps HornetStudio-specific rules in the solution module.
- Build is not required if only Markdown and agent-rule documentation changed, but a formatting/link review is required.

## Out of Scope
- Help menu implementation.
- Markdown viewer implementation.
- PDF export/build pipeline.
- Final user manual copywriting.
- Screenshot generation.
- Registry or widget code changes.
- Release versioning.
- Notice file changes.

## Risks / Watchouts
- Avoid merging `description` and `help` into one physical source; they have different user-interface roles.
- Avoid making `FunctionRegistry` appear item-owned or item-scoped unless code review proves that relationship.
- Avoid documenting speculative architecture as final behavior.
- Avoid duplicating technical Python documentation into the manual; reference or summarize it instead.
- Watch for duplicated Markdown copies under root `docs/` and `src/HornetStudio/docs/`; do not delete or move them without a separate cleanup decision.
