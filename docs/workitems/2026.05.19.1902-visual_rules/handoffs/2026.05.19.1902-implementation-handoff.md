# IMPLEMENTATION HANDOFF

## Goal
Add widget-level `VisualRules` so a widget can change visual presentation based on boolean Monitor widget rule states. The first implementation focuses on background colors for widget sections and an optional blink effect.

## Scope
- Add persisted `VisualRules` next to existing `InteractionRules` on widget items.
- Add an editor surface in the existing `Action` tab under a new `Visual` section.
- Allow sources only from Monitor widget rules that expose boolean active/inactive state.
- Allow targets only for `Header`, `Body`, `Footer`, and optionally the whole `Widget` container if the existing control composition supports it cleanly.
- Allow property only for `Background` in the first implementation.
- Allow effect values `None` and `Blink`.
- Apply rules at runtime/editor preview without writing blinking state back into registries or persisted layout.

## Starting Point
- Widgets already have `InteractionRules` persisted through `FolderItemDocument.InteractionRules`.
- Interaction rules are modeled in `src/HornetStudio.Editor/Models/ItemInteractionRule.cs`.
- Interaction rule editor UI exists in `src/HornetStudio.Editor/Widgets/Common/InteractionRulesEditorDialogWindow.axaml(.cs)`.
- `MainWindowViewModel` serializes and deserializes `InteractionRules` for JSON/YAML layout paths.
- Monitor widgets evaluate rules and publish status values through `MonitorRuleRow` in `src/HornetStudio.Editor/Widgets/Monitor/MonitorControl.axaml.cs`.
- The Monitor rule active state already exists internally as a boolean and should be the only source for VisualRules in version 1.

## Tasks
1. Define the `VisualRules` data model and codec.
2. Persist `VisualRules` in layout documents and import/export paths.
3. Expose `VisualRules` as an editable widget property in the Action tab.
4. Create a Visual rule editor UI matching the existing Interaction rule editor style.
5. Implement monitor-rule source selection.
6. Apply visual rules to widget section backgrounds.
7. Implement blink as a UI-only effect.
8. Add focused tests for parse/serialize and layout roundtrip behavior.
9. Update widget documentation and changelog.

## Implementation Order
1. Add `VisualRule` model with enums:
   - `VisualRuleSourceKind.MonitorRule`
   - `VisualRuleTarget.Header`, `Body`, `Footer`, `Widget`
   - `VisualRuleProperty.Background`
   - `VisualRuleEffect.None`, `Blink`
2. Add a line-based codec similar to `ItemInteractionRuleCodec`, or a structured document mapping if the current layout pipeline already prefers JSON arrays for rule lists.
3. Add `VisualRules` to `FolderItemModel` and `FolderItemDocument`.
4. Extend layout serialization/deserialization in `MainWindowViewModel`.
5. Add VisualRules to the editable Action-tab property list near `InteractionRules`.
6. Build the Visual editor:
   - Source monitor widget/rule picker.
   - Target section selector.
   - Active background color selector/text value.
   - Optional inactive/default color behavior.
   - Effect selector.
7. Implement rule evaluation:
   - Resolve source to monitor rule active state.
   - Skip unresolved sources visibly in editor diagnostics or validation.
   - Apply the last matching rule for the same target/property.
8. Add blink:
   - Use a UI timer or existing Avalonia animation mechanism.
   - Blink only when the matching rule is active and effect is `Blink`.
   - Do not persist or publish blink phase.
9. Add tests and documentation.
10. Run restore/build/tests according to repository conventions.

## Technical Constraints
- Do not merge VisualRules into `InteractionRules`.
- Do not introduce a second formula language or generic condition builder.
- Do not allow arbitrary item paths as sources in version 1.
- Use existing Avalonia patterns, existing color handling, and existing editor dialog styling.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Chat summaries remain German.
- Public classes and public methods require XML documentation.
- Avoid new dependencies.
- Keep changes scoped to VisualRules and directly required integration.

## Acceptance Criteria
- A widget can store one or more `VisualRules`.
- The Action tab shows an additional `Visual` area beside/under the existing Interactions area.
- A VisualRule can reference a Monitor widget rule active state.
- When the referenced Monitor rule is active, the selected widget section background changes to the configured color.
- When the rule is inactive, the section returns to its normal configured/theme background unless an inactive color is explicitly included.
- Blink works as a UI-only visual effect for active rules.
- Existing `InteractionRules` behavior is unchanged.
- Existing layouts without `VisualRules` still load without changes.
- Export/import preserves VisualRules.

## Verification
- `dotnet restore HornetStudio.sln`
- `dotnet build HornetStudio.sln --no-restore`
- Run `src/HornetStudio.Editor.Tests` according to its console self-test pattern if it is affected by the implementation.
- Manual UI check:
  - Create a Monitor widget rule.
  - Create a Signal/Button widget VisualRule referencing that Monitor rule.
  - Trigger active and inactive states.
  - Verify Header/Body/Footer background behavior.
  - Verify blink stops cleanly when the rule clears.

## Out of Scope
- Generic expressions for VisualRules.
- Non-Monitor sources.
- Foreground, border, corner radius, font, icon, or arbitrary style properties.
- Per-rule transition animations beyond simple blink.
- Remote/runtime control of VisualRules.
- Monitor widgets directly pushing styles into other widgets.

## Risks / Watchouts
- Avoid tight coupling from Monitor widget instances to target widgets. Resolve by stable published monitor status paths or a central lookup in existing view model state.
- Multiple matching rules need deterministic behavior. Use list order, with the last matching rule winning for the same target/property.
- Blink timers must be disposed when widgets unload to avoid leaks.
- Theme fallback must remain intact when no VisualRule is active.
- Color parsing must follow existing project behavior and fail visibly or ignore invalid colors without crashing.
- Footer rules should be harmless when `ShowFooter` is false.
