# IMPLEMENTATION HANDOFF

## Goal
Restrict VisualRules to a small widget-specific version 1 surface:
- `SignalWidget` / `ItemModel` -> `BodyBackColor`
- `ButtonWidget` -> `ButtonBackColor`
- `CircleDisplay` -> `DisplayBackColor`
- all other widget types -> no VisualRules editing or runtime application

## Scope
- Replace the generic `Target` + `Property = Background` VisualRules editing model with a constrained visual property model.
- Keep existing Monitor rule source behavior, `Effect = None`, and `Effect = Blink`.
- Keep persisted `VisualRules` as the property name.
- Preserve backward compatibility for already saved generic rules where practical.
- Update tests and documentation for the limited version 1 scope.

## Starting Point
- `VisualRule` currently uses `SourceKind`, `SourcePath`, `Target`, `Property`, `Effect`, `ActiveValue`, and `InactiveValue`.
- `PageItemModel.ApplyVisualRuleOverrides` currently resolves generic targets `Widget`, `Header`, `Body`, and `Footer`.
- `SignalWidget` works with VisualRules against body background.
- `ButtonWidget` does not work visually with the generic body background because the button fills the body and uses `EffectiveButtonBodyBackground`.
- `CircleDisplay` has a concrete `DisplayBackColor` / `EffectiveDisplayBackColorBrush` path.
- VisualRules are exposed in the Action tab via `MainWindowViewModel` and `VisualRulesEditorDialogWindow`.

## Tasks
1. Introduce explicit VisualRule properties for the limited v1 surface.
2. Filter VisualRules editing by widget type.
3. Apply VisualRules only to the matching concrete effective property.
4. Preserve or migrate compatible existing rules.
5. Remove or hide irrelevant target/property choices from the VisualRules editor.
6. Update tests for Signal, Button, and CircleDisplay VisualRule behavior.
7. Update docs and changelog.

## File-Level Changes
- `src/HornetStudio.Editor/Models/VisualRule.cs` -> add or replace property enum values with concrete design properties such as `BodyBackColor`, `ButtonBackColor`, `DisplayBackColor`; keep legacy parse support for `Target + Background`.
- `src/HornetStudio.Editor/Models/PageItemModel.cs` -> apply VisualRules only when `Kind` supports them; update effective background properties for the three supported cases.
- `src/HornetStudio.Editor/ViewModels/MainWindowViewModel.cs` -> expose the `VisualRules` editor field only for supported widget kinds; provide supported visual property options per owner item.
- `src/HornetStudio.Editor/ViewModels/ItemVisualRuleEditorRow.cs` -> remove or de-emphasize target options; bind a constrained `VisualProperty`/`PropertyName` option list.
- `src/HornetStudio.Editor/Widgets/Common/VisualRulesEditorDialogWindow.axaml` -> remove target column or make it hidden; show only the allowed property list for the owner widget.
- `src/HornetStudio.Editor/Widgets/Common/VisualRulesEditorDialogWindow.axaml.cs` -> initialize row property options from the owner widget; reject unsupported rows on save.
- `src/HornetStudio.Editor/Persistence/LayoutDocument.cs` -> keep layout document compatible; add new enum values if needed.
- `src/HornetStudio.Editor.Tests/Program.cs` -> update VisualRule codec and layout tests; add limited property mapping tests.
- `src/HornetStudio/docs/widgets/help/Signal.help.md` -> document Signal VisualRules limited to body back color.
- `src/HornetStudio/docs/widgets/help/Button.help.md` -> document Button VisualRules limited to button back color.
- `src/HornetStudio/docs/widgets/help/CircleDisplay.help.md` -> document CircleDisplay VisualRules limited to display back color.
- `CHANGELOG.md` -> document constrained VisualRules v1 behavior.

## Implementation Order
1. Extend the model with a concrete property concept.
   - Preferred: keep `VisualRuleProperty` but change values to `BodyBackColor`, `ButtonBackColor`, `DisplayBackColor`.
   - Keep legacy `Background` parse support for compatibility.
2. Add a helper that returns supported VisualRule properties for a `FolderItemModel`:
   - `IsItem` / Signal -> `BodyBackColor`
   - `IsButton` -> `ButtonBackColor`
   - `IsCircleDisplay` -> `DisplayBackColor`
   - otherwise empty
3. Update editor field generation:
   - show `VisualRules` only when the selected item supports at least one VisualRule property.
   - hide VisualRules for unsupported widgets.
4. Update `VisualRulesEditorDialogWindow`:
   - remove target selection from the visible UI, or keep it internal for compatibility only.
   - show only the owner widget's allowed property.
   - default the new row property to the first allowed property.
   - prevent saving rows with unsupported properties.
5. Update `ApplyVisualRuleOverrides`:
   - start from normal theme/design values.
   - for supported widgets only, resolve the last matching rule for the concrete property.
   - apply `BodyBackColor` to `EffectiveBodyBackground`.
   - apply `ButtonBackColor` to the effective button background path.
   - apply `DisplayBackColor` to `EffectiveDisplayBackColor`.
   - keep Blink behavior identical for all three properties.
6. Update button effective background handling:
   - if a VisualRule result exists for `ButtonBackColor`, use it before `ButtonBodyBackground`/theme fallback.
   - ensure hover/press backgrounds derive from the current effective button back color so Blink is visible when not hovered and remains coherent when hovered.
7. Add compatibility mapping:
   - legacy `Target = Body`, `Property = Background` on Signal/Item -> `BodyBackColor`
   - legacy `Target = Body` or `Target = Widget`, `Property = Background` on Button should not silently map unless this is explicitly wanted; preferred mapping is only new `ButtonBackColor`.
   - legacy CircleDisplay rules should map only if a future/old rule clearly targets display; otherwise ignore.
8. Update tests and docs.
9. Run solution build and focused editor self-tests.

## Technical Constraints
- Do not add a generic styling system.
- Do not support Header/Footer/Widget-level VisualRules in v1.
- Do not show VisualRules for unsupported widgets.
- Do not introduce new dependencies.
- Keep existing `VisualRules` persistence readable.
- Code, comments, XML documentation, UI text, file names, and technical identifiers stay English.
- Chat responses stay German.
- Public classes and public methods require XML documentation.

## Acceptance Criteria
- `SignalWidget` VisualRules can only affect `BodyBackColor`.
- `ButtonWidget` VisualRules can only affect `ButtonBackColor`.
- `CircleDisplay` VisualRules can only affect `DisplayBackColor`.
- Unsupported widgets do not show a VisualRules editor entry.
- `Effect = Blink` switches at 1 Hz between active and inactive/default color for all supported properties.
- Existing Signal body VisualRules continue to work.
- Button VisualRules visibly affect the button fill, not a hidden body layer.
- CircleDisplay VisualRules visibly affect the display fill.
- Layout import/export preserves supported VisualRules.

## Verification
- `dotnet build HornetStudio.sln --no-restore`
- `dotnet src\HornetStudio.Editor.Tests\bin\Debug\net9.0-windows\HornetStudio.Editor.Tests.dll`
- Manual UI checks:
  - SignalWidget VisualRule changes/blinks body background.
  - ButtonWidget VisualRule changes/blinks button back color.
  - CircleDisplay VisualRule changes/blinks display back color.
  - Log/Monitor/List/Chart/etc. do not expose VisualRules in the editor.

## Out of Scope
- Header/Footer/whole-widget VisualRules.
- Foreground, border, corner radius, font, icon, opacity, or arbitrary style properties.
- Generic expressions or non-Monitor sources.
- Per-widget multiple visual property sets beyond the three listed v1 properties.
- Cross-widget style pushing from MonitorWidget.

## Risks / Watchouts
- `ButtonBackColor` currently maps to `ButtonBodyBackground` in the model; keep naming clear in UI and persistence conversion.
- Existing persisted generic rules may exist from the first VisualRules implementation. Avoid crashing on them; either map supported Signal body rules or ignore unsupported rules.
- Blink fallback must restore the normal effective design/theme color when `InactiveValue` is empty.
- If `EffectiveButtonBodyBackground` remains computed-only, a separate visual override field may be needed so Blink can override it without mutating `ButtonBodyBackground`.
- Hidden unsupported rules should not be re-saved accidentally if the editor is not exposed for unsupported widgets.
