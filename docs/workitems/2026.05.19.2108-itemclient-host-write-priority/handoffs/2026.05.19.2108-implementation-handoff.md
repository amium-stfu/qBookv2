# IMPLEMENTATION HANDOFF

## Goal
Make the ItemClient write-back behavior owner-aware so locally hosted writes from the Host process keep priority over stale or echoed MQTT state while MQTT write requests remain allowed.

## Scope
- Add a focused host-priority guard to the ItemClient write-back path for writable published items.
- Preserve MQTT write capability for external clients.
- Prevent old retained MQTT `read` values, state echoes, or broker property echoes from reverting a value that was just written locally by the Host process.
- Keep the existing MQTT topic layout, Item model, DataRegistry API, and server architecture unchanged.

## Starting Point
- `HostItemBrokerWriteBackClient` subscribes to broker messages for writable published ItemClient definitions and writes broker updates back into `HostRegistries.Data`.
- `ItemClientControl.HostItemBrokerPublisher` publishes local Host registry changes to MQTT and already records locally published `write` commands through `OwnWriteEchoTracker`.
- `HostRegistries.Data` is intentionally transport-neutral and should not gain MQTT-specific ownership logic.
- The observed failure mode is a feedback loop where UI or Monitor Rule writes value A, then an older or echoed MQTT value B is applied back locally, causing visible switching between A and B.

## Tasks
1. Audit the exact message flow for writable ItemClient definitions:
   - local UI write
   - Monitor action write
   - MQTT `read` update
   - MQTT property update
   - MQTT write request
2. Add a local host write observation mechanism in the ItemClient write-back boundary.
3. Record local Host registry writes per resolved target path and parameter when the ItemClient publisher observes relevant local changes.
4. Before applying non-request broker state back to local items, compare it against the recent local Host write state.
5. Keep broker `ItemWriteRequestMessage` handling enabled, but apply it only through the existing write-channel path.
6. Add focused coverage for stale MQTT state being ignored after a local Host write.
7. Update ItemClient widget documentation if user-facing behavior is documented there.
8. Update `CHANGELOG.md` with the behavior fix.

## Implementation Order
1. Inspect `HostItemBrokerWriteBackClient.HandleMessageAsync(...)` and `ItemClientControl.HostItemBrokerPublisher.OnDataItemChanged(...)` to confirm the best local write signal.
2. Introduce a small per-path local-write tracker near existing `OwnWriteEchoTracker` in `ItemClientControl.axaml.cs`, or reuse/extend the existing tracker if it can stay clear and focused.
3. Pass the tracker into `HostItemBrokerWriteBackClient` through constructor callbacks, following the existing `tryConsumeOwnWriteEcho` pattern.
4. Record local Host writes when the publisher receives a local `DataChangedEventArgs` for an active writable definition and the change targets the published local path.
5. In write-back, resolve the same effective write target path used for application and skip non-`ItemWriteRequestMessage` broker updates when they conflict with a recent local Host write for that target.
6. Keep `ItemWriteRequestMessage` accepted so external MQTT control still works; if it is the same value as a recent local Host write, consume/ignore only true own echoes.
7. Add diagnostics using the existing `HornetStudio.ItemClient.WriteBackDiagnostics` switch with reasons such as `hostWritePriority`.
8. Add/update tests and documentation.
9. Run focused tests, then build the solution if restore state allows it.

## Technical Constraints
- Do not add dependencies.
- Do not move priority logic into `DataRegistry`; keep it in the ItemClient write-back boundary.
- Do not reintroduce server-side item ownership or retained state authority.
- Keep MQTT write requests allowed.
- Keep public APIs stable unless a constructor change is internal to editor widget composition.
- Use existing patterns: `OwnWriteEchoTracker`, `TargetPathHelper`, `HostRegistries.Data`, `BrokerPublishedItemDefinition`, and diagnostic switches.
- Code, comments, XML docs, user-visible messages, and identifiers must be English.

## Acceptance Criteria
- A manual UI write to a published writable ItemClient item is not reverted by an older retained MQTT `read` value for the same broker path.
- A Monitor action write to the same item is not reverted by an older retained MQTT `read` value or broker echo.
- An external MQTT write request can still write to the item when the Host has not just asserted a conflicting local value.
- The write-back diagnostics can explain skipped broker updates with a clear reason.
- Existing ItemClient receive, attach, and publish behavior remains unchanged outside conflict handling.

## Out of Scope
- Changing MQTT topic names or retained-topic structure.
- Changing the Item server architecture.
- Changing UDL CAN write behavior.
- Adding a general global ownership model to `DataRegistry`.
- Adding new UI settings for ownership priority unless explicitly requested later.

## Risks / Watchouts
- A protection window that is too long could make legitimate MQTT control feel ignored after a local rule writes.
- A protection window that is too short might still allow retained MQTT state to revert the Host value.
- The tracker must key by resolved local write target, not only by broker path, because published definitions may use write targets or `write_path`.
- Numeric equality should follow existing tolerant comparison behavior where possible to avoid false conflicts between `1`, `1.0`, and `1f`.
- Avoid suppressing `ItemWriteRequestMessage` broadly; only stale state/echo updates should be blocked by Host priority.

## Notes
- The intended ownership rule is: the Host process is authoritative for its published writable items, while MQTT remains a permitted request/input source.
- The most likely implementation shape is a small local-write tracker with short retention, integrated with existing ItemClient publisher/write-back objects.
- Prefer minimal local changes over new ownership services or registry-wide abstractions.
