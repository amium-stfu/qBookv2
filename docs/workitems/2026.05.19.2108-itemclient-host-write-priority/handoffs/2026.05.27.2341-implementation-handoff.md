# IMPLEMENTATION HANDOFF

## Goal
Extend ItemClient write-back priority so Host-originated value writes from declarative YAML functions and Python host-value writes are treated like local Host writes and are not reverted by stale broker state or echoes.

## Scope
- Keep the existing ItemClient publish/write-back architecture.
- Include writes triggered by declarative functions through `FunctionSetValueStepDefinition`.
- Include Python host-value writes handled by `PythonClient.HandleHostValueWrite`.
- Preserve external broker write requests.
- Avoid broad registry ownership or MQTT topic changes.
- Do not add dependencies.

## Starting Point
- The existing ItemClient host-priority implementation is scoped to the ItemClient boundary.
- `HostItemBrokerWriteBackClient` already accepts a `hasRecentLocalHostWriteConflict` callback and skips conflicting non-request broker state.
- `ItemClientControl.LocalHostWriteTracker` stores recent local writes by target path, parameter, value, and short retention.
- `ItemClientControl.HostItemBrokerPublisher.RecordLocalHostWriteState` records changes only while publishing observes matching `DataChangedEventArgs`.
- YAML functions write through `PageItemModel.ExecuteRunFunctionSetValueAsync`, which delegates to `TryApplyInteractionWrite`.
- Python host-value writes write directly through `HostRegistries.Data.UpdateValue` in `PythonClient.HandleHostValueWrite`.
- The observed issue is that a YAML function can set a value, then broker state applies an older value back, causing value oscillation.

## Tasks
1. Confirm whether the existing `LocalHostWriteTracker` fails for YAML functions because the publish matcher only matches `definition.LocalPath` and misses the resolved write target path.
2. Make local write tracking key resolution consistent with write-back application:
   - value writes must record the resolved value write target path,
   - `write` property changes must record the resolved write-request target path,
   - non-write user properties must record the local item path.
3. Extend `BrokerPublishedItemChangeMatcher.ShouldPublish` or add a sibling matcher so publisher observation can detect registry changes on the effective write target, not only the configured local publish path.
4. Ensure declarative YAML function `SetValue` writes are observed as local authoritative registry changes without adding a separate YAML-specific path if generic effective-target matching is sufficient.
5. Ensure Python host-value writes are covered when the Python write target is a published writable ItemClient target.
6. Keep `ItemWriteRequestMessage` behavior unchanged: broker write requests remain allowed and must not be blocked by the host-priority guard.
7. Add focused tests around the matcher/tracker/write-back behavior, preferably in `src/HornetStudio.Editor.Tests`.
8. Update ItemClient documentation only if it currently describes write-back priority or writable ownership behavior.
9. Update `CHANGELOG.md` with a concise fix entry if this is implemented as a user-visible behavior fix.

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/Broker/ItemClientControl.axaml.cs` -> adjust local write observation so YAML/Python writes against effective writable targets are recorded in `LocalHostWriteTracker`.
- `src/HornetStudio.Editor/Widgets/Broker/HostItemBrokerWriteBackClient.cs` -> keep the existing priority decision, but expose or reuse target-resolution helpers only if tests require it.
- `src/HornetStudio.Editor.Tests/Program.cs` -> add focused regression coverage for function-originated local writes preventing stale broker state write-back.
- `src/HornetStudio.Host/Python/Client/PythonClient.cs` -> change only if registry write origin must be made explicit; prefer no change if generic registry observation covers Python writes.
- `src/HornetStudio/docs/widgets/ItemClient.md` -> update only if current docs mention write-back conflict behavior.
- `src/HornetStudio/docs/widgets/help/ItemClient.help.md` -> update only if detailed help mentions writable ownership.
- `CHANGELOG.md` -> add a short fix note if the project changelog is maintained for current changes.

## Implementation Order
1. Inspect `BrokerPublishedItemChangeMatcher.ShouldPublish`, `HostItemBrokerPublisher.RecordLocalHostWriteState`, and `HostItemBrokerWriteBackClient.ResolveValueWriteTargetPath`.
2. Add or adjust a matcher that returns true when a registry change targets either the configured local item or the resolved effective write target for a writable definition.
3. Use that matcher in `HostItemBrokerPublisher.OnDataItemChanged` before calling `RecordLocalHostWriteState` and `PublishDefinitionIfAvailable`.
4. Keep publishing behavior for snapshots/change messages compatible with the current matcher; if needed, separate "should publish" from "should record host priority" to avoid publishing wrong paths.
5. Verify `RecordLocalHostWriteState` records the same target path and parameter that `HostItemBrokerWriteBackClient` later checks.
6. Add tests for:
   - YAML-style local value update on resolved write target followed by broker `read` state with a different value is ignored.
   - Python-style direct value update on the same resolved target followed by broker `read` state with a different value is ignored.
   - broker `ItemWriteRequestMessage` is still applied.
   - same-value broker echo is not treated as a conflict.
7. Update docs/changelog only where relevant.
8. Run focused editor tests, then run the solution build if restore state allows it.

## Technical Constraints
- Chat response must be German; code, comments, XML docs, identifiers, and user-visible messages must be English.
- Prefer existing helpers and nested types over new services.
- Do not move MQTT-specific priority into `DataRegistry`.
- Do not add dependencies.
- Do not change MQTT topic names or retained-message layout.
- Keep constructor/API changes internal to the editor widget boundary where possible.
- Use named arguments for unclear multi-parameter calls.

## Acceptance Criteria
- A YAML declarative function `SetValue` write to a published writable item remains authoritative over stale broker `read` or property state.
- A Python host-value write to a published writable item remains authoritative over stale broker `read` or property state.
- The ItemClient no longer visibly switches a value back and forth after a function-originated write.
- External broker write requests still work.
- Existing ItemClient attach, receive, publish, and retained snapshot behavior is unchanged outside conflict handling.
- Diagnostics can still explain skipped broker updates with `hostWritePriority`.

## Out of Scope
- Global registry ownership model.
- New UI settings for priority behavior.
- MQTT protocol/topic redesign.
- UDL write semantics beyond observing registry changes.
- Refactoring function execution or Python client architecture.

## Risks / Watchouts
- Mixing publish matching and priority tracking may accidentally publish changes for target paths that should only be used for conflict suppression.
- A too-broad matcher could suppress legitimate broker state for unrelated published items.
- A too-short retention window can still allow retained broker state to revert function writes.
- A too-long retention window can make legitimate broker control appear ignored.
- Resolved write target paths must be normalized consistently between tracker recording and write-back conflict checks.
- Numeric value comparisons should preserve existing tolerant equality behavior.

## Notes
- Prefer solving this by making the existing ItemClient local-write observation see function-originated registry changes.
- Only introduce explicit registry write origin metadata if effective-target observation cannot distinguish the needed cases safely.
- The intended rule is: locally asserted Host state, including YAML and Python functions running in the Host context, has priority over stale broker state; explicit broker write requests remain accepted inputs.
