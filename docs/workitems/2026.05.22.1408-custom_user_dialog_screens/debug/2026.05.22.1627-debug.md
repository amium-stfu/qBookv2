# DEBUG REPORT

## Problem
`DialogWidget` overlays render much larger than the dialog widget definition, and the DialogWidget property editor does not expose `Rows` and `Columns`.

After reloading a project from `Folder.yaml`, a `DialogWidget` control is treated like a normal signal/item because the loaded YAML definition is not mapped back to the dialog widget kind.

After project reload, a button `OpenDialog` interaction can fail until the interaction is saved again. Opening the dialog also shifts the visible screen layout, and the dialog overlay shows an empty scrollable editor workspace instead of the dialog content.

After the overlay shell renders correctly, the dialog widget body can still miss the nested cell content.

## Expected Behavior
An opened `DialogWidget` should use the widget definition bounds as its visible overlay content size. The property editor should expose the table-layout `Rows` and `Columns` used by the widget.

A YAML control with `Type: 'DialogWidget'` should load as `ControlKind.DialogWidget`, and top-level `Rows` / `Columns` written by the YAML serializer should be read back into the item properties.

Dialog widget ids should remain stable across YAML load/save so interaction rules keep working immediately after reload. Opening a dialog should not participate in the main screen layout measurement, and the overlay should render the dialog widget content directly without the full folder editor workspace or scrollbars.

Nested table/body items inside the dialog widget should remain renderable when the dialog widget is cloned into an overlay.

## Actual Behavior
The overlay renders a full `FolderEditorWidget` workspace inside a large overlay card, so the visible area is the editor canvas instead of the dialog widget size. `DialogWidget` reuses table layout behavior but had no `Rows` / `Columns` property section.

The YAML loader preserved the raw `Type` string, but the runtime `GetControlKindFromUiType` did not recognize `DialogWidget` and fell back to `ControlKind.Signal`. Additionally, top-level `Rows` and `Columns` from saved YAML were not copied into loaded control properties unless they were nested under `Properties` / `Control`.

`ApplyKnownUiProperties` ignored the loaded `Identity.Id` and generated a new item id on every YAML load, which broke `OpenDialog` targets that reference the stable dialog widget id. The overlay was also hosted as a grid-spanning layout child and rendered a complete `FolderEditorWidget`, so its desired size could affect the main grid and the visible content was the editor canvas rather than the dialog table itself.

The overlay clone used the normal document roundtrip, including legacy kind resolution. That can reinterpret nested signal/body items with table metadata as `CircleDisplay`, and `CircleDisplay` is not a valid table child control, so the table renderer skipped the nested content.

## Current Hypothesis
- The large visual defect came from rendering the cloned dialog widget inside an unbounded editor workspace. Bounding the host to the original widget definition size makes the overlay match the definition.
- The reload defect came from a split load path: YAML parsing preserved `Type: DialogWidget`, but the runtime conversion layer did not include the new type in its known control-kind mapping.
- The remaining reload/action defect came from unstable item ids, not from action execution itself.
- The visual shift and empty scrollbars came from using a full editor workspace as dialog content and letting the overlay span the layout grid.
- The missing body content came from legacy kind migration during overlay cloning, not from missing YAML child data.
- The remaining body size mismatch came from hosting `EditorTableControl` in the overlay without the explicit width and height bindings that the normal page canvas applies. With no explicit size, the table control measured only to its content, which visually shrank and clipped nested body widgets inside the fixed-size overlay frame.
- The remaining grid mismatch came from the overlay clone using the internal `FolderItemDocument` roundtrip without persisting table child cell metadata. That dropped each nested child's row, column, and span during cloning, so the overlay re-rendered multiple body children as if they all belonged to the first cell.
- The duplicate header came from rendering both the outer overlay title bar and the cloned dialog widget header. Hiding the caption only on the clone keeps the runtime overlay title while leaving the original dialog widget unchanged in the designer.
