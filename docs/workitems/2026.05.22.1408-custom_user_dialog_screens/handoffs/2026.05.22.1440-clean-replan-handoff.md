# IMPLEMENTATION HANDOFF

## Goal
Cleanly discard the already implemented dialog-screen approach and replace it with a `DialogWidget` based design. Dialog definitions should be normal widgets that can be placed on any screen. `OpenDialog` and `CloseDialog` should target a dialog widget, not a screen.

## Scope
- Remove the previous `ScreenKind = Dialog` / dialog-screen implementation.
- Restore screens to normal page-only behavior with no dialog classification.
- Add a new `DialogWidget` widget type that can be placed on any screen.
- Allow developers to organize dialog widgets wherever they prefer, for example on a dedicated "Dialogs" screen, without any required screen id or special folder.
- Add or update `OpenDialog` and `CloseDialog` so they reference a stable dialog widget id.
- Extend `OpenDialog` with placement options:
  - `origin`, default `Screen`
  - `position`, default `Center`
- Render opened dialog widgets as application-internal overlays in the MVP.
- Add a picker that lists dialog widgets, with their containing screen/folder context.
- Update tests and documentation to describe the widget-based dialog model.

## Starting Point
- A previous implementation introduced dialog screens through `ScreenKind`, `DialogScreenOverlayEntry`, screen-kind serialization, dialog-screen picker logic, and overlay handling.
- This previous implementation has not been used productively, so no migration path is required.
- Existing normal screen projects should remain loadable without dialog metadata.
- Widget infrastructure already supports adding new `ControlKind` values and creating widgets through the existing widget picker.
- Interaction rules already include `OpenDialog` and `CloseDialog`, but currently they are wired to screen ids from dialog screens.

## Tasks
1. Remove the old dialog-screen model completely:
   - remove `ScreenKind` enum usage if it only exists for dialog screens
   - remove `PageModel.ScreenKinds`
   - remove `CurrentViewKind`, `IsDialogScreen`, and `UpdateScreenKind`
   - remove screen-kind parsing and emitting from layout loaders
   - remove dialog-screen menu entries and screen-kind toggles
   - remove dialog-screen overlay view model and overlay state
   - remove dialog-screen picker logic
   - remove dialog-screen tests and documentation references
2. Restore screen serialization to the previous scalar-compatible model:
   - `Screens:`
   - `  1: 'HomeScreen'`
   - no persisted `Kind: Dialog` metadata
3. Add a new `DialogWidget` control kind:
   - persisted type name should be stable and English, for example `DialogWidget`
   - default name/body caption should make the widget identifiable in the designer
   - include it in widget creation, serialization, deserialization, and preview/selection paths
4. Define dialog widget identity:
   - use an existing stable item id if one already exists and survives save/load
   - otherwise add a focused stable dialog id property to `FolderItemModel`
   - picker stores the stable id, not the mutable display name
5. Define dialog widget content model:
   - prefer reusing the existing table/container mechanics if that is how complex widget layouts are already represented
   - avoid creating a second layout engine
   - keep `DialogWidget` behavior focused on being a dialog definition and runtime host
6. Update interaction actions:
   - keep `OpenDialog` and `CloseDialog` action names if already user-visible
   - change their target semantics from screen id to dialog widget id
   - `OpenDialog` supports optional placement arguments: `origin` and `position`
   - `CloseDialog` only needs the dialog widget id
7. Add picker support:
   - picker lists all dialog widgets in the current project/folder context
   - display should include dialog name and containing screen/folder context
   - picker must not list normal screens
8. Add overlay runtime behavior:
   - `OpenDialog(dialogWidgetId, origin = Screen, position = Center)` opens the selected dialog widget as an internal overlay
   - opening the same dialog again brings/replaces the existing overlay instead of creating uncontrolled duplicates
   - `CloseDialog(dialogWidgetId)` closes the matching overlay if open
   - overlay placement supports at least `Screen + Center`
   - non-default `origin` and `position` values may be parsed/stored even if only the MVP placement behavior is fully implemented
9. Add editor workflow:
   - developer can place a `DialogWidget` on any screen through the existing widget picker
   - the containing screen is only the design-time organization location
   - designer can edit the dialog widget directly where it is placed
   - live testing can open the widget through `OpenDialog`
10. Add focused tests:
   - old dialog-screen metadata is no longer emitted
   - normal screen YAML remains scalar-compatible
   - dialog widget round-trips through layout persistence
   - dialog widget picker lists only dialog widgets
   - `OpenDialog` uses widget ids and keeps one overlay per dialog widget
   - placement defaults are applied when omitted
11. Update documentation:
   - remove dialog-screen wording
   - document `DialogWidget`
   - document `OpenDialog(dialogWidgetId, origin = Screen, position = Center)`
   - document `CloseDialog(dialogWidgetId)`

## Implementation Order
1. Create a checkpoint by reviewing `git status` and identifying unrelated user changes.
2. Remove old dialog-screen tests first or mark the expected new tests, so stale behavior is visible.
3. Remove `ScreenKind` persistence from loaders, layout documents, `PageModel`, and screen serialization.
4. Remove old dialog-screen UI affordances from `MainWindow.axaml.cs` and any menu or toggle logic.
5. Remove old `DialogScreenOverlayEntry` and old overlay binding/state from `MainWindow.axaml` and `MainWindowViewModel`.
6. Ensure the solution builds back to the pre-dialog-screen screen model before adding `DialogWidget`.
7. Add `DialogWidget` as a new control kind and wire it through widget creation, selection, serialization, and rendering paths.
8. Add stable dialog widget identity and project-wide discovery.
9. Rewire `OpenDialog`/`CloseDialog` to target dialog widgets.
10. Add placement argument support and defaults for `OpenDialog`.
11. Add the new internal overlay host/state for dialog widgets.
12. Add tests for persistence, picker filtering, action execution, and placement defaults.
13. Update widget documentation and short descriptions.
14. Run build/tests and manually verify the editor workflow.

## Technical Constraints
- Do not keep two parallel dialog concepts.
- Do not implement a compatibility layer for dialog screens.
- Do not reserve or special-case any screen id.
- `DialogWidget` may be placed on any screen; its containing screen is only the design-time organization location.
- Do not introduce new dependencies.
- Use existing widget, picker, interaction-rule, Avalonia, and MVVM patterns.
- Keep the MVP as an internal overlay; no detached `Window`, no second-monitor placement, no movable external dialog window.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Public new types/members require XML documentation.
- Update matching widget docs under `src/HornetStudio/docs/widgets/`, `descriptions/`, and `help/`.

## Acceptance Criteria
- No `ScreenKind.Dialog`, dialog-screen picker, or dialog-screen overlay behavior remains.
- Screen YAML is not expanded with `Kind: Dialog` metadata.
- Existing normal projects load as normal screens.
- `DialogWidget` is available in the existing widget picker.
- A `DialogWidget` can be placed on any screen and saved/reloaded.
- `OpenDialog` picker lists dialog widgets, not screens.
- Picker display includes enough context to distinguish dialogs with similar names, such as containing screen/folder.
- `OpenDialog(dialogWidgetId)` opens the dialog widget as an internal overlay centered on the screen by default.
- `OpenDialog(dialogWidgetId, origin, position)` stores/parses placement options and applies supported MVP placement behavior.
- `CloseDialog(dialogWidgetId)` closes the matching overlay.
- Reopening the same dialog widget does not create duplicate overlays.
- Documentation describes the widget-based dialog model and no longer describes dialog screens as the target model.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore has already completed.
- If restore is required, run `dotnet restore HornetStudio.sln` first.
- Run `dotnet src/HornetStudio.Editor.Tests/bin/Debug/net*/HornetStudio.Editor.Tests.dll` or the currently valid self-test command after building.
- Manually verify adding a `DialogWidget` through the widget picker.
- Manually verify saving and reloading a project containing a `DialogWidget`.
- Manually verify a button interaction can select a dialog widget through `OpenDialog`.
- Manually verify `OpenDialog` opens an overlay and `CloseDialog` closes it.
- Manually verify an existing project without any dialog feature still loads with unchanged normal screens.

## Out of Scope
- Migrating old dialog-screen projects.
- Supporting `ScreenKind = Dialog` as legacy behavior.
- Detached Avalonia windows for dialogs.
- Moving dialogs between monitors.
- `AllowMove`, `AllowResize`, `TopMost`, `RememberPlacement`, or persistent placement.
- Dialog return values.
- Parameter passing into dialog widgets beyond the placement options.
- A separate dialog manager menu or dialog-specific screen tree.
- Refactoring unrelated widget architecture.

## Risks / Watchouts
- `OpenDialog` and `CloseDialog` already exist in the codebase; changing their target semantics must update all picker, codec, execution, test, and documentation paths together.
- If item ids are not currently stable across save/load, dialog references need a focused stable id solution before action references are reliable.
- `DialogWidget` should reuse existing layout/container mechanics where possible; avoid creating a separate nested layout engine unless the current widget model requires it.
- Overlay rendering must not accidentally render the original design-time widget twice in runtime views.
- Removing screen-kind support may require careful cleanup across both `ProjectUiLayoutLoader` and `BookUiLayoutLoader`.
- Documentation must avoid implying that a specific screen number such as `100` has any technical meaning.
