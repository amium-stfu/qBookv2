# IMPLEMENTATION HANDOFF

## Goal
Add custom user dialogs as designable dialog screens that stay inside the existing screen and folder model. Developers should be able to create, organize, edit, and live-test dialogs without a separate menu or a detached dialog subsystem.

## Scope
- Add a dialog classification for screens, for example `ScreenKind: Page | Dialog`, using the existing screen/folder model.
- Keep dialog entries represented in `Folder.yaml` and related persisted folder/layout data.
- Keep existing user-level based accessibility behavior.
- Add runtime actions/functions for opening and closing dialog screens:
  - `OpenDialog(screenId)`
  - `CloseDialog(screenId)`
- Add a dialog picker for these functions that only offers screens classified as dialogs.
- Render opened dialogs as application-internal overlays in the MVP.
- Allow direct design/edit access to dialog screens through the existing screen tree.
- Allow live preview/testing of dialog behavior from the editor.

## Starting Point
- Screens are currently represented under `Screens:` in `Folder.yaml`.
- Widgets reference screens through `Screen`/`View` properties.
- `MainWindowViewModel` contains screen serialization, screen option binding, user-level behavior, and editor command/action setup.
- `ProjectUiLayoutLoader` and `BookUiLayoutLoader` parse and emit screen/view layout data.
- `FolderEditorControl` exposes folder/screen editing behavior and widget selection/editor UI.
- Existing Avalonia dialog windows are implementation/editor dialogs and should not be confused with user-designed runtime dialogs.

## Tasks
1. Inspect the current screen/folder data model and determine the least invasive place to store screen kind metadata.
2. Add a persisted screen classification that distinguishes normal page screens from dialog screens.
3. Keep backward compatibility for existing projects where no screen kind metadata exists; those screens must remain normal page screens.
4. Update folder/layout loading and saving so dialog screen metadata round-trips through `Folder.yaml`.
5. Update the editor screen/folder UI so dialog screens are visible in the normal screen hierarchy with a clear visual distinction.
6. Add a creation/editing path for dialog screens that does not require a separate top-level menu.
7. Add or extend picker behavior so `OpenDialog` and `CloseDialog` can select only dialog screens.
8. Add runtime dialog overlay management:
   - one open dialog instance per dialog screen id
   - `OpenDialog` opens or brings the matching overlay forward
   - `CloseDialog` closes the matching overlay if open
9. Add editor live-test behavior so a dialog screen can be previewed as an overlay while still being directly editable as a screen.
10. Add focused tests for serialization/backward compatibility and dialog picker filtering where test infrastructure exists.
11. Update relevant widget/action documentation if action/function definitions are changed.

## Implementation Order
1. Read the current screen persistence and option binding flow in `MainWindowViewModel`, `ProjectUiLayoutLoader`, `BookUiLayoutLoader`, and `FolderEditorControl`.
2. Define the persisted metadata shape for dialog screens. Prefer explicit metadata over reserved id ranges.
3. Implement parsing defaults so missing metadata maps to normal page screens.
4. Implement saving so dialog metadata round-trips without rewriting unrelated layout data more than the existing serializer already does.
5. Add editor UI support for marking/creating a screen as a dialog in the existing screen/folder workflow.
6. Add filtered dialog screen options for `OpenDialog` and `CloseDialog`.
7. Add the runtime overlay host and dialog state management.
8. Add editor preview/open behavior for dialog screens.
9. Add or update focused tests.
10. Run verification and update documentation only where behavior or user-visible actions changed.

## Technical Constraints
- Chat responses and documentation discussion stay German; code, comments, XML documentation, UI text, file names, and technical identifiers stay English.
- Do not introduce new dependencies.
- Do not use `screenId < 100` or any other reserved id range as dialog classification.
- Keep normal screen behavior unchanged for existing projects.
- Keep dialog screens inside the existing folder/screen structure.
- Use existing Avalonia, MVVM, serialization, picker, and action patterns.
- MVP uses internal overlays only; no detached windows, no multi-monitor placement, and no movable external dialog window.
- Keep public APIs documented with XML documentation where new public types or members are introduced.

## Acceptance Criteria
- Existing projects without dialog metadata load with all existing screens as normal page screens.
- A developer can create or mark a screen as a dialog without using a separate top-level dialog menu.
- Dialog screens are visible and directly editable in the existing screen/folder workflow.
- Dialog screens are persisted in `Folder.yaml` or the matching layout data and survive reload.
- `OpenDialog(screenId)` and `CloseDialog(screenId)` are available through the action/function editor.
- The dialog picker only lists screens classified as dialogs.
- `OpenDialog` shows the selected dialog screen as an overlay above the current UI.
- `CloseDialog` removes the selected dialog overlay.
- Opening the same dialog repeatedly does not create uncontrolled duplicate overlays.
- User-level based accessibility remains compatible with existing behavior.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore has already completed.
- If restore is required, run `dotnet restore HornetStudio.sln` first.
- Run the focused editor self-tests if available through the existing test workflow.
- Manually verify loading an existing project with no dialog metadata.
- Manually verify creating/marking a dialog screen, saving, reloading, and confirming the metadata remains present.
- Manually verify `OpenDialog` and `CloseDialog` from an interaction/action using the filtered picker.
- Manually verify design-time direct editing and live overlay preview.

## Out of Scope
- Detached Avalonia `Window` hosting for user dialogs.
- Moving dialogs to a second monitor.
- `AllowMove`, `AllowResize`, `TopMost`, `RememberPlacement`, or display placement settings.
- Multiple simultaneous instances of the same dialog screen.
- Dialog return values.
- Parameter passing into dialogs.
- A separate dialog manager menu or independent dialog project tree.
- Refactoring unrelated widget, screen, or folder architecture.

## Risks / Watchouts
- The current code may use `Screens` as a simple `Dictionary<int, string>`. Adding metadata may require a backward-compatible YAML shape that still accepts the old scalar format.
- Existing action pickers may be generic screen pickers; filter behavior should not break normal screen selection elsewhere.
- Overlay rendering must not interfere with normal screen navigation or editor selection state.
- User-level filtering must be checked carefully so dialog screens are neither hidden from developers unintentionally nor exposed to runtime users beyond existing rules.
- Avoid coupling the MVP overlay implementation to future detached-window behavior.

## Notes
- Prefer the name `ScreenKind` with values `Page` and `Dialog` unless the existing codebase already has a better matching naming pattern.
- The old scalar YAML shape should remain valid:
  - `1: 'HomeScreen'`
- A possible new YAML shape can be evaluated during implementation, for example:
  - `1:`
  - `  Name: 'AlarmDialog'`
  - `  Kind: 'Dialog'`
- Keep the first implementation intentionally small. The architecture should leave room for later detached windows and movable monitor dialogs, but those capabilities are not part of this workitem.
