# IMPLEMENTATION HANDOFF

## Goal
Simplify the Function Editor step presentation after the `IfThenElse` condition dialog work. Remove redundant summary/detail display in editable steps and render each step as one compact editable row wherever practical:

```text
[Type] [Primary Parameter] [Attribute] [Value/Text/Condition] [Up] [Down] [Delete]
```

The editor should make individual step boundaries easier to see through a subtle row background or border, while keeping nested `If` / `Else` branches compact and readable.

## Scope
- Included:
  - Remove redundant summary text for editable steps when the same data is already shown in editable controls.
  - Convert top-level and branch step rows to a compact single-row layout where practical.
  - Keep `Log`, `SetValue`, `Delay`, and `IfThenElse` rows visually consistent.
  - Improve individual step boundary visibility with a subtle background, border, or left accent.
  - Keep `IfThenElse` as a compact controller row with Condition dialog button and branch counts.
  - Keep `If Block` and `Else Block` branch groups lightly nested and readable.
  - Preserve all existing editing behavior, YAML schema, runtime behavior, and condition dialog behavior.
  - Update Functions docs/help only if they describe the editor layout.
- Not included:
  - New workflow functionality.
  - Changing `IfThenElse` condition schema or runtime evaluation.
  - Reworking the Monitor condition editor.
  - Visual canvas, drag/drop, or full table component rewrite.
  - Changing Python, Button, or Action Rule behavior.

## Starting Point
- The shared Boolean Condition Editor and editable `IfThenElse` implementation already exists.
- A follow-up plan moved `IfThenElse` condition editing into a dialog.
- Current step rows still show redundant information:
  - a summary line such as `Log Info -> studio.main.logs.my_log : Greater than 100`
  - followed by controls containing the same target, level, and text
- The extra summary/detail split makes rows taller than needed.
- Individual step boundaries are hard to see, especially in nested branches and dark theme.
- Desired examples:
  ```text
  [Log]      [studio.main.logs.my_log]        [Info]        [Greater than 100] [Up] [Down] [Delete]
  [SetValue] [enhanced_signals.filtered_1.set]              [0]                [Up] [Down] [Delete]
  [Delay]    [1000 ms]                                                        [Up] [Down] [Delete]
  [IfThenElse] [Condition: {A} > 100] [Then: 1] [Else: 1] [Condition]          [Up] [Down] [Delete]
  ```

## Tasks
1. Inspect current workflow editor row XAML:
   - Find where summary text is shown for editable rows.
   - Find layout differences between top-level rows and branch rows.
   - Identify existing theme brushes for row background, border, hover, and input fields.
2. Remove redundant editable-row summaries:
   - Do not show summary text for `Log`, `SetValue`, and `Delay` when their editable fields are visible in the same row.
   - Keep concise summary only for preserved/unsupported read-only rows if such fallback still exists.
   - Keep `IfThenElse` condition/branch summary because its full condition details are in a dialog.
3. Define one compact row grid:
   - Suggested columns:
     - `Type`
     - `Primary`
     - `Attribute`
     - `Value`
     - `Actions`
   - Keep the grid flexible so step types do not need artificial fields.
   - For narrow widths, wrap predictably or let the value column shrink without overlapping buttons.
4. Map step types to row fields:
   - `Log`:
     - `Type`: step type selector
     - `Primary`: target log selector/input
     - `Attribute`: log level selector
     - `Value`: log text input
   - `SetValue`:
     - `Type`: step type selector
     - `Primary`: target path input or selector
     - `Attribute`: optional empty/spacer unless a meaningful attribute exists
     - `Value`: value input
   - `Delay`:
     - `Type`: step type selector
     - `Primary`: milliseconds input
     - `Attribute` / `Value`: empty/spacer unless needed
   - `IfThenElse`:
     - `Type`: step type selector or fixed label
     - `Primary`: condition summary
     - `Attribute`: `Then: <count>` and `Else: <count>` if layout fits
     - `Value`: `Condition` / `Add Condition` button, or place it before actions if clearer
5. Improve step boundary visibility:
   - Add a subtle row container background that differs from the parent.
   - Add a low-contrast border or left accent line.
   - Use consistent spacing and padding for every step row.
   - Avoid heavy nested cards or large section boxes.
6. Refine branch group layout:
   - Keep `If Block` and `Else Block` headings.
   - Put each branch `Add` controls in the heading row.
   - Render branch steps with the same compact row component/pattern as top-level steps.
   - Use slight indentation or a left guide line to show nesting.
   - Avoid repeating branch row summaries.
7. Preserve controls and behavior:
   - Existing browse/source-picker buttons must remain available where they exist today.
   - Existing move up/down/delete buttons must continue to work for top-level and branch steps.
   - Existing add step type pickers for top-level, `If Block`, and `Else Block` must continue to work.
   - Existing condition dialog button must continue to open and commit/cancel correctly.
8. Review responsive behavior:
   - At the current dialog width, common rows should stay one line.
   - Long target paths and log text should not overlap action buttons.
   - Use trimming, min/max widths, or flexible columns as appropriate.
9. Update docs only if needed:
   - If Functions help describes the row layout, update it to mention compact editable rows and condition dialog.
   - Do not update docs for purely cosmetic changes unless existing docs become inaccurate.
10. Verify manually and with build/tests:
   - Build solution.
   - Open Function Editor in dark and light theme if practical.
   - Confirm rows are compact and boundaries are clear.

## Out of Scope
- Replacing the editor with a table component.
- Drag/drop sorting.
- Inline validation redesign.
- Condition dialog changes.
- Monitor editor changes.
- Runtime or YAML schema changes.

## Risks / Watchouts
- Long paths can break one-line layout. Prefer flexible columns and text trimming rather than letting controls overlap.
- A fully table-like row can become cramped if all fields are forced to equal widths. Let `Primary` and `Value` take most space.
- Do not hide required fields just to keep one line. If a narrow viewport must wrap, wrap cleanly.
- Branch nesting can become visually noisy if each row is too card-like. Use subtle background/border, not heavy panels.
- If existing summary properties are used by read-only preserved rows, keep them for that fallback.
