# IMPLEMENTATION HANDOFF

## Goal
Refine the Function Editor step UI after the compact row work so it remains readable in Dark Theme and gives `IfThenElse` a clearer container/block presentation. Normal steps should stay compact editable rows. `IfThenElse` should be visually distinct as a branch container with a compact header row and nested `If Block` / `Else Block` sections.

## Scope
- Included:
  - Improve Dark Theme contrast for editable fields inside function step rows.
  - Make inputs, combo boxes, and buttons clearly distinguishable from row backgrounds.
  - Preserve theme-driven colors and existing theme brush patterns.
  - Make normal steps (`Log`, `SetValue`, `Delay`) compact rows.
  - Render `IfThenElse` as a compact container block:
    - header row with type, condition button/summary, branch summary, and row actions
    - nested `If Block` and `Else Block`
    - branch add controls in each branch header
    - nested steps using the same compact row pattern
  - Add subtle visual hierarchy:
    - step row boundary
    - left accent/guide for step rows
    - stronger container boundary for `IfThenElse`
    - light branch grouping without heavy nested cards
  - Keep condition editing in the existing Condition dialog.
  - Preserve YAML, runtime behavior, and condition editor behavior.
- Not included:
  - New workflow functionality.
  - Changing `IfThenElse` schema or execution behavior.
  - Changing the shared Boolean Condition Editor dialog, except if a theme brush fix is needed.
  - Reworking Monitor editor UI.
  - Visual canvas, drag/drop, or table component rewrite.
  - Python, Button, or Action Rule changes.

## Starting Point
- The `IfThenElse` condition editor has been moved out of inline row content into a dialog.
- The Function Editor now uses compact rows, but in Dark Theme some editable fields are hard to distinguish from the row background.
- The current `IfThenElse` row is functionally usable but visually feels inconsistent with its role as a container.
- User design direction:
  - normal rows are compact horizontal editable rows
  - `IfThenElse` is a larger bordered container with nested blocks
  - a vertical accent/guide on the left helps identify row nesting
  - branch headers have small `Type` + `Add` controls
  - colors and icons must follow the application theme, not the sketch colors

## Tasks
1. Inspect current Function Editor theme bindings:
   - Identify row background, row border, dialog background, section background, input background, input foreground, button background, and border brushes currently used.
   - Check whether the poor Dark Theme contrast comes from hard-coded colors, reused row colors on inputs, or missing style overrides.
2. Fix editable control contrast in Dark Theme:
   - Ensure `TextBox`, `ComboBox`, and action buttons inside step rows use a control/input background that is visibly different from the row background.
   - Ensure borders are visible in normal, hover, and focused states.
   - Ensure dropdown arrows and button icons use theme foreground colors.
   - Avoid hard-coded colors unless the surrounding code already uses theme-backed fallback values.
3. Define visual levels:
   - Editor content background.
   - Step list background.
   - Normal step row background/border.
   - `IfThenElse` container background/border.
   - Branch section background/border or left guide.
   - Input/control background/border.
4. Keep normal steps as compact rows:
   - `Log`: type, target log, level, text, actions.
   - `SetValue`: type, target, optional browse, value, actions.
   - `Delay`: type, milliseconds, unit label, actions.
   - Do not reintroduce duplicate summary rows for editable normal steps.
5. Redesign `IfThenElse` as a container block:
   - Header row:
     ```text
     [IfThenElse] [Condition/Add Condition] [Condition summary or branch summary] [Up] [Down] [Delete]
     ```
   - Body:
     ```text
     If Block                                      [Type] [Add]
       [Log] [target] [Info] [text] [Up] [Down] [Delete]

     Else Block                                    [Type] [Add]
       [Log] [target] [Info] [text] [Up] [Down] [Delete]
     ```
   - Keep the condition details in the Condition dialog, not inline.
6. Add or refine left accents/guides:
   - Normal rows may have a small vertical accent strip on the left.
   - `IfThenElse` container may have a stronger left guide.
   - Nested branch rows should be slightly indented and keep their own small row guide.
   - Use theme-compatible colors or existing accent brushes.
7. Refine branch headers:
   - Keep branch labels (`If Block`, `Else Block`) visually clear.
   - Place branch add controls on the same line.
   - Keep branch headers smaller/lighter than the `IfThenElse` header.
   - Avoid large nested cards.
8. Review action buttons:
   - Keep Up/Down/Delete compact and consistently aligned.
   - Use theme-colored icons.
   - If text buttons exist for these actions, prefer existing icon-button pattern if the project already has one.
9. Preserve behavior:
   - Top-level Add Step still works.
   - Branch Add still works.
   - Move/delete still works at top-level and branch level.
   - Condition dialog still opens, saves, cancels, and updates the summary.
   - Save/reopen preserves data.
10. Update docs only if needed:
   - If docs mention inline layout or row behavior, update them.
   - Do not document purely visual theme adjustments unless existing docs become inaccurate.
11. Verify Dark and Light Theme manually:
   - In Dark Theme, inputs must be clearly visible against row backgrounds.
   - In Light Theme, contrast must remain acceptable.

## Technical Constraints

- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.

- Do not introduce new dependencies.
- Preserve existing Avalonia/MVVM patterns.

## Acceptance Criteria
- In Dark Theme, `TextBox` and `ComboBox` fields inside Function Editor step rows are clearly visible.
- In Light Theme, fields remain readable and not overly heavy.
- Normal steps remain compact editable rows without duplicate summary rows.
- `IfThenElse` is visually recognizable as a container, not just another flat row.
- `IfThenElse` header stays compact and contains condition access, summary, and row actions.
- `If Block` and `Else Block` are clearly nested inside the `IfThenElse` container.
- Nested branch steps use the same compact row language as top-level steps.
- Step boundaries are easy to recognize through subtle background, border, or left accent.
- Existing add, move, delete, condition dialog, save, and reload behavior remains intact.
- Build passes.

## Verification
- Build:
  - `dotnet build HornetStudio.sln --no-restore`
  - If a running app locks output, use the established project-level or separate `OutDir` fallback.
- Manual:
  - Open Function Editor in Dark Theme.
  - Add `Delay`, `Log`, `SetValue`, and `IfThenElse`.
  - Confirm editable fields are visible against each row.
  - Add one or more steps inside `If Block` and `Else Block`.
  - Confirm nested steps are readable and clearly bounded.
  - Open and save/cancel the Condition dialog.
  - Move and delete top-level and branch steps.
  - Save/reopen the function and confirm data is unchanged.
  - Repeat a quick visual check in Light Theme if available.

## Out of Scope
- Runtime changes.
- YAML schema changes.
- New condition editor capabilities.
- Monitor layout redesign.
- Drag/drop or canvas editing.
- Replacing the editor with a full table/grid component.

## Risks / Watchouts
- Dark Theme fixes must not make Light Theme look too heavy. Use theme-aware brushes rather than one-off dark colors.
- `IfThenElse` needs stronger visual weight than a normal row, but avoid large nested card structures.
- Long target paths can still crowd compact rows. Ensure action buttons never overlap field text.
- If icon buttons are changed, keep accessibility/tooltips consistent with existing UI patterns.
- Do not move condition editing back inline while improving the container layout.
