# IMPLEMENTATION HANDOFF

## Goal
Refine the implemented shared Boolean Condition Editor integration so `IfThenElse` remains compact in the Function Editor. Move condition editing out of the inline step body into a dedicated Condition dialog, fix the variable add button, and keep branch editing readable with mostly one-line step rows.

## Scope
- Included:
  - Replace the large inline `IfThenElse` condition editor in the Function Editor with a compact summary and an `Add Condition` / `Condition` button.
  - Open the shared Boolean Condition Editor in a dedicated Condition dialog for `IfThenElse`.
  - Fix the `+` button so condition variables can be added reliably.
  - Keep `IfThenElse` rows compact and scannable.
  - Render `If` and `Else` branches as compact step lists with local `Add` controls.
  - Keep step entries as one line wherever practical.
  - Preserve existing YAML shape and runtime behavior from the previous implementation.
  - Update docs/help only where the user-facing editor behavior changed.
  - Add focused regression coverage for variable addition and condition dialog round-trip where feasible.
- Not included:
  - Reworking Monitor rule layout unless the shared editor fix requires it.
  - Visual canvas workflow editing.
  - New condition syntax.
  - Function-wide variables.
  - Python, Button, or Action Rule changes.
  - Broad redesign of all workflow step editors.

## Starting Point
- The first implementation of the shared Boolean Condition Editor and editable `IfThenElse` is already present.
- Current Function Editor issue:
  - `IfThenElse` embeds the full condition editor inline.
  - The result is visually too large and hard to scan.
  - The variable `+` button in the shared editor does not add variables in at least the `IfThenElse` context.
- Desired user-facing structure:
  ```text
  IfThenElse    Condition: {A} == true    Then: 0    Else: 0    [Condition] [Up] [Down] [Delete]

  If Block      [Add]
    - Log       ...

  Else Block    [Add]
    - Delay     ...
  ```
- Alternative compact first state:
  ```text
  IfThenElse    No condition configured    Then: 0    Else: 0    [Add Condition]
  If Block      [Add]
  Else Block    [Add]
  ```

## Acceptance Criteria
- Clicking `+` in the condition editor adds a variable row.
- In the Function Editor, `IfThenElse` no longer shows the full condition editor inline.
- `IfThenElse` shows a compact condition summary and branch counts.
- `Add Condition` / `Condition` opens a dialog with the shared Boolean Condition Editor.
- Saving the Condition dialog updates the `IfThenElse` row summary and persisted condition data.
- Canceling the Condition dialog leaves the existing condition unchanged.
- `If Block` and `Else Block` can add steps and remain readable.
- Existing `IfThenElse` YAML still loads and saves.
- Monitor custom condition editing still works if it uses the shared editor.
- Build and focused tests pass.

## Out of Scope
- Replacing the whole Function Editor with a different editor pattern.
- Visual workflow canvas.
- Full drag/drop nesting.
- New expression functions or syntax.
- Function-wide variable scope.
- Runtime behavior changes not required by the UI fix.
- Python or interaction rule changes.

## Risks / Watchouts
- Avalonia `DataContext` inheritance can break inside reusable controls. Prefer explicit bindings or commands that are owned by the condition editor view-model.
- Dialog editing must clone state first. Directly editing the row and reverting on Cancel is brittle.
- Keep the condition dialog generic. Do not introduce references to `FunctionStepEditorRow` inside the shared condition editor itself.
- Avoid nested cards and large fixed heights; the screenshot shows a lot of wasted vertical space.
- Branch rows should remain compact, but do not silently remove fields needed to edit existing step types.
- If compact one-line editing conflicts with existing inline editing for non-conditional steps, limit this handoff to `IfThenElse` layout and document a follow-up.
