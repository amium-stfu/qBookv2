# IMPLEMENTATION HANDOFF

## Goal
Introduce a reusable shared Boolean Condition Editor and use it as the foundation for editable `IfThenElse` function steps in the Functions/Workflow editor. The editor should reuse the existing Monitor formula experience where appropriate: variables, source-path picking, operator buttons, variable token buttons, formula text editing, and live boolean validation.

## Scope
- Included:
  - Extract the reusable condition-editing behavior from the Monitor rule editor without moving Monitor-specific rule behavior.
  - Provide a shared condition editor/view-model that can be used by Monitor now and Functions/Workflow next.
  - Keep one formula language by continuing to use `CustomSignalFormulaEngine.TryEvaluateBooleanExpression(...)`.
  - Extend `IfThenElse` function steps so they can carry condition variables in addition to the existing `condition` string.
  - Preserve old `IfThenElse` YAML definitions that only contain `condition`, `then`, and optional `else`.
  - Make `IfThenElse` editable in the Functions/Workflow step editor after the shared condition editor exists.
  - Support nested `then` and `else` step lists using the existing function step editing model as much as practical.
  - Update widget documentation/help and focused tests.
- Not included:
  - Creating a visual workflow canvas.
  - Introducing a second expression language or generic rules engine.
  - Changing Monitor runtime semantics such as timeout, inhibit, event metadata, actions, or logging.
  - Function-wide/global variable scoping.
  - Function return values, typed argument schemas, or Python changes.
  - Broad UI redesign of the Functions widget.

## Starting Point
- `CustomSignalFormulaEngine` already evaluates formulas and boolean conditions.
- Monitor rule custom mode already has the desired formula UX:
  - variable rows with short names and source paths
  - source picker
  - formula text box
  - operator buttons
  - variable token buttons
  - live validation status
- `FunctionExecutor` already supports `FunctionIfThenElseStepDefinition` and evaluates `Condition` through `CustomSignalFormulaEngine` when no custom callback is provided.
- `FunctionDefinitionCodec` already parses and serializes `IfThenElse` with:
  - `type`
  - `condition`
  - `then`
  - optional `else`
- `FunctionStepEditorRow` currently preserves `IfThenElse` as read-only and rejects editing it.
- Current docs explicitly say existing `IfThenElse` steps are preserved read-only.

## Tasks
1. Identify the smallest reusable condition editor boundary:
   - Keep formula text, variables, token buttons, insertion behavior, and validation together.
   - Keep Monitor-specific fields outside the shared component.
2. Introduce a shared condition variable model:
   - Suggested model shape:
     - `Name`
     - `SourcePath`
   - Reuse or adapt the existing monitor/custom-signal variable view-model only if it does not create Monitor coupling.
3. Introduce a shared condition editor view-model:
   - Suggested name: `BooleanConditionEditorViewModel`
   - Responsibilities:
     - manage condition variables
     - generate variable buttons
     - expose operator buttons
     - insert formula tokens
     - validate with preview variables
     - build/persist the formula and variables
   - Validation messages must remain English.
4. Introduce a shared condition editor UI control or reusable XAML pattern:
   - Suggested location: `src/HornetStudio.Editor/Widgets/Common/`
   - It should expose the same practical UX as the Monitor screenshot.
   - It must support a host-provided source-picker action because Monitor and Functions may have different source option lists.
5. Refactor Monitor editor to use the shared condition editor:
   - Preserve existing Monitor YAML/model behavior:
     - `CustomFormula`
     - `CustomVariables`
   - Preserve existing validation behavior and messages unless a shared message is equivalent.
   - Keep Default mode behavior unchanged.
6. Extend function step models for step-local condition variables:
   - Add condition variables to `FunctionIfThenElseStepDefinition`.
   - Prefer a step-local YAML shape:
     ```yaml
     - type: IfThenElse
       condition: "{A} == true"
       variables:
         - name: A
           sourcePath: custom_signals_1.dummy_bool
       then:
         - type: Log
           targetLog: process_log_1
           text: Condition matched
       else:
         - type: Delay
           milliseconds: 1000
     ```
   - Old YAML without `variables` must continue to parse and serialize safely.
7. Update `FunctionDefinitionCodec`:
   - Parse optional `variables` for `IfThenElse`.
   - Serialize `variables` only when at least one variable exists.
   - Validate variable entries with clear English messages.
   - Preserve existing `condition`, `then`, and `else` semantics.
8. Update `FunctionExecutor`:
   - Resolve step-local condition variables before evaluating the `IfThenElse` condition.
   - Merge with existing `environment.ConditionVariables` carefully:
     - explicit runtime/environment variables should remain available
     - step-local variable names should provide values for formula tokens
     - reserved names such as `value`/`source` should be handled consistently with Monitor if reused
   - Add a clear failure when a configured source path cannot be resolved.
   - Prefer using existing item/value lookup paths already used by `SetValue` or Monitor instead of adding a new runtime dependency.
9. Make `IfThenElse` editable in `FunctionStepEditorRow`:
   - Add `IfThenElse` to editable step types after condition editing and branch editing are supported.
   - Add fields/state for condition editor, `Then` rows, and `Else` rows.
   - Preserve read-only fallback for unsupported nested content only if full editing cannot be completed safely in this scope.
10. Update the Functions/Workflow editor UI:
    - Show the shared condition editor when the selected step type is `IfThenElse`.
    - Show editable `Then` and `Else` branch lists.
    - Reuse existing Add/Edit/Delete row patterns where possible.
    - Avoid a canvas or deep redesign.
11. Update conversion and cloning:
    - `FunctionEditorDefinitionConverter.CreateRows(...)` must populate editable condition and branch state.
    - `TryBuildDefinition(...)` must build nested `IfThenElse` definitions.
    - `CloneStep(...)` must deep-clone condition variables and branch steps.
12. Update tests:
    - Formula editor validation remains compatible with Monitor behavior.
    - `IfThenElse` YAML without `variables` still parses.
    - `IfThenElse` YAML with `variables` parses and serializes.
    - Editable row conversion round-trips `IfThenElse` with branches.
    - Executor chooses the correct branch when condition variables resolve to true/false.
13. Update documentation:
    - Monitor docs should still describe custom formulas, now backed by shared editor behavior.
    - Functions/Workflow docs should remove the "read-only IfThenElse" limitation when implemented.
    - Help docs should document `IfThenElse.variables`, formula syntax, and examples.

## Implementation Order
1. Inspect current Monitor formula UI and isolate the exact shared responsibilities.
2. Add the shared condition variable model/view-model with no Monitor behavior.
3. Add or extract the shared condition editor UI.
4. Move Monitor editor to the shared condition editor and verify no Monitor behavior changes.
5. Extend `FunctionIfThenElseStepDefinition` with step-local variables.
6. Update YAML parse/serialize and clone paths for `IfThenElse.variables`.
7. Add executor support for resolving step-local condition variables.
8. Update function step editor rows to represent editable `IfThenElse` state.
9. Update the Functions/Workflow editor UI for condition and branch editing.
10. Update conversion/build validation for nested branches.
11. Add focused tests.
12. Update docs/help/description files.
13. Run build and focused verification.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation/error messages must be English.
- Do not introduce new dependencies.
- Keep `CustomSignalFormulaEngine` as the single condition/formula engine.
- Prefer existing Avalonia, MVVM, and widget patterns.
- Keep the shared component focused on boolean conditions; do not turn it into a generic rules engine.
- Avoid broad architecture changes outside Monitor and Functions/Workflow condition editing.
- Preserve old persisted YAML and existing Monitor definitions.
- Public classes and public methods require XML documentation.
- Update matching widget docs when widget code changes.

## Acceptance Criteria
- Monitor custom formula editing still works with the same observable behavior as before.
- The reusable Boolean Condition Editor can be used outside Monitor without referencing Monitor-specific types.
- `IfThenElse` steps can be edited in the Functions/Workflow editor instead of only being preserved read-only.
- `IfThenElse` supports step-local variables with source paths.
- Existing `IfThenElse` YAML without `variables` still loads, preserves branches, and executes.
- New `IfThenElse` YAML with `variables` round-trips through load/save.
- The executor evaluates conditions using resolved variable values and selects the correct branch.
- Invalid formulas, duplicate/invalid variable names, and missing source paths produce clear English errors.
- Functions/Workflow docs no longer claim `IfThenElse` is read-only after the editor supports it.
- Build and focused tests pass.

## Verification
- Build:
  - `dotnet build HornetStudio.sln --no-restore`
  - If a running app locks output, use the established project-level or separate `OutDir` fallback.
- Focused tests:
  - Run existing editor self-tests if available.
  - Add/run parser, serializer, converter, and executor tests for `IfThenElse.variables`.
- Manual:
  - Open a Monitor rule in Custom mode and confirm variables, formula token insertion, validation, save, and reload still work.
  - Create or edit a Function with an `IfThenElse` step.
  - Add a condition variable, select a source path, enter a formula, add `Then` and `Else` steps, save, reload, and confirm the UI remains editable.
  - Execute a function where the condition is true and confirm `Then` runs.
  - Execute a function where the condition is false and confirm `Else` runs.
  - Open an older `IfThenElse` YAML without variables and confirm it still loads.

## Out of Scope
- Visual workflow/canvas editor.
- Drag/drop branch editing unless it already exists and is trivial to reuse.
- Function-wide condition variable scope.
- New runtime expression syntax.
- Python runtime changes.
- Button/Action Rule changes.
- Release versioning or publishing.

## Risks / Watchouts
- Extracting too much from Monitor can accidentally pull Monitor-specific concepts into shared code. Keep the boundary limited to formula/variables/validation.
- Nested `IfThenElse` editing can become visually dense. Use simple branch lists first.
- Step-local condition variables require runtime value lookup. Reuse an existing value-resolution path; do not create a parallel item access layer unless required.
- `environment.ConditionVariables` already exists. Define merge precedence explicitly before implementation.
- Old YAML must remain readable. Optional `variables` parsing should default to an empty list.
- Avoid serializing empty `variables` blocks to reduce churn in existing files.
- If full nested branch editing becomes too large, stop after shared editor + schema support and keep unsupported nested content preserved with a clear follow-up.
