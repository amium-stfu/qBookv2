# IMPLEMENTATION HANDOFF

## Goal
Add a first MVP `WorkflowWidget` that lets users build and run simple declarative workflows from YAML files under the folder-local `Scripts/Workflows/` directory. Workflows are ordered step lists, not a scripting language. Complex logic remains in Python applications and is out of scope.

## Scope
- Included:
  - Add a new `WorkflowWidget` widget type to the editor and runtime UI.
  - Store workflow definitions as separate YAML files under the current folder's `Scripts/Workflows/` directory.
  - Keep the normal `Folder.yaml` widget entry small and UI-focused.
  - Load, list, start, stop, and refresh available workflow YAML files.
  - Track runtime state per workflow: `Idle`, `Running`, `Stopping`, `Done`, `Failed`, `Canceled`.
  - Implement MVP step types:
    - `SetValue`
    - `Delay`
    - `IfThenElse`
    - `Log`
  - Provide an editor experience for creating and editing simple workflow YAML definitions.
  - Add widget documentation, picker description, detailed help, and widget index entries.
  - Add focused tests for parsing, validation, state transitions, cancellation, and step execution behavior where test seams already exist or can be introduced locally.
- Not included:
  - Python integration changes.
  - `RunWorkflow`, `StartWorkflow`, `WaitWorkflowDone`, child workflow orchestration, or workflow dependencies.
  - Loops, variables, return values, parameters, parallel execution, or a general scripting/expression language.
  - Persisting runtime state back into workflow YAML files.
  - New external dependencies unless an existing YAML/persistence helper cannot support the needed shape.
  - Broad refactoring of existing widget infrastructure.

## Starting Point
- Folder directories can already contain `Scripts/` next to `Folder.yaml`.
- Python applications are stored elsewhere and are not part of this change.
- Existing widget registration and rendering patterns are centered around `ControlKind`, `PageItemModel` boolean helpers, editor XAML hosts, picker entries, and widget documentation files.
- `Monitor` already has action concepts such as `SetValue` and `WriteLog`; reuse its semantics and implementation patterns where practical without coupling workflow execution to monitor rules.
- `CustomSignals` and `EnhancedSignals` already provide examples for formula/expression UI concepts; reuse existing expression patterns for `IfThenElse` conditions if needed.
- `LogControl` and logger-related widgets already exist; the `Log` workflow step should integrate with existing log target resolution instead of inventing a separate logging sink.

## Tasks
1. Identify existing helpers for folder path resolution, YAML parsing/writing, target item selection, log selection, expression evaluation, and action value conversion.
2. Add the `WorkflowWidget` kind to widget registration, selection, rendering, persistence mapping, default size logic, and preview creation.
3. Add workflow model types:
   - `WorkflowDefinition`
   - `WorkflowStepDefinition`
   - concrete step definitions for `SetValue`, `Delay`, `IfThenElse`, and `Log`
   - `WorkflowState`
   - validation result/error types if existing validation patterns are insufficient
4. Add a workflow YAML loader/saver scoped to the current folder's `Scripts/Workflows/` directory.
5. Define and document the MVP YAML schema.
6. Add a workflow runtime/executor that:
   - runs steps sequentially with `async`/`await`
   - supports cancellation
   - updates per-workflow state and error text
   - never writes runtime state to YAML
7. Implement `SetValue` by reusing existing item resolution and write conversion behavior where possible.
8. Implement `Delay` with cancellable `Task.Delay`.
9. Implement `IfThenElse` with a boolean condition and nested `then`/`else` step lists.
10. Implement `Log` with target log picker support, level, and text. Missing explicit log targets should fail clearly unless a documented default process log can be resolved.
11. Build the `WorkflowWidget` UI:
    - compact list of workflow files/definitions
    - state indicator
    - start/stop controls
    - refresh control
    - edit/create/delete actions if consistent with nearby widget patterns
12. Build a workflow editor dialog:
    - workflow name
    - ordered step list
    - add/edit/delete/reorder step actions
    - step-specific editors
    - path picker for `SetValue`
    - log picker, level selector, and text input for `Log`
13. Add documentation files for summary, picker description, and detailed help.
14. Add tests for model parsing, validation, executor ordering, cancellation, and failure behavior.
15. Build the solution and manually inspect the widget in a folder that contains multiple workflow YAML files.

## Technical Constraints
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Chat and final status must be German.
- Use existing Avalonia patterns and widget infrastructure.
- Use existing YAML/persistence libraries already present in the project.
- Do not add new external dependencies without a clear reason and FOSS/license review.
- Public classes and public methods require XML documentation.
- Keep `Folder.yaml` from storing full workflow step lists; workflow definitions live in separate files.
- Runtime state is in memory or runtime registry only; it must not be persisted to workflow YAML.
- Use `async`/`await`; do not block with `.Result` or `.Wait()`.
- User-visible validation and error messages must be English.
- Prefer named parameters for unclear or multi-argument calls.

## Acceptance Criteria
- A user can add a `WorkflowWidget` to a folder.
- The widget discovers workflow files from `Scripts/Workflows/`.
- The widget can create or edit a simple workflow YAML file.
- The widget lists workflows with visible state.
- A workflow with `Log`, `SetValue`, `Delay`, and `IfThenElse` runs in step order.
- Stop/cancel changes a running workflow through `Stopping` to `Canceled`.
- Successful completion changes state to `Done`.
- Failed target resolution, invalid condition, invalid delay, or missing explicit log target produces `Failed` with a useful error message.
- Reloading the folder preserves widget configuration and workflow files without embedding workflow step definitions into `Folder.yaml`.
- Documentation describes the workflow file location, YAML schema, supported steps, states, and limitations.

## Verification
- Run focused tests for workflow codec and executor.
- Run `dotnet build HornetStudio.sln --no-restore` after restore has already completed.
- If restore is required, run `dotnet restore HornetStudio.sln` first.
- Manually create a folder containing:
  - `Scripts/Workflows/start_sequence.yaml`
  - one `WorkflowWidget`
- Verify:
  - workflow files are discovered
  - `Log` writes to the selected log target
  - `SetValue` writes to the configured target
  - `Delay` is cancellable
  - `IfThenElse` chooses the expected branch
  - stop during delay results in `Canceled`
  - invalid YAML or missing targets show clear errors
  - saving/reloading does not inline workflows into `Folder.yaml`

## Out of Scope
- Workflow-to-workflow calls such as `RunWorkflow`, `StartWorkflow`, and `WaitWorkflowDone`.
- Any Python application folder or Python execution changes.
- Parallel workflows started by one workflow.
- Loops, variables, return values, parameters, local scoped data, or a custom scripting language.
- Full debugger, breakpoints, or step-through execution.
- Runtime state persistence.
- Cross-folder shared workflow libraries.

## Risks / Watchouts
- `IfThenElse` can become a scripting language entry point if the expression surface is expanded too far. Keep it boolean and reuse existing expression mechanisms.
- `SetValue` should behave consistently with existing Monitor action writes; avoid introducing a second incompatible conversion policy.
- Missing log targets must not silently drop messages.
- Nested `IfThenElse` steps can make YAML hard to edit. Keep the editor simple and validation strong.
- Creating workflow files beside `Folder.yaml` affects project file layout; ensure relative path handling is stable when folders are moved.
- Multiple simultaneous starts of the same workflow need a clear rule. MVP should reject starting an already running workflow.
- Stopping should cancel only the selected workflow instance.

## Notes
- Recommended workflow file path:
  - `Scripts/Workflows/<workflow-name>.yaml`
- Recommended minimal YAML shape:
  - top-level `name`
  - top-level `steps`
  - each step has `type`
  - `IfThenElse` has `condition`, `then`, and optional `else`
- Recommended `Log` fields:
  - `targetLog`
  - `level`
  - `text`
- Recommended `SetValue` fields:
  - `target`
  - `value`
- Recommended `Delay` field:
  - `milliseconds`
- Treat workflows as user-maintained project artifacts, not generated transient files.
