# IMPLEMENTATION HANDOFF

## Goal
Add the first `WorkflowWidget` editor experience so users can create, edit, and delete workflow YAML files from the widget UI. The editor should follow the dense dialog style of `InteractionRulesEditorDialogWindow` and support a practical MVP for flat workflows.

## Scope
- Included:
  - Add `Edit` and `Delete` actions per workflow row in `WorkflowControl`.
  - Change `Add` so it opens a workflow editor dialog instead of silently creating an empty YAML file.
  - Add `WorkflowEditorDialogWindow` styled like `InteractionRulesEditorDialogWindow`.
  - Edit workflow metadata:
    - workflow display `Name`
    - file name preview or editable safe file name, depending on implementation fit
  - Edit a flat ordered step list for:
    - `SetValue`
    - `Delay`
    - `Log`
  - Preserve existing `IfThenElse` steps when loading and saving existing YAML if possible.
  - Show unsupported or nested steps clearly instead of dropping them silently.
  - Use target picker support for `SetValue`.
  - Use process log target options for `Log`.
  - Save only the selected workflow YAML file under `Scripts/Workflows/`.
  - Refresh only the current `WorkflowWidget` list after create/edit/delete.
  - Update widget documentation/help if it currently implies file-only/manual editing.
- Not included:
  - A full branch editor for `IfThenElse`.
  - Visual node editor or canvas-based workflow design.
  - Workflow execution, start/stop state, or runtime behavior changes.
  - Workflow-to-workflow calls.
  - Loops, variables, parameters, return values, or scripting syntax.
  - Changing the project watcher or folder reload logic unless a bug is discovered during implementation.

## Starting Point
- `WorkflowWidget` currently discovers `.yaml` and `.yml` files below `Scripts/Workflows/`.
- `Add` currently creates a new YAML file directly.
- Existing workflow files are parsed through `WorkflowDefinitionCodec`.
- Existing model types are in `src/HornetStudio.Editor/Models/WorkflowDefinition.cs`.
- Existing codec and discovery logic are under `src/HornetStudio.Editor/Widgets/Workflow/`.
- The preferred visual/dialog pattern is `src/HornetStudio.Editor/Widgets/Common/InteractionRulesEditorDialogWindow.axaml` and `.axaml.cs`.
- The folder-layout watcher was debugged so `Scripts/Workflows/*.yaml` should not reload the whole folder.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation/error messages must be English.
- Use existing Avalonia dialog patterns and theme bindings.
- Use existing YAML libraries and `WorkflowDefinitionCodec`.
- Do not introduce new dependencies.
- Do not add a node canvas or scripting features.
- Do not reload the whole folder after workflow create/edit/delete.
- Public methods/classes require XML documentation.
- Prefer named parameters for unclear or multi-argument calls.

## Acceptance Criteria
- `WorkflowWidget` shows `Add` in the header.
- `Add` opens a workflow editor dialog.
- Saving a new workflow creates one YAML file under `Scripts/Workflows/`.
- Creating a workflow does not make the `WorkflowWidget` disappear.
- Existing workflows show `Edit` and `Delete` actions.
- `Edit` opens the selected workflow and saves changes back to YAML.
- `Delete` asks for confirmation and removes the selected workflow file.
- The editor can add/remove/reorder flat `SetValue`, `Delay`, and `Log` steps.
- Invalid fields prevent save with a clear English error message.
- Existing unsupported/nested steps are not silently lost.
- After save/delete, only the widget workflow list refreshes.

## Verification
- Build:
  - `dotnet build src\HornetStudio.Editor\HornetStudio.Editor.csproj --no-restore`
  - If the running app locks output, build with a separate output directory or build the relevant project only.
- Manual UI verification:
  - Place a `WorkflowWidget`.
  - Click `Add`.
  - Create a workflow with `Log`, `SetValue`, and `Delay`.
  - Confirm the file appears under `Scripts/Workflows/`.
  - Confirm the widget remains visible and shows the new workflow immediately.
  - Edit the workflow and confirm YAML changes.
  - Delete the workflow and confirm it disappears from the list and filesystem.
  - Restart/reload the folder and confirm the workflow list remains correct.

## Out of Scope
- `IfThenElse` branch editor.
- Runtime execution changes.
- Workflow state changes.
- Workflow dependency/orchestration steps.
- Cross-folder shared workflow library.
- Python application integration.
- New global watcher behavior beyond avoiding folder reload regressions.

## Risks / Watchouts
- Saving via the editor must not trigger full folder reloads.
- `IfThenElse` preservation is risky if the editor converts everything into flat rows. Do not drop nested content silently.
- Empty workflows may be useful during editing, but invalid YAML should not be created accidentally.
- File renaming can introduce edge cases. If safe rename is not implemented confidently, keep edit/save to the same file and make file name read-only.
- Log target discovery should match existing Monitor behavior.
- `SetValue` target options should avoid suggesting read-only targets if an existing write-target filter exists.
