# IMPLEMENTATION HANDOFF

## Goal
Add Python functions to the widget-independent `FunctionRegistry` as read-only registry entries. The `Functions` widget should visualize these Python entries as a separate provider section/source, while Python storage, Python execution, templates, and existing Python interaction behavior remain unchanged.

## Scope
- Included:
  - Add a Python-backed registry provider that reads existing runtime/application metadata.
  - Keep the provider independent from `FunctionsControl`; callers can query `FunctionRegistry` without placing a `Functions` widget.
  - Display Python registry entries in `Functions` as read-only rows.
  - Keep declarative YAML function behavior unchanged.
  - Update docs/help to describe Python entries as registry-visible but externally owned.
  - Add focused tests where the existing test structure supports it.
- Not included:
  - Executing Python functions through `RunFunction`.
  - Changing `InvokePythonFunction`.
  - Moving, generating, scanning, or editing Python files.
  - Changing Python templates, Python bridge commands, or Python runtime startup.
  - Adding the future `Workflows` widget.
  - Broad refactoring of existing widget file names such as `Widgets/Workflow`.

## Starting Point
- `FunctionRegistry` already exists in `src/HornetStudio.Editor/Functions/FunctionRegistry.cs`.
- It currently has a `DeclarativeFunctionRegistryProvider` and returns `FunctionCatalogEntry` entries.
- Buttons can execute declarative entries through `RunFunction`, but Python execution still uses the existing `InvokePythonFunction` path.
- Existing Python function discovery/invocation is tied to `PythonClientRuntimeRegistry` and application/runtime target paths.
- The `Functions` widget already visualizes registry entries and edits/deletes declarative YAML functions.

## Tasks
1. Inspect the existing Python function lookup paths:
   - `PythonClientRuntimeRegistry.GetRegisteredTargetPaths()`
   - `PythonClientRuntimeRegistry.GetFunctionNames(...)`
   - `ApplicationExplorerRuntime.ResolveInteractionTargetPath(...)`
   - existing `InvokePythonFunction` picker code.
2. Add a provider boundary for Python entries:
   - preferred name: `PythonFunctionRegistryProvider`
   - location: `src/HornetStudio.Editor/Functions/`
   - interface: existing `IFunctionRegistryEntryProvider`
3. Register the provider in `FunctionRegistry.Providers` after the declarative provider.
4. Generate stable Python registry references without changing persisted Python interaction data:
   - preferred format: `python:<target-path>:<function-name>`
   - normalize/trim values consistently
   - skip empty target paths or empty function names
5. Create Python `FunctionCatalogEntry` rows with:
   - `Kind = FunctionCatalogKind.Python`
   - `Source = FunctionCatalogSource.PythonApplication`
   - `CanEdit = false`
   - `CanDelete = false`
   - `CanRun = false` for this step, because `RunFunction` Python execution is explicitly out of scope
   - clear `DisplaySource`, target/source identifier, and validation state
6. Extend `FunctionCatalogKind` or related DTOs only if Python is not already represented cleanly.
7. Update `FunctionsControl` only where needed so Python entries render correctly and read-only capabilities disable edit/delete actions.
8. Ensure `RunFunction` keeps rejecting unsupported/non-declarative entries with a clear log/message and does not accidentally attempt Python execution.
9. Update docs:
   - `src/HornetStudio/docs/widgets/Functions.md`
   - `src/HornetStudio/docs/widgets/help/Functions.help.md`
   - `src/HornetStudio/docs/widgets/descriptions/Functions.md` only if the short picker text should mention Python visibility.
10. Add or update focused tests:
   - registry combines declarative and Python provider entries if Python runtime metadata is available/mocked
   - Python entries are read-only and not executable through `RunFunction`
   - registry query does not require `FunctionsControl`

## File-Level Changes
- `src/HornetStudio.Editor/Functions/FunctionRegistry.cs` -> register the Python provider and keep the registry widget-independent.
- `src/HornetStudio.Editor/Functions/PythonFunctionRegistryProvider.cs` -> add provider if a separate file is cleaner; otherwise keep provider in `FunctionRegistry.cs` only if the implementation remains small.
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml.cs` -> ensure Python entries are displayed as read-only rows; no ownership/edit logic for Python.
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowControl.axaml` -> only adjust labels/commands if current bindings do not respect read-only capabilities.
- `src/HornetStudio.Editor/Models/PageItemModel.cs` -> only verify existing `RunFunction` unsupported-kind handling; avoid broad changes.
- `src/HornetStudio.Editor/ViewModels/EditorDialogField.cs` -> no behavior change expected; use as reference for existing Python function lookup.
- `src/HornetStudio.Editor/Widgets/Monitor/MonitorEditorDialogWindow.axaml.cs` -> no behavior change expected; use as reference for existing Python function lookup.
- `src/HornetStudio/docs/widgets/Functions.md` -> document Python entries as read-only registry entries owned by Applications/Python runtime.
- `src/HornetStudio/docs/widgets/help/Functions.help.md` -> document provider behavior and limitation that `RunFunction` remains declarative-only.
- `src/HornetStudio/docs/widgets/descriptions/Functions.md` -> update only if needed for concise picker wording.
- `src/HornetStudio.Editor.Tests/Program.cs` or relevant test file -> add focused provider/registry checks if the current tests can cover it without starting Python.

## Implementation Order
1. Inspect current DTO enums and Python runtime registry APIs.
2. Add `PythonFunctionRegistryProvider` with no dependency on `FunctionsControl`.
3. Register the provider in `FunctionRegistry`.
4. Confirm `FunctionCatalogEntry` supports Python kind/source; add minimal enum values only if missing.
5. Adjust `FunctionsControl` rendering/capability handling if Python read-only rows expose gaps.
6. Verify `RunFunction` still only executes valid declarative entries.
7. Update widget documentation.
8. Add focused tests or document why runtime-backed Python enumeration cannot be tested without a larger seam.
9. Build and manually verify.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation/error messages must be English.
- Do not introduce new dependencies.
- Do not change Python file ownership, templates, runtime startup, or bridge protocol.
- Do not require a placed `Functions` widget for registry availability.
- Keep Python entries read-only in `Functions`.
- Keep `RunFunction` declarative-only in this step.

## Acceptance Criteria
- `FunctionRegistry.EnumerateEntries(folderDirectory)` can return both declarative and Python entries without constructing `FunctionsControl`.
- Python entries appear in `Functions` when Python runtime/application metadata exposes registered target paths and function names.
- Python rows are visibly/readably separated by kind/source and cannot be edited or deleted from `Functions`.
- `RunFunction` still executes declarative YAML functions and rejects Python registry references.
- Existing `InvokePythonFunction` picker/invocation behavior remains unchanged.
- No Python files, templates, generated application folders, or bridge command docs are changed unless an implementation detail unexpectedly touches Python bridge behavior.
- Documentation clearly states that `Functions` is a visualizer/editor and `FunctionRegistry` is the central lookup.

## Verification
- Build:
  - `dotnet build HornetStudio.sln --no-restore`
  - If the running app locks outputs, use the established project-level or separate `OutDir` fallback.
- Tests:
  - Run focused editor tests if present and applicable.
  - Add/run registry provider tests where possible.
- Manual:
  - Start/open a folder with declarative functions and confirm existing entries still show.
  - Start/register a Python client/application as before and confirm Python functions appear in `Functions`.
  - Confirm Python rows have no edit/delete action.
  - Confirm existing `InvokePythonFunction` still works.
  - Confirm a button `RunFunction` still works for declarative functions and does not execute Python entries.

## Out of Scope
- Python execution through `RunFunction`.
- Python argument editor changes.
- Python client command protocol changes.
- Python file scanning from disk.
- Workflow composition.
- Monitor/Button conversion from Python-specific pickers to unified registry pickers.

## Risks / Watchouts
- `PythonClientRuntimeRegistry` may only know functions after a client has registered at runtime. The provider should tolerate empty runtime state and return no Python entries instead of failing.
- Stable references must not depend on display text alone; use target path plus function name.
- Do not make `Functions` the owner of Python entries. It only displays what the registry reports.
- Avoid triggering Python startup, file generation, or application scanning just to populate the registry.
- If the current Python APIs cannot enumerate safely without side effects, stop after documenting the missing read-only API and add only the minimal API needed for enumeration.
