# IMPLEMENTATION HANDOFF

## Goal
Rename and stabilize the current workflow-step implementation as `Functions`. The current YAML files are not full workflows; each file is a named function/procedure made of sequential steps such as `SetValue`, `Delay`, and `Log`. This handoff supersedes the workflow-editor naming in the earlier handoff and prepares a clean foundation for a later, separate `Workflows` feature.

## Scope
- Included:
  - Rename the visible widget from `WorkflowWidget` / `Workflow` to `Functions`.
  - Rename user-facing picker text, header text, dialog titles, empty states, validation text, and documentation to `Functions` / `Function`.
  - Move the persisted function files from `Scripts/Workflows/` to `Scripts/Functions/` for new files.
  - Treat one YAML file as one `FunctionDefinition` with a sequential `steps` list.
  - Rename code types where practical:
    - `WorkflowControl` -> `FunctionsControl`
    - `WorkflowEditorDialogWindow` -> `FunctionEditorDialogWindow`
    - `WorkflowDefinition` -> `FunctionDefinition`
    - `WorkflowStepDefinition` -> `FunctionStepDefinition`
    - `WorkflowDefinitionCodec` -> `FunctionDefinitionCodec`
    - `WorkflowExecutor` -> `FunctionExecutor`
  - Use persisted layout type `Functions` going forward.
  - Keep backwards compatibility for already-created `WorkflowWidget` / `WorkflowWidget` docs and `Scripts/Workflows/` files only as a transitional loader/alias if this can be done locally.
  - Finish/stabilize the Functions editor that was planned in `2026.05.23.0716-workflow-editor-implementation-handoff.md`.
  - Ensure the project watcher continues to ignore function YAML files and does not reload folders when functions are added/edited/deleted.
  - Update tests and docs to match the new naming.
- Not included:
  - Building the later `Workflows` widget.
  - Orchestrating functions from workflow YAML.
  - Button integration for `RunFunction`, except keeping the naming/design ready for it.
  - Monitor action integration for `RunFunction`.
  - Branch editor for `IfThenElse`.
  - New runtime features beyond preserving the current sequential executor behavior under the new `Function` naming.

## Starting Point
- Current implementation uses `WorkflowWidget`, `WorkflowControl`, `WorkflowDefinition`, `WorkflowDefinitionCodec`, `WorkflowExecutor`, and `Scripts/Workflows/`.
- The current editor handoff planned `WorkflowEditorDialogWindow`, but the desired terminology changed before the feature should be finalized.
- Existing widget docs reference `WorkflowWidget`.
- The structured project watcher was fixed to process only `Folder.yaml` / legacy `Page.yaml`, so function YAML files should not trigger full folder reloads.
- The final intended architecture is two-level:
  - `Functions`: reusable named procedures made of atomic steps.
  - `Workflows`: later orchestration layer made from function calls.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation/error messages must be English.
- Do not introduce new dependencies.
- Keep the change scoped to naming/structure stabilization plus completing the current function editor.
- Do not implement the later `Workflows` widget in this stage.

- Prefer compatibility aliases over breaking existing test layouts or user-created files unless the old implementation is known to be throwaway.
- Use existing Avalonia and widget patterns.

## Acceptance Criteria
- The widget picker shows `Functions`, not `WorkflowWidget`.
- Newly placed widgets persist as `Type: Functions`.
- Existing `Type: WorkflowWidget` layouts still load as `Functions` if compatibility is implemented.
- New function YAML files are created under `Scripts/Functions`.
- Existing files under `Scripts/Workflows` are either still visible as legacy functions or a clear one-time migration path is documented.
- UI text says `Function` / `Functions`, not `Workflow`, for the current feature.
- Documentation describes Functions as named sequential step procedures.
- The current function editor supports add/edit/delete for function YAML files.
- Adding/editing/deleting function YAML files does not reload or remove the current folder/widget.
- Build and focused tests pass.

## Risks / Watchouts
- File/class renames can be noisy. Keep them mechanical and avoid unrelated refactoring.
- Persisted type changes can break old layouts if aliases are missed.
- Documentation loader may require doc file names to match persisted `Type`; make sure `Functions.md` and `Functions.help.md` are available.
- If both `Scripts/Functions/foo.yaml` and legacy `Scripts/Workflows/foo.yaml` exist, duplicate display names need deterministic handling.
- If legacy `WorkflowWidget` docs remain, do not let users see both `WorkflowWidget` and `Functions` as separate picker options for the same feature.
- The later `Workflows` feature should start from this clean `Functions` base, not reuse the old `WorkflowWidget` naming.
