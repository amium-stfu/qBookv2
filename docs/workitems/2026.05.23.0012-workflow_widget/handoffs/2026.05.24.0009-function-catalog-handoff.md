# IMPLEMENTATION HANDOFF

## Goal
Introduce a small central `FunctionCatalog` concept so `Functions` becomes the single lookup surface for callable functions. The catalog must not take ownership of Python scripts or Python application files. It only provides a unified registry/view that can later be used by Buttons, Workflows, Monitor actions, and other callers.

## Scope
- Included:
  - Add a lightweight function catalog model and provider boundary.
  - Represent declarative YAML functions as catalog entries.
  - Prepare the catalog shape for Python function entries without changing Python storage or execution.
  - Update the `Functions` widget structure so it can display catalog entries by source/kind.
  - Keep YAML function editing/deleting limited to declarative function entries.
  - Keep Python function entries read-only when they are added later.
  - Document the distinction between:
    - function definitions
    - function catalog entries
    - future workflow function calls
- Not included:
  - Implementing Python function discovery in this step unless it is already exposed through a tiny existing API and can be added without touching Python runtime behavior.
  - Moving or editing Python scripts.
  - Button `RunFunction` integration.
  - Monitor `RunFunction` integration.
  - New `Workflows` widget.
  - Changing Python execution, application startup, or Python templates.

## Starting Point
- The current feature is being renamed/stabilized from `WorkflowWidget` to `Functions`.
- Declarative functions live under `Scripts/Functions/*.yaml` and are represented by function definitions with sequential steps.
- Existing Python invocation paths already exist through Application Explorer / Python client concepts:
  - interaction rules use `InvokePythonFunction`
  - monitor actions use `InvokeFunction`
  - `PythonClientRuntimeRegistry.GetFunctionNames(...)` is already used by existing editors
- The user intent is a central register/lookup surface only. Python remains owned by Applications/Python infrastructure.

## Tasks
1. Add catalog model types:
   - `FunctionCatalogEntry`
   - `FunctionCatalogKind`
   - optional `FunctionCatalogSource`
2. Define catalog entry fields:
   - stable `Id` or `Reference`
   - `Name`
   - `Kind` (`Declarative`, future `Python`)
   - `SourcePath` or source identifier
   - `DisplaySource`
   - `CanEdit`
   - `CanDelete`
   - `CanRun`
3. Add a declarative catalog provider:
   - enumerates `Scripts/Functions/*.yaml`
   - optionally includes legacy `Scripts/Workflows/*.yaml` during transition
   - maps parsed YAML functions to `FunctionCatalogEntry`
   - surfaces invalid YAML as entries with clear status if this matches current UI behavior
4. Update `FunctionsControl` to build its row list from catalog entries instead of directly from raw workflow/function file enumeration.
5. Keep edit/delete behavior guarded by catalog capabilities:
   - declarative YAML entries: editable/deletable
   - future Python entries: read-only/non-deletable from `Functions`
6. Keep the existing `FunctionDefinitionCodec` focused on declarative YAML files.
7. Avoid Python dependency churn:
   - do not scan Python files directly
   - do not change Python application folders
   - do not change Python runtime registration
8. Add a documented extension point for future Python provider:
   - provider can later read existing Application/Python runtime metadata
   - provider returns `FunctionCatalogEntry` with `Kind=Python`
9. Update docs/help to state that `Functions` is the central catalog and that Python functions remain owned by Applications.
10. Add focused tests for catalog entry creation from declarative function files and capability flags.

## Implementation Order
1. Complete or preserve the `Functions` rename/stabilization work first.
2. Add catalog model types with no UI behavior change.
3. Add declarative provider that returns entries from existing function YAML discovery.
4. Switch `FunctionsControl` row construction to catalog entries.
5. Apply `CanEdit`/`CanDelete` to row actions.
6. Update docs/help.
7. Add focused tests.
8. Build and manually verify the Functions widget still behaves exactly as before for YAML functions.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation/error messages must be English.
- Do not introduce new dependencies.
- Keep Python storage and execution untouched.
- Keep catalog implementation small and local.
- Do not create a broad service layer unless at least the declarative provider and one future provider boundary clearly need it.
- Do not implement button, monitor, or workflow callers yet.

## Acceptance Criteria
- `Functions` still lists declarative YAML functions from `Scripts/Functions`.
- Each row is backed by a `FunctionCatalogEntry` or equivalent catalog model.
- Declarative entries show editable/deletable actions.
- Catalog model can represent future Python entries without changing Python files.
- Documentation states that Python functions are referenced through the catalog but remain owned by Applications/Python.
- No full-folder reload occurs when function YAML files change.
- Build and focused tests pass.

## Out of Scope
- Python provider implementation if it requires Python/runtime changes.
- Python script scanning.
- Editing Python functions inside `Functions`.
- `RunFunction` button action.
- `RunFunction` monitor action.
- `Workflows` widget.
- Function execution UI changes.

## Risks / Watchouts
- Avoid turning `Functions` into a monolithic owner of all function implementations.
- Keep `FunctionDefinition` and `FunctionCatalogEntry` separate:
  - definition is persisted YAML content
  - catalog entry is a lookup/display/reference record
- Do not duplicate existing Python function discovery logic unless a shared read API already exists.
- Future references should use stable IDs such as `declarative:<name>` or `python:<application>:<function>` rather than display text only.
