# IMPLEMENTATION HANDOFF

## Goal
Adjust the last `FunctionCatalog` implementation so the central function lookup is a widget-independent `FunctionRegistry`. The `Functions` widget must only visualize the registry and edit/delete its own declarative YAML functions. Python and other future providers must remain usable even when no `Functions` widget is placed on a folder.

## Scope
- Included:
  - Rename/reframe the central concept from widget-owned `FunctionCatalog` to widget-independent `FunctionRegistry`.
  - Keep `FunctionCatalogEntry` or equivalent entry DTO if already implemented; it can remain the row/reference model.
  - Move registry/provider code out of widget-only ownership if it currently lives only as `Widgets/Workflow/FunctionCatalogProvider.cs`.
  - Ensure the registry can be queried by future callers without a `FunctionsControl` instance.
  - Keep the `Functions` widget as a consumer/editor:
    - displays registry entries
    - edits/deletes declarative YAML function entries only
    - does not own Python, application functions, or the registry lifecycle
  - Document that Python functions remain available through existing Python/Application mechanisms without placing `Functions`.
  - Keep current declarative YAML behavior working.
- Not included:
  - Implementing Button `RunFunction`.
  - Implementing Monitor `RunFunction`.
  - Implementing Python provider if it requires runtime/template changes.
  - Moving or editing Python files.
  - New `Workflows` widget.
  - Changing function execution semantics.

## Starting Point
- A `Functions` widget and related docs are already present.
- Search shows a current local implementation around:
  - `src/HornetStudio.Editor/Widgets/Workflow/FunctionCatalogProvider.cs`
  - `FunctionCatalogEntry`
  - `FunctionCatalogKind`
  - `FunctionCatalogSource`
  - `FunctionsControl`
- The current implementation likely makes the catalog look like part of the widget feature. The required adjustment is architectural ownership, not a full rewrite.
- Existing Python invocation remains based on Application Explorer / Python client registries.

## Tasks
1. Inspect the current `FunctionCatalogProvider` implementation.
2. Decide the smallest relocation/naming change:
   - preferred namespace/folder: `src/HornetStudio.Editor/Functions/`
   - acceptable if less churn: keep files where they are but rename and document as widget-independent; avoid `FunctionsControl` dependency.
3. Introduce or rename to:
   - `FunctionRegistry`
   - `FunctionRegistryEntryProvider` or keep provider naming if clear
   - `DeclarativeFunctionRegistryProvider`
4. Keep `FunctionCatalogEntry` only if it is a neutral entry DTO:
   - if it is UI-specific, rename to `FunctionRegistryEntry` and create a separate UI row model in `FunctionsControl`.
5. Ensure `FunctionRegistry` API does not require a widget:
   - accepts project/folder context explicitly, e.g. folder directory, owner item, or project root as needed
   - returns entries from registered providers
   - has no direct dependency on `FunctionsControl`
6. Ensure `FunctionsControl` only does:
   - resolve current folder directory
   - call `FunctionRegistry`
   - convert entries to UI rows
   - route edit/delete only when entry capabilities allow it
7. Add clear capability flags:
   - `CanEdit`
   - `CanDelete`
   - `CanRun`
   - optional `CanOpenSource`
8. Document provider ownership:
   - declarative provider owns YAML discovery/editing paths
   - Python provider, when added later, reads existing Python runtime/application metadata only
   - no provider requires `Functions` widget placement
9. Update docs/help:
   - replace “Functions widget is the central catalog” wording with “FunctionRegistry is the central lookup; Functions widget visualizes and edits declarative entries.”
10. Update tests:
   - registry can enumerate declarative functions without constructing `FunctionsControl`
   - widget row conversion remains separate from registry entry creation
   - Python placeholder/provider boundary does not require a widget
11. Build and manually verify unchanged YAML function behavior.

1. Inspect current catalog/provider types and dependencies.
2. Extract or rename the central provider into `FunctionRegistry`.
3. Ensure the registry has a clean static or injectable query API independent of UI controls.
4. Update `FunctionsControl` to call the registry.
5. Keep UI row models inside the widget/control layer.
6. Update docs and tests.
7. Build and manually verify.

- The registry can enumerate declarative YAML functions without constructing `FunctionsControl`.
- `FunctionsControl` is a consumer of the registry, not the owner.
- Documentation explicitly states that `Functions` widget placement is optional for function availability.
- Python remains usable through existing mechanisms even with no `Functions` widget on a page.
- Future callers can be directed to query the registry directly.
- Current Functions widget add/edit/delete behavior still works.
- Build and focused tests pass.

## Risks / Watchouts

- Avoid mixing UI row state with registry entry state.
- Keep stable references separate from display names.
- If entry DTOs stay named `FunctionCatalogEntry`, document that “catalog entry” is a data shape while `FunctionRegistry` is the central owner/lookup.
- Be careful not to turn the registry into a broad application service prematurely; one clean static/query boundary is sufficient for now.
