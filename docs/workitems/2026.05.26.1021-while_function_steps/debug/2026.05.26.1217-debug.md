# DEBUG REPORT

## Problem
`StopFunction` interaction rules lost their selected function reference when saved to `folder.yaml`.

## Expected Behavior
Saving an interaction rule such as `StopFunction` with selected `YAML / new_workflow` should persist the selected function reference in `FunctionName`.

## Actual Behavior
The saved YAML contained `TargetPath: 'this'` and `FunctionName: ''` for `StopFunction`.

## Error Messages / Logs
No runtime exception was involved. The issue was visible in the persisted `InteractionRules` YAML.

## Relevant Code
- `src/HornetStudio.Editor/ViewModels/MainWindowViewModel.cs`
  - `TryBuildInteractionRulesJson(...)` persisted `FunctionName` only for `InvokePythonFunction` and `RunFunction`.
  - `ReadInteractionRulesProperty(...)` also treated `StopFunction` as a non-function action and moved/cleared function data on load.
- `src/HornetStudio.Editor/Models/ItemInteractionRule.cs`
  - The low-level interaction rule codec already treated `StopFunction` as a function-name action.
- `src/HornetStudio.Editor/Widgets/Common/InteractionRulesEditorDialogWindow.axaml.cs`
  - The editor already showed `StopFunction` with the RunFunction picker behavior.

## Attempted Fixes
- Fix 1 -> Added a shared `IsInteractionFunctionNameAction(...)` helper in `MainWindowViewModel` and included `StopFunction` with `InvokePythonFunction` and `RunFunction`. Result: focused build and editor tests passed.
- Fix 1 -> Added a regression test proving that `StopFunction` persists `FunctionName` into the YAML control definition. Result: `Editor tests passed: 161`.

## Current Hypothesis
- Fixed. The bug was a split-brain action classification: the dialog and compact codec knew `StopFunction` uses `FunctionName`, but the layout YAML export/import path did not.
