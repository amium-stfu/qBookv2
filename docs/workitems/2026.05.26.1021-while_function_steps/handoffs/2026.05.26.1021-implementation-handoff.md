# IMPLEMENTATION HANDOFF

## Goal
Add declarative `While` loop support to the Function/Workflow step model, editor, YAML codec, executor, tests, and widget documentation. Add a `StopFunction` step that ends the current function execution as `Done`.

## Scope
- Add `While` as a first-class `FunctionStepType`.
- Reuse the existing boolean condition editor pattern used by `IfThenElse`.
- Add one loop body step list for `While`.
- Add a required delay guard for every `While` loop.
- Add `StopFunction` as a first-class step.
- Update codec, runtime execution, editor conversion/UI, tests, and function/workflow documentation.
- Do not add new external dependencies.
- Do not change Python function discovery or execution behavior.
- Do not redesign the Function/Workflow widget beyond the controls needed for these step types.

## Starting Point
- `IfThenElse` is already editable and uses a shared condition editor dialog with condition variables.
- Existing editable step types are `SetValue`, `Delay`, `Log`, and `IfThenElse`.
- Existing function execution is sequential and recursive for `IfThenElse` branches.
- `Delay` currently accepts zero or greater in runtime validation, but `While` requires a stricter loop guard.
- `StopFunction` should complete the function with `FunctionState.Done`; any higher-level error handling belongs to the caller/instance that executes the function.

## Tasks
1. Extend the workflow model with `FunctionStepType.While`, `FunctionStepType.StopFunction`, `FunctionWhileStepDefinition`, and `FunctionStopFunctionStepDefinition`.
2. Model `While` with `Condition`, `Variables`, and `Steps`, matching `IfThenElse` condition semantics but with one body block and no `Else`.
3. Define loop delay guard rules:
   - New `While` rows must get a default `Delay` step with `Milliseconds = 100`.
   - The delay minimum inside a `While` guard is `1` ms.
   - The required delay may be moved within the `While` body.
   - The required delay must not be deleted from the `While` body.
   - Validation must reject a `While` body with no positive `Delay`.
4. Extend YAML parsing and serialization for `While` and `StopFunction`.
5. Extend executor behavior:
   - Evaluate the `While` condition before every iteration.
   - Execute the body while the condition is true.
   - Check cancellation before condition evaluation, before each body step, and between iterations.
   - Treat `StopFunction` as a controlled function completion with final state `Done`.
   - Ensure `StopFunction` exits nested `While` and `IfThenElse` execution immediately without being reported as failure.
6. Extend editor row conversion and UI:
   - Add `While` and `StopFunction` to editable step type options.
   - Render `While` similarly to `IfThenElse`, with compact header, condition summary, body count, condition dialog, and nested body list.
   - Render `StopFunction` as a compact row without additional fields.
   - Prevent deleting the last required positive `Delay` from a `While` body.
   - Allow moving the required `Delay` within the body.
   - Enforce a minimum of `1` ms for `Delay` rows that are required as a `While` guard.
7. Add or update tests for parser, serializer, editor row conversion, executor loop behavior, delay guard validation, and `StopFunction`.
8. Update Function and Workflow widget documentation, descriptions, and help files to include `While`, delay guard behavior, and `StopFunction`.

## Implementation Order
1. Add model types and XML documentation.
2. Add codec parsing/serialization and validation for `While`/`StopFunction`.
3. Add executor control flow for `While` and `StopFunction`, using an internal controlled-completion signal instead of reporting an exception as a failure.
4. Extend editor row state/conversion for `While` and `StopFunction`.
5. Extend AXAML layout for `While` body rows and compact `StopFunction`.
6. Add tests around model round-trip and runtime behavior.
7. Update widget docs and help content.
8. Run focused tests and then the solution build.

## Technical Constraints
- Follow existing `IfThenElse` condition editor patterns and do not introduce a separate condition system.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Chat responses remain German.
- Use existing libraries and patterns.
- Keep the architecture minimally fragmented; avoid new helper abstractions unless they clearly reduce duplication.
- Widget code changes under widget paths require matching documentation, help, and description updates.
- Public classes and public members require XML documentation.
- No new runtime dependencies.

## Acceptance Criteria
- Function YAML can load and save `While` with `condition`, optional `variables`, and nested `steps`.
- Function YAML can load and save `StopFunction`.
- A newly created `While` step contains a default `Delay` of `100` ms.
- The default/required `Delay` can be reordered within the `While` body.
- The required `Delay` cannot be deleted if it is the only positive delay guard in the `While` body.
- `While` validation rejects missing body steps, missing condition, or absence of a positive delay guard.
- Delay guard minimum for `While` is `1` ms.
- `StopFunction` ends execution with `FunctionState.Done`, including from inside nested `While` or `IfThenElse`.
- Normal cancellation still returns `FunctionState.Canceled`.
- Docs and help describe the new step types and delay guard behavior.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore`.
- Run `src/HornetStudio.Editor.Tests` using the repository's existing focused test workflow.
- Manually inspect Function Editor behavior for adding `While`, editing condition, moving the delay guard, preventing guard deletion, and adding `StopFunction`.

## Out of Scope
- Starting/stopping functions from a new widget surface.
- Python function loop support or Python API changes.
- Global scheduler changes.
- New visual redesign of the Function/Workflow editor.
- Release versioning, NuGet packaging, or Git tags.

## Risks / Watchouts
- Avoid a busy loop if YAML is edited manually; codec validation and executor safeguards must both protect against missing positive delays.
- `StopFunction` must unwind nested execution without turning into `Failed`.
- Be careful not to break existing `IfThenElse` round-trip behavior while reusing condition code.
- Existing docs still say the supported MVP schema only includes `Log`, `SetValue`, `Delay`, and `IfThenElse`; update all matching locations.
- If the editor tracks row actions generically, ensure the required `Delay` cannot be deleted through nested row commands.

## Notes
- Suggested YAML shape:

```yaml
- type: While
  condition: "{enabled} == true"
  variables:
    - name: enabled
      sourcePath: "System/Enabled"
  steps:
    - type: Delay
      milliseconds: 100
    - type: StopFunction
```

- `StopFunction -> Done` is an explicit product decision from planning.
- The executor should not treat `StopFunction` as error handling. The instance that executes the function owns any higher-level decision about whether this is expected.
