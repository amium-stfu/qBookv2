# IMPLEMENTATION HANDOFF

## Goal
Add declarative `While` loop support to the Function/Workflow step model, editor, YAML codec, executor, tests, and widget documentation. Add `StopFunction` as an Interaction Rules action that stops a running function instance from outside the Function Step Editor.

This handoff supersedes `docs/workitems/2026.05.26.1021-while_function_steps/handoffs/2026.05.26.1021-implementation-handoff.md` because that earlier handoff incorrectly planned `StopFunction` as a function step type.

## Scope
- Add `While` as a first-class declarative function step.
- Reuse the existing boolean condition editor pattern used by `IfThenElse`.
- Add one loop body step list for `While`.
- Add a required positive delay guard for every `While` loop.
- Add or wire `StopFunction` as an Interaction Rules action/call surface.
- `StopFunction` must stop an already running function instance and make that function execution end as `Done`.
- Do not add `StopFunction` to the Function Step Editor.
- Do not add `StopFunction` to the function YAML schema.
- Do not parse or serialize `StopFunction` as a `FunctionStepType`.
- Do not add new external dependencies.
- Do not change Python function discovery unless required to keep existing `RunFunction` behavior intact.

## Starting Point
- `IfThenElse` is already editable and uses a shared condition editor dialog with condition variables.
- Existing editable declarative step types are `SetValue`, `Delay`, `Log`, and `IfThenElse`.
- Existing function execution is sequential and recursive for `IfThenElse` branches.
- Interaction Rules already support `RunFunction` through `ItemInteractionAction.RunFunction`.
- Declarative `RunFunction` execution currently starts from `PageItemModel` through `TryStartRunFunction`, `TryStartRunDeclarativeFunction`, and `ExecuteRunFunctionAsync`.
- The current `RunFunction` execution path should be analyzed for how running function instances are tracked or need to be tracked.
- Product decision: `StopFunction` is an external Interaction Rules action, not a step inside the function being edited.
- Product decision: when `StopFunction` successfully stops a function instance, the stopped function's final state is `Done`.
- Product decision: error handling for `StopFunction` belongs to the instance/component that executes the stop action.

## Tasks
1. Extend the declarative workflow model with `FunctionStepType.While` and `FunctionWhileStepDefinition`.
2. Model `While` with `Condition`, `Variables`, and `Steps`, matching `IfThenElse` condition semantics but with one body block and no `Else`.
3. Define loop delay guard rules:
   - New `While` rows must get a default `Delay` step with `Milliseconds = 100`.
   - The delay minimum inside a `While` guard is `1` ms.
   - The required delay may be moved within the `While` body.
   - The required delay must not be deleted if it is the only positive delay guard in the `While` body.
   - Codec/editor validation must reject a `While` body with no positive `Delay`.
   - Executor safeguards must also prevent running a `While` with no positive delay guard.
4. Extend YAML parsing and serialization for `While` only.
5. Extend executor behavior for `While`:
   - Evaluate the `While` condition before every iteration.
   - Execute the body while the condition is true.
   - Check cancellation before condition evaluation, before each body step, and between iterations.
   - Preserve existing `Canceled`, `Failed`, and `Done` semantics for normal completion, errors, and external cancellation.
6. Design and implement function instance tracking for declarative `RunFunction` if no suitable mechanism already exists:
   - Track active declarative function executions by stable function reference and/or owning interaction context as appropriate.
   - Store enough control state to request a stop of a running execution.
   - Avoid blocking calls and use async/cancellation correctly.
7. Add `StopFunction` to Interaction Rules:
   - Add an `ItemInteractionAction.StopFunction` action or equivalent existing action mapping if one already exists.
   - Expose it in the Interaction Rules editor.
   - Reuse the function picker/reference behavior from `RunFunction` where the user needs to choose which function instance/reference to stop.
   - Do not show `StopFunction` in the Function Step Editor step type list.
8. Implement `StopFunction` runtime behavior:
   - Resolve the selected function reference/target.
   - Request controlled stop for matching running declarative function execution.
   - The stopped execution should complete with `FunctionState.Done`, not `Canceled` or `Failed`.
   - If no matching execution exists, report/log that through the caller's existing Interaction Rules error handling path.
9. Extend editor row conversion and UI for `While`:
   - Add `While` to editable function step type options.
   - Render `While` similarly to `IfThenElse`, with compact header, condition summary, body count, condition dialog, and nested body list.
   - Prevent deleting the last required positive `Delay` from a `While` body.
   - Allow moving the required `Delay` within the body.
   - Enforce a minimum of `1` ms for `Delay` rows that are required as a `While` guard.
10. Add or update tests for parser, serializer, editor row conversion, executor loop behavior, delay guard validation, Interaction Rules codec/editor options, `RunFunction` instance tracking, and external `StopFunction`.
11. Update Function, Workflow, Button, and Interaction Rules related documentation where they describe supported steps or interaction actions.

4. Extend Function Step Editor row state/conversion/layout for `While`.

## Technical Constraints
- Follow existing `IfThenElse` condition editor patterns and do not introduce a separate condition system.
- `StopFunction` is not a `FunctionStepType`.
- `StopFunction` must not appear in YAML function definitions.
- `StopFunction` must not appear in the Function Step Editor step picker.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Chat responses remain German.
- Use existing libraries and patterns.
- Keep the architecture minimally fragmented; avoid new services unless needed for function instance tracking.
- Widget code changes under widget paths require matching documentation, help, and description updates.
- Public classes and public members require XML documentation.
- No new runtime dependencies.

## Acceptance Criteria
- Function YAML can load and save `While` with `condition`, optional `variables`, and nested `steps`.
- Function YAML does not accept or emit `StopFunction` as a step.
- A newly created `While` step contains a default `Delay` of `100` ms.
- The default/required `Delay` can be reordered within the `While` body.
- The required `Delay` cannot be deleted if it is the only positive delay guard in the `While` body.
- `While` validation rejects missing body steps, missing condition, or absence of a positive delay guard.
- Delay guard minimum for `While` is `1` ms.
- `While` conditions are evaluated before every iteration.
- `StopFunction` is available as an Interaction Rules action/call, not as a Function Step Editor option.
- `StopFunction` can stop a running declarative function instance selected/resolved by reference.
- A function stopped through `StopFunction` finishes as `FunctionState.Done`.
- Normal external cancellation still returns `FunctionState.Canceled`.
- Missing or non-running target handling is reported through the caller's existing Interaction Rules error/logging path.
- Docs and help describe `While`, delay guard behavior, and external `StopFunction` behavior accurately.

## Out of Scope
- Adding `StopFunction` as a YAML step.
- Adding `StopFunction` to the Function Step Editor.
- Python function stop semantics unless a current shared function instance mechanism naturally supports them.
- Global scheduler changes.
- New visual redesign of the Function/Workflow editor.
- Release versioning, NuGet packaging, or Git tags.

## Risks / Watchouts
- Avoid a busy loop if YAML is edited manually; codec validation and executor safeguards must both protect against missing positive delays.
- Do not let `StopFunction` use ordinary cancellation if that would make the stopped function report `Canceled`; the planned final state is `Done`.
- Be careful not to break existing `RunFunction` behavior for Python or declarative entries.
- The correct identity for a running function instance may need a deliberate choice: stable function reference, owner item path, or both. Prefer the smallest model that matches current `RunFunction` behavior.
- Existing docs may already mention `StopFunction` as if it were a step; correct that wording.

## Notes
- Suggested YAML shape for `While`:

```yaml
- type: While
  condition: "{enabled} == true"
  variables:
    - name: enabled
      sourcePath: "System/Enabled"
  steps:
    - type: Delay
      milliseconds: 100
    - type: Log
      targetLog: "Main"
      level: Info
      text: "Loop tick"
```

- `StopFunction` should be configured through Interaction Rules in the same conceptual area as `RunFunction`.
- `StopFunction -> Done` is an explicit product decision from planning.
- The instance/component executing `StopFunction` owns higher-level error handling for missing target, duplicate running instances, or unsupported function kinds.
