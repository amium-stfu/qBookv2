# IMPLEMENTATION HANDOFF

## Goal
Introduce explicit registry target type metadata through a required `type` property so editors, target pickers, TreeView, and `SetValue` no longer have to infer value semantics from current values or naming conventions.

## Scope
- Add a central type model/parser for registry target types.
- Prefer explicit `item.Properties["type"]` when resolving target metadata.
- Keep existing `format` behavior unchanged as display/formatting metadata.
- Make `type` required for newly created or runtime-published writable targets controlled by HornetStudio code.
- Add `type` metadata to `UdlModule` channels immediately:
  - `read` -> `float`
  - `set` -> `float`
  - `out` -> `float`
  - `state` -> `int`
  - `alert` -> `int`
- Connect the existing `Edit SetValue` target classification to explicit `type` first, with existing fallback behavior preserved for legacy/external targets.
- Expose the type in registry/target metadata paths where currently available, so the TreeView and future pickers can use the same source.

Explicitly not included:
- Redesigning `format`.
- Introducing combined `format = float:0.00` syntax.
- Migrating every producer in one pass, such as all ItemClient/Broker/EnhancedSignals targets, unless a tiny local change is clearly necessary for compilation or first verification.
- Breaking existing targets that do not yet publish `type`.

## Starting Point
- Runtime registry items currently expose values, write/read properties, `writable`, `write_path`, `format`, and similar metadata inconsistently.
- `Edit SetValue` and target metadata currently infer type from `write`/`read` values and fallbacks such as `.set -> .read`.
- `UdlModule` currently creates flat channels `read`, `set`, `out`, `state`, and `alert`, but does not publish explicit value type metadata.
- According to the CanBus/UDL convention supplied by the user, `read`, `set`, and `out` are floats; `state` and `alert` are ints.
- Existing external or legacy targets may not have `type`; they must remain usable.

## Tasks
1. Define the canonical registry target type values and central parser.
2. Update target metadata resolution to read explicit `type` before value-based inference.
3. Update `SetValue` target-kind classification to use the central type parser.
4. Add `type` metadata to `UdlModule` channel creation/metadata refresh.
5. Ensure the TreeView/registry display can surface the `type` property where properties are listed.
6. Add focused tests for type parsing, metadata resolution priority, and UDL channel metadata.
7. Update relevant documentation describing runtime target metadata and UDL channel semantics.
8. Run focused tests and build verification.

## File-Level Changes
- `src/HornetStudio.Editor/Models/TargetValueType.cs` or similarly local model file -> define canonical type enum/model and parse/normalize helpers.
- `src/HornetStudio.Editor/Models/SetValueOperation.cs` -> adapt classification to consume the new central type model instead of only raw CLR value types.
- `src/HornetStudio.Editor/ViewModels/EditorDialogField.cs` -> prefer explicit `type` in `GetSetValueTargetDescriptor(...)` / target descriptor resolution.
- `src/HornetStudio.Editor/Models/PageItemModel.cs` -> prefer explicit `type` in runtime `SetValue` target-kind resolution.
- `src/HornetStudio.Host/UdlModule.cs` -> publish `type` on `read`, `set`, `out`, `state`, and `alert` channels and preserve it during `EnsureWriteMetadata()`.
- `src/HornetStudio.Editor.Tests/Program.cs` or existing relevant test entry -> add focused tests for parser, resolver behavior, and UDL channel types.
- `src/HornetStudio/docs/manual/` or relevant widget docs -> document the `type` metadata convention if registry/target metadata is described there.

## Implementation Order
1. Add the central target type model/parser.
2. Wire explicit `type` into editor metadata resolution while keeping current value-based fallbacks.
3. Wire explicit `type` into runtime `SetValue` target-kind classification while keeping current fallbacks.
4. Add UDL channel `type` metadata in `UdlModule`.
5. Add/update tests for the central parser and UDL module channel metadata.
6. Add tests or assertions proving explicit `type` wins over empty values for `set` targets.
7. Update documentation for the new `type` requirement and UDL channel type mapping.
8. Build and run focused tests.

## Technical Constraints
- Chat response is German; code, comments, XML documentation, UI text, file names, and technical identifiers are English.
- Do not change `format` semantics in this work.
- Do not introduce new runtime dependencies.
- Keep the first implementation minimally fragmented.
- Prefer a central parser/resolver over duplicated widget-specific type inference.
- Existing targets without `type` must not crash or become unusable.
- Public classes/methods added or behaviorally changed need XML documentation.

## Acceptance Criteria
- A canonical `type` parser exists and accepts at least:
  - `unknown`
  - `string`
  - `bool`
  - `int`
  - `long`
  - `float`
  - `double`
  - `decimal`
  - `epoch`
  - `bits`
  - `object`
- `format` remains a separate display/formatting property.
- Target metadata resolution prefers explicit `type` over inferred CLR/current-value type.
- `Edit SetValue` detects UDL `set` targets as numeric when `type=float` exists, even if the current set value is empty.
- `UdlModule` publishes:
  - `read.type = float`
  - `set.type = float`
  - `out.type = float`
  - `state.type = int`
  - `alert.type = int`
- Existing legacy targets without `type` still use fallback inference and may resolve to `unknown`.
- New code-controlled writable targets should be treated as invalid/incomplete during development if they omit `type`; external/legacy targets remain tolerant.

## Verification
- Run focused tests covering:
  - type parser normalization
  - explicit `type` priority over empty value
  - numeric mapping for `int`, `long`, `float`, `double`, `decimal`, `epoch`, and `bits`
  - boolean/string/object/unknown mapping
  - UDL channel type metadata
- Run `dotnet build src/HornetStudio.Editor/HornetStudio.Editor.csproj --no-restore` for focused editor validation.
- Run `dotnet build HornetStudio.sln --no-restore` when no running app locks output files.
- Manually inspect the registry TreeView for a UDL module and confirm the `type` properties are visible on the channels.
- Manually verify `Edit SetValue` detects a UDL `set` target as numeric.

## Out of Scope
- Full ItemClient/Broker metadata migration.
- Full EnhancedSignals metadata migration beyond existing fallback behavior.
- Changing persisted interaction rule format.
- Changing `format` to include type.
- Adding specialized editors for `epoch` or `bits`; they may map to numeric behavior for now.

## Risks / Watchouts
- `type` as a generic property name could conflict with existing external producers. Normalize known values and treat unrecognized values as `unknown`.
- Some code may already use `kind` or `format` for UI display. Do not repurpose those fields.
- Avoid breaking legacy/external registry items that cannot immediately publish `type`.
- If tests instantiate `UdlModule` from `HornetStudio.Host`, visibility may require testing through public host behavior instead of direct construction because `UdlModule` is internal.

## Notes
- `type` is the semantic source for validation, compatible item filtering, and operation availability.
- `format` remains display-oriented and should not be parsed as the primary semantic type in this plan.
- First producer with guaranteed mapping is `UdlModule`; other producers can be migrated incrementally in follow-up workitems.
