# IMPLEMENTATION HANDOFF

## Goal
Add a dedicated Monitoring property group to the Signal widget and extend runtime monitoring beyond timeout to optional lower and upper limit checks.

## Scope
Included:
- Introduce a Monitoring settings group for Signal widget configuration.
- Change `TimeoutMs` from required integer semantics to optional/null semantics.
- Add optional `LowerLimit` and `UpperLimit` monitoring thresholds.
- Add `InhibitMs` with default `0` as shared activation delay for monitoring status outputs.
- Publish bool runtime status items for timeout, lower-limit, and upper-limit violations.
- Update Signal widget documentation to describe the new monitoring model.

Excluded:
- Moving monitoring into global host item properties.
- Changing the target `read` value semantics.
- Adding aggregated quality/state objects beyond the explicit bool status outputs.
- Per-check inhibit settings such as `TimeoutInhibitMs`, `LowerLimitInhibitMs`, or `UpperLimitInhibitMs`.
- Non-Signal widgets.

## Starting Point
- Current Signal widget config exposes `TimeoutMs` as a flat property in the Signal `Properties` section.
- Current timeout monitoring is widget-local and published under `studio.<page>.signal_runtime.<widget>.status.timeout`.
- Current persistence and editor bindings model `TimeoutMs` as non-nullable `int`.
- The editor property system supports mandatory `int` and `double`, but not nullable numeric values.
- Runtime timeout evaluation already happens in `PageItemModel` using the source item epoch and a widget-local timer.

Assumptions confirmed in planning:
- New settings group name: `Monitoring`.
- `TimeoutMs` default: `null`.
- `LowerLimit` default: `null`.
- `UpperLimit` default: `null`.
- `InhibitMs` default: `0`.
- `InhibitMs` means a shared activation delay: a violation must remain active for at least `InhibitMs` before the corresponding bool status becomes `true`.
- Threshold monitoring only applies to numeric resolved read values.

## Tasks
1. Refactor Signal widget configuration to introduce a dedicated Monitoring group in the editor instead of keeping monitoring values as flat properties.
2. Extend the persistence and item model to support nullable monitoring values for timeout and numeric thresholds.
3. Extend Signal runtime monitoring logic to evaluate timeout, lower-limit, and upper-limit conditions with shared inhibit handling.
4. Publish separate bool runtime status items for each monitoring condition.
5. Update Signal widget YAML/example definitions and Markdown documentation to match the new configuration and runtime behavior.
6. Validate the touched editor/model surface with targeted compile verification and, if practical, focused tests.

## File-Level Changes
- `src/HornetStudio.Editor/Models/PageItemModel.cs` -> change monitoring-related properties to support nullable timeout and nullable numeric thresholds; add lower/upper-limit evaluation and shared inhibit tracking; keep widget-local runtime status publishing.
- `src/HornetStudio.Editor/Persistence/LayoutDocument.cs` -> persist `TimeoutMs`, `LowerLimit`, and `UpperLimit` as nullable values; persist `InhibitMs` with default `0`.
- `src/HornetStudio.Editor/ViewModels/MainWindowViewModel.cs` -> move Signal monitoring settings into a dedicated `Monitoring` section; add bindings for nullable numeric fields and `InhibitMs`.
- `src/HornetStudio.Editor/ViewModels/EditorPropertyType.cs` -> add nullable numeric field types if needed.
- `src/HornetStudio.Editor/ViewModels/EditorDialogBindingDefinition.cs` -> support empty-string round-tripping for nullable numeric editor fields.
- `src/HornetStudio.Editor/YamlDefinition/SignalWidget.yaml` -> update sample/config shape to reflect the Monitoring group and new defaults.
- `src/HornetStudio/docs/widgets/Signal.md` -> update summary documentation for Monitoring settings and runtime outputs.
- `src/HornetStudio/docs/widgets/help/Signal.help.md` -> update detailed help for `TimeoutMs`, `LowerLimit`, `UpperLimit`, `InhibitMs`, and runtime status behavior.
- `src/HornetStudio/docs/widgets/descriptions/Signal.md` -> refresh the concise picker description if monitoring capabilities become more prominent.
- `src/HornetStudio.Editor.Tests/Program.cs` -> add or extend focused tests if there is already suitable coverage around Signal runtime status publishing or editor field persistence.

## Implementation Order
1. Update the editor field/binding infrastructure to support nullable numeric input without string-space hacks.
2. Update persisted document and `PageItemModel` properties to use the new nullable monitoring model.
3. Rework Signal editor sections so monitoring fields appear under a dedicated `Monitoring` group.
4. Extend runtime evaluation in `PageItemModel` for lower/upper-limit status and shared inhibit timing.
5. Update YAML and widget documentation.
6. Run focused verification and fix any local fallout.

## Technical Constraints
- Keep monitoring widget-local; do not push threshold or timeout state into source host items.
- Keep `read` semantics unchanged; monitoring outputs remain separate status items.
- Use English for code, comments, and user-visible configuration labels.
- Preserve existing Signal runtime path conventions under `studio.<page>.signal_runtime.<widget>.status.*`.
- Prefer minimal, local changes over introducing new abstraction layers.
- Do not use sentinel strings like `' '` to represent disabled numeric monitoring.
- `TimeoutMs`, `LowerLimit`, and `UpperLimit` should use true null/empty semantics.

## Acceptance Criteria
- Signal editor shows a dedicated `Monitoring` group.
- `TimeoutMs`, `LowerLimit`, and `UpperLimit` can be left empty and persist as not configured.
- `InhibitMs` defaults to `0` and is applied as shared activation delay for monitoring violations.
- Timeout monitoring still publishes a bool under `status.timeout` only when configured.
- Numeric Signals publish bool statuses under `status.lower_limit` and `status.upper_limit` when their configured thresholds are violated.
- Non-numeric values do not crash monitoring evaluation.
- Existing Signal widgets without monitoring config continue to work without forced values.
- Signal widget docs describe the new Monitoring configuration and runtime outputs.

## Verification
- `dotnet build HornetStudio.sln --no-restore -c Debug`
- If targeted tests are added or updated, run the narrowest relevant test executable/project for the touched slice.
- Manually verify in the editor that empty nullable monitoring fields remain empty after reopen/save and that runtime status items appear only when configured.

## Out of Scope
- Global signal quality/state framework.
- UI styling changes beyond adding the Monitoring group.
- Aggregated `out_of_range` or `quality.state` items.
- Automatic migration tooling beyond normal persisted compatibility handling.

## Risks / Watchouts
- Changing persisted numeric fields from non-nullable to nullable can break existing load/save assumptions if not handled carefully.
- Editor field parsing must distinguish empty input from invalid numeric input.
- Existing Signal runtime status code currently assumes timeout-only state and will need careful extension to avoid stale published paths or mixed caching.
- Inhibit timing must be based on condition activation time, not simply the last target update time.
- Numeric comparison must use a deterministic conversion path and fail closed when the displayed/read value is not comparable.
