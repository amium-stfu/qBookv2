# DEBUG REPORT

## Problem

The PID controller runtime stayed stopped after tests wrote `true` to the published `run` item.

## Expected Behavior

Writing `true` to `studio.<folder>.controller_widget.<controller>.run` or to its `write` property should start the PID runtime. Writing `false` should stop it.

## Actual Behavior

The focused host self-tests reported `State='Stopped'`, an empty alert, and output `0` after `run` was written.

## Error Messages / Logs

- `PID controller output should be positive after start, actual '0'. State='Stopped', Alert=''`
- `PID controller published unexpected alert '' with state 'Stopped'.`
- `PID controller output should clamp to 10, actual '0'. State='Stopped', Alert='', RuntimeOutput=''`

## Relevant Code

- `src/HornetStudio.Host/PidControllerRuntime.cs`
- `src/HornetStudio.Host.Tests/Program.cs`

The registry channel behavior was isolated with a focused host self-test: direct `UpdateValue` on an item that has `read` and `write` channels updates the `write` property, while `Item.Value` continues to expose the `read` value.

## Attempted Fixes

- Fixed `StopPidTimer` definite-assignment build blocker in the host self-test helper -> solution build became possible.
- Added PID host diagnostics and a focused writable-channel regression test -> confirmed the `run` request was stored in `run.write`, not in `run.Value`.
- Updated `PidControllerRuntime` to consume `run.write` and `set.write` as write requests and to mirror the accepted run state into the published `read`/`write` channel values -> host PID tests passed.

## Current Hypothesis (optional)

- Resolved: the blocker was caused by using generic item-value reads for a writable channel. PID runtime control must follow the existing registry writable-channel contract, where write requests are consumed from the `write` property and stable state is published through `read`.
