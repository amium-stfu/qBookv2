# IMPLEMENTATION HANDOFF

## Goal
Add a new HornetStudio `ControllerWidget` that can create, persist, display, and run PID controllers. The first implementation scope is PID only, based on the behavior of the provided `FunkyLib.PIDController.cs` reference, adapted to the current HornetStudio runtime registry and widget architecture.

## Scope
- Add a new widget type named `ControllerWidget`.
- Persist controller definitions in `Folder.yaml` under the widget properties.
- Support only one controller type in this workitem: `PID`.
- Add an `[Add]` flow with a dialog that configures PID definitions.
- Publish each PID controller as runtime items.
- Start/stop each PID controller through a writable `run` boolean runtime item.
- Implement PID runtime behavior equivalent in intent to the reference: source process value, setpoint, output target, CHR-derived PID parameters, compute/output intervals, scaling, clamping, anti-windup, and D filter.

## Implementation Order
1. Inspect `ExtendedSignalDefinition` and `MonitorDefinition` codec patterns, then create `ControllerDefinition` with a single supported `Type = PID`.
2. Extend persistence and `FolderItemModel` with a raw definition property and parsed DTO support.
3. Add `ControlKind.ControllerWidget` and wire default creation, type mapping, save/load, and rendering.
4. Create a minimal `ControllerControl` using the `EnhancedSignalsControl` row/persist/rebuild pattern.
5. Create `ControllerEditorDialogWindow` with grouped fields:
   - General: `Name`, `SourcePath`, `SetpointPath`, `OutputPath`
   - Run: optional initial run/default run flag if needed, but runtime control is the published `run` item
   - PID tuning: `Ks`, `Tu`, `Tg`, `DFilterTauMs`
   - Limits: `SetMin`, `SetMax`, `OutMin`, `OutMax`
   - Timing: `ComputeIntervalMs`, `OutputIntervalMs`
6. Implement `PidControllerRuntime`:
   - publish base runtime path `studio.{folder}.controller_widget.{controller_name}`
   - publish child items such as `run`, `source`, `set`, `out`, `state`, `alert`, `parameters.*`
   - resolve source/setpoint/output paths through existing registry path helpers
   - subscribe to registry changes for source/setpoint/run/parameter updates
   - use `ATimer` or the existing timer abstraction preferred in `EnhancedSignalRuntime`
   - compute PID only while `run == true`
   - write scaled output to the configured output target through `HostRegistries.Data.UpdateValue(...)`
7. Add `ControllerRuntimeManager` and call it from `MainWindowViewModel` wherever `EnhancedSignalRuntimeManager` is currently synchronized or released.
8. Add focused tests before or alongside runtime implementation, then update them until they pass.
9. Add widget documentation and update solution file documentation entries if required.
10. Run focused tests and solution build.

## Technical Constraints
- Chat and planning output stays German; code, comments, UI text, XML documentation, and identifiers stay English.
- Do not introduce new external dependencies.
- Follow existing Avalonia and runtime registry patterns.
- Keep the first implementation PID-only. Do not implement PWM in this workitem.
- Prefer existing path normalization and registry helper APIs before adding new helpers.
- Public classes and public methods require XML documentation.
- User-visible validation/error text must be English.
- Persist structured controller definitions in `Folder.yaml`, not runtime-only state.
- Runtime path names should use stable lowercase snake_case segments where existing runtime conventions do.

## Acceptance Criteria
- A user can add a `ControllerWidget` to a folder.
- A user can add, edit, and delete a PID controller from that widget.
- The PID controller definition is saved into and restored from `Folder.yaml`.
- Each PID controller publishes runtime items under a stable controller runtime path.
- Writing `true` to the controller `run` item starts the controller; writing `false` stops it.
- While running, the PID reads the configured source and setpoint, computes the internal output, clamps/scales it, and writes to the configured output target.
- Invalid or missing numeric values do not crash the runtime and publish a useful `alert`/`state`.
- Removing or changing definitions releases obsolete runtime instances.
- Existing `EnhancedSignals`, `CustomSignals`, and `Monitor` behavior remains unchanged.

## Out of Scope
- PWM support.
- PID autotune.
- Graphing/tuning charts.
- Multiple PID algorithm variants.
- External package dependencies.
- Broad refactoring of the widget system or registry APIs.
- Release/version/tag changes.

## Risks / Watchouts
- Timer callbacks must not block the UI thread.
- Runtime writes can create feedback loops if source/setpoint/output paths overlap; guard against self-triggered update loops where needed.
- Changing parameter values while running must either apply safely on the next tick or stop/restart predictably.
- Output path writes must go through registry update APIs so broker/tree/listeners observe changes.
- The reference PID uses `Set` from the base module; HornetStudio should model setpoint explicitly as a configured path or a published writable `set` runtime item.
- `Folder.yaml` save/load may have both JSON-node and document-code paths; update both if both are active.
