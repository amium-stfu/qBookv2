# IMPLEMENTATION HANDOFF

## Goal
Adjust the PID controller widget contract so `Source` and `Out` remain external picker connections, while `Set`/setpoint becomes an owned writable runtime subitem of the PID controller itself.

## Scope
Included:
- Remove the PID `Set` target picker from the controller editor experience.
- Keep `Source` configured through a readable item picker.
- Keep `Out` configured through a writable item picker.
- Publish an owned writable PID setpoint subitem at the controller runtime path.
- Make the PID runtime read its setpoint from the owned PID subitem instead of an externally picked `SetpointPath`.
- Update persistence/model code only as far as needed for this contract.
- Update PID widget documentation and focused host tests.

Not included:
- Adding support for a remote/external setpoint picker.
- Adding controller types other than PID.
- Redesigning the ControllerWidget UI beyond removing the `Set` picker and exposing the owned setpoint contract.
- Broad refactoring of the widget or runtime infrastructure.

## Starting Point
- `ControllerWidget` has already been implemented under `src/HornetStudio.Editor/Widgets/Controller/`.
- The editor/controller model currently includes PID paths for source, setpoint, and output.
- `PidControllerRuntime` exists under `src/HornetStudio.Host/PidControllerRuntime.cs`.
- Focused PID self-tests already exist in `src/HornetStudio.Host.Tests/Program.cs`.
- Existing workitem: `docs/workitems/2026.05.19.2332-pid_controller_widget/`.
- Repository rule: widget changes under `src/HornetStudio.Editor/Widgets/**` require matching updates under `src/HornetStudio/docs/widgets/`, `src/HornetStudio/docs/widgets/descriptions/`, and `src/HornetStudio/docs/widgets/help/`.

## Tasks
1. Inspect the current controller definition flow from editor model to contracts to runtime.
2. Remove `Set`/setpoint picker controls and validation from `ControllerEditorDialogWindow`.
3. Keep only `Source` and `Out` picker fields in the PID editor UI.
4. Change the persisted PID/controller definition so no new PID configuration requires a picked setpoint path.
5. Publish a writable PID-owned setpoint runtime item, preferably at a stable path under the controller runtime item such as `<controller_path>.set`.
6. Make `PidControllerRuntime` read the current setpoint value from the owned runtime item.
7. Ensure writes to the owned setpoint item update the value used by the PID evaluation loop.
8. Preserve clear error handling for missing/invalid `Source`, missing/invalid `Out`, and invalid numeric setpoint values.
9. Update focused PID host tests to cover the owned writable setpoint item and the removal of external setpoint dependency.
10. Update widget documentation to describe `Set` as an owned writable PID subitem, not a picker connection.

## Implementation Order
1. Trace the current persisted `SetpointPath` flow from editor dialog to contract to host runtime.
2. Update the contract/model with the smallest compatible change:
   - Prefer making setpoint path optional/obsolete only if needed for old saved definitions.
   - Avoid breaking unrelated controller serialization unless unavoidable.
3. Update the editor dialog to create/edit PID definitions with only `SourcePath` and `OutputPath` picker inputs.
4. Update runtime item publication to include an owned writable setpoint item.
5. Update PID evaluation to read the owned setpoint value.
6. Add or update tests for runtime path publication, setpoint write behavior, invalid setpoint handling, and output calculation.
7. Update ControllerWidget documentation files.
8. Run focused validation, then solution build if restore/build state allows it.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Do not introduce new dependencies.
- Keep changes limited to ControllerWidget/PID model, runtime, tests, and required docs.
- Use existing item registry/write-property patterns; do not introduce a parallel write mechanism.
- Use named arguments where calls become ambiguous or have more than two parameters.
- Public classes and public methods touched by the implementation must keep or receive XML documentation.
- User-visible error and validation messages must be English.

## Acceptance Criteria
- PID editor no longer shows or requires a `Set`/setpoint picker.
- PID editor still supports choosing `Source` via picker.
- PID editor still supports choosing `Out` via picker.
- Each PID runtime publishes its own writable setpoint subitem.
- Writing the PID-owned setpoint item changes the setpoint used by the PID evaluation.
- PID output is computed from external `Source`, owned `Set`, and external `Out`.
- Existing saved definitions with an older external setpoint path are either handled intentionally or documented as a migration/breaking concern.
- ControllerWidget documentation reflects the new role split.

## Verification
- `dotnet build HornetStudio.sln --no-restore`
- Run focused host self-tests via the built `src/HornetStudio.Host.Tests` executable, following `agents/solution.md` guidance.
- Manual editor check: open/add PID controller and confirm only `Source` and `Out` picker controls are available.
- Manual runtime check: confirm the PID-owned `set` item is writable and affects output calculation.

## Out of Scope
- Remote/external setpoint selection.
- PID tuning UX changes unrelated to the setpoint ownership contract.
- New controller widget layout redesign.
- Release versioning, tags, package publishing, or notice-file updates.

## Risks / Watchouts
- Existing persisted widgets may still contain `SetpointPath`; decide during implementation whether to ignore it, migrate it, or keep it as obsolete compatibility data.
- Runtime path casing/naming must stay consistent with existing widget runtime path conventions.
- The owned setpoint item must expose write semantics in the same way other writable runtime items do, otherwise external writes may appear to succeed without affecting PID evaluation.
- Tests currently create PID definitions through helper methods that may still require `setpointPath`; those helpers must be updated with the contract.
- Documentation must not describe `Set` as a target picker after this change.

## Notes
- Preferred semantic split: `Source` = external process value, `Set` = owned writable PID setpoint item, `Out` = external writable output target.
- If a future remote setpoint is needed, model it explicitly as a separate optional feature such as `SetSource` or `RemoteSetpoint`; do not reintroduce ambiguity into `Set`.
