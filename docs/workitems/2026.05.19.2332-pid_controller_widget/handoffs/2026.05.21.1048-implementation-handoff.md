# IMPLEMENTATION HANDOFF

## Goal
Remove the unnecessary `write` property publishing from PID-owned runtime items and make the PID controller use direct runtime values for its own `run` and `set` items.

The intended PID-owned runtime contract is:
- `<controller_path>.run` stores the active run state directly in `Value`.
- `<controller_path>.set` stores the controller-owned setpoint directly in `Value`.
- `source` remains a read-only view of the external process value.
- `out` remains a read-only view of the last calculated PID output.
- External output writing still targets the configured output path; this work does not change external item write mechanics.

## Scope
Included:
- Update `PidControllerRuntime` so `run` and `set` are direct local runtime values.
- Stop publishing `writable`, `write`, `write_path`, and `write_mode` on PID-owned `run` and `set` items.
- Keep `read` mirrored on `run` and `set` only if existing UI/resolution patterns still need it for display.
- Prefer reading user changes from `item.Value` for PID-owned `run` and `set`.
- Optionally consume legacy `write` values if an already-existing item still carries them, but do not publish new `write` metadata.
- Update focused host self-tests to reflect the direct-value contract.
- Update ControllerWidget documentation to describe direct PID-owned runtime values.

Not included:
- Global removal of `write` properties from the registry system.
- Changes to CAN, UDL, EnhancedSignal, ProcessLog, or other interfaces where separate write semantics may be valid.
- Changes to external output target write behavior.
- PID algorithm changes.
- ControllerWidget UI redesign.
- Persistence schema changes.

## Starting Point
- `PidControllerRuntime.PublishSnapshot` currently publishes:
  - `run.Value`
  - `run.Properties["read"]`
  - `run.Properties["writable"]`
  - `run.Properties["write"]`
  - `run.Properties["write_path"]`
  - `run.Properties["write_mode"]`
  - the same writable metadata for `set`
- `PidControllerRuntime.TryReadRunRequest` currently checks `write`, then `Value`, then extracted value.
- `ReadSetpointRequest` currently prefers `write`.
- Host PID tests currently verify `write_path` and include `run.write` / `set.write` update behavior.
- The user clarified that `write` should be reserved for real separated write interfaces such as CAN or other drivers with different write addresses.

## Technical Constraints

- User-visible runtime alerts and validation messages must be English.

## Acceptance Criteria
- PID runtime no longer publishes `write`, `write_path`, `write_mode`, or `writable` metadata on `<controller_path>.run`.
- PID runtime no longer publishes `write`, `write_path`, `write_mode`, or `writable` metadata on `<controller_path>.set`.
- Writing `<controller_path>.run` through `HostRegistries.Data.UpdateValue(path, value)` starts and stops the runtime.
- Writing `<controller_path>.set` through `HostRegistries.Data.UpdateValue(path, value)` changes the setpoint used by the PID calculation.
- `source`, `set`, `out`, `state`, and `alert` runtime display values remain readable.
- External output target writing still works.
- ControllerWidget documentation no longer describes PID-owned `set` or `run` as separate writable-channel items.
- Focused host self-tests pass.

## Out of Scope
- Removing `write` from CAN, UDL, EnhancedSignal, ProcessLog, or other runtime items.
- Reworking `HostRegistries`.
- Reworking generic item editor behavior.
- Reworking PID tuning, CHR formulas, or simulation behavior.
- Release versioning, tags, package publishing, or notice-file updates.

## Risks / Watchouts
- Some generic UI/editor code may currently use `writable = true` or `write_path` to decide whether a value can be edited. If so, the implementation must either keep a PID-specific UI path for direct values or postpone full removal after identifying the dependency.
- `UpsertSnapshot(..., pruneMissingMembers: true)` should remove no-longer-published metadata, but verify that stale properties are actually pruned for child items.
- Keeping legacy `write` fallback can make tests ambiguous. Tests should assert that new snapshots do not publish `write` metadata even if fallback consumption remains.
- Do not remove or weaken `write` semantics globally; they remain valid for interfaces with separate read/write addresses.

## Notes
- This plan intentionally follows the user’s domain model: `write` is useful for interfaces where writing has a separate physical/logical target, such as CAN IDs or driver write registers. PID-owned `run` and `set` are local controller values and should not model that separation.
