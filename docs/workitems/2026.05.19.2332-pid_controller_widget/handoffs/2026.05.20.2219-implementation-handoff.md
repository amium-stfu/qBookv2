# IMPLEMENTATION HANDOFF

## Goal
Rework the HornetStudio PID runtime calculation so it follows the FunkyLib/CHR reference behavior more closely while keeping the existing ControllerWidget runtime contract intact.

The target is the PID control algorithm only: source process value, owned setpoint value, output target, CHR-derived parameters, compute/output intervals, normalization, clamping, anti-windup, and validation.

## Scope
Included:
- Replace the current normalized position-PID calculation in `PidControllerRuntime` with the FunkyLib-style discrete CHR PID core.
- Use CHR parameters derived from `Ks`, `Tu`, and `Tg`:
  - `kr = 0.95 * tg / (ks * tu)`
  - `tn = 2.4 * tu`
  - `tv = 0.42 * tu`
- Normalize source and setpoint from `SetMin..SetMax` to an internal `0..100` controller range.
- Keep output scaling from internal `0..100` to `OutMin..OutMax`.
- Preserve owned setpoint behavior at `<controller_path>.set`.
- Preserve existing runtime publishing and run-state mechanics unless a calculation change requires a narrow adjustment.
- Add or adjust focused host self-tests for the new control behavior.

Not included:
- Simplifying or removing the `write` property pattern for `run` and `set`.
- Redesigning the ControllerWidget UI.
- Adding controller types other than PID.
- PID autotuning.
- New dependencies.
- Broad registry or item model refactoring.

## Starting Point
- `src/HornetStudio.Host/PidControllerRuntime.cs` currently implements a readable position-PID-like calculation:
  - normalizes source/setpoint to `0..1`
  - uses `bias = normalizedSetpoint`
  - uses `Kp = 0.6 * Tg / (Ks * Tu)`, `Ti = Tg`, `Td = 0.5 * Tu`
  - computes output directly from bias, P, I, and D terms
- The original workitem references `B:\QbookProjects\AkbSps\AkbSps.code\Pages\FunkyLib\FunkyLib.PIDController.cs` as the intended behavior basis.
- The FunkyLib reference uses an incremental discrete PID:
  - process values normalized to `0..100`
  - error history `_e1`, `_e2`
  - previous internal output `_o_prev`
  - coefficients `b0`, `b1`, `b2`
  - output clamping and anti-windup feedback during output publishing
- Existing PID runtime tests live in `src/HornetStudio.Host.Tests/Program.cs`.
- The item write-model cleanup is intentionally deferred to a separate plan.

## Tasks
1. Inspect `PidControllerRuntime` state fields and map them to the FunkyLib state model.
2. Replace the current calculation state with discrete PID state:
   - previous error 1
   - previous error 2
   - previous internal output
   - optional last compute timestamp if needed
   - filtered derivative/time value state if preserving the reference filter behavior
3. Update PID parameter derivation to match the FunkyLib/CHR formula.
4. Replace the current `bias + P + I + D` output calculation with the incremental coefficient calculation:
   - compute normalized `pv`
   - compute normalized `sp`
   - compute `error = sp - pv`
   - compute `b0`, `b1`, `b2`
   - update internal output from previous output and error history
5. Preserve output publishing interval behavior:
   - compute internal output every compute tick
   - write scaled output only when `OutputIntervalMs` allows it
6. Apply clamping and anti-windup in a way equivalent to the reference:
   - clamp internal output to `0..100`
   - feed saturation difference back into previous output state
7. Strengthen validation:
   - reject `Ks <= 0`
   - reject `Tu <= 0`
   - reject `Tg <= 0`
   - reject `NaN` and infinity for source, setpoint, parameters, and computed output
   - publish clear English runtime alerts
8. Reset PID state consistently when the controller transitions from running to stopped.
9. Update focused host tests to assert the new behavior and guard against the previous setpoint-bias behavior.
10. Run focused host self-tests and solution build.

## Acceptance Criteria
- `PidControllerRuntime` no longer uses `bias = normalizedSetpoint` as a direct output baseline.
- PID parameters are derived with the FunkyLib/CHR formula.
- Source and setpoint are normalized to the internal `0..100` controller range.
- Output is scaled from internal `0..100` to `OutMin..OutMax`.
- Output clamps correctly at both configured limits.
- Invalid PID parameters and invalid numeric inputs produce clear runtime alerts.
- Owned setpoint writes still affect the PID calculation.
- Existing run/start/stop runtime behavior remains intact.
- Focused PID host tests pass.

## Risks / Watchouts
- First compute tick behavior may differ from the current implementation because the FunkyLib reference initializes history before producing a meaningful output.
- Existing tests that only assert "output is positive" may be too weak and should be replaced with behavior-specific expectations.
- The FunkyLib reference contains a D-filter implementation that filters `tv`, not the measured derivative; preserve intent carefully and document any intentional deviation in code comments only if necessary.
- Output interval and compute interval are currently handled by one timer plus timestamp gating; do not introduce multiple timers unless there is a concrete runtime need.
- Negative process gain is currently accepted by the HornetStudio runtime but rejected by the reference. This plan treats `Ks <= 0` as invalid.
- If existing saved controllers relied on negative `Ks`, this is a behavioral breaking change and should be mentioned in the final implementation summary.
