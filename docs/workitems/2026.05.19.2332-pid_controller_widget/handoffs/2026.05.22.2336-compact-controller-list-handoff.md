# IMPLEMENTATION HANDOFF

## Goal
Reduce the `ControllerWidget` controller list entries to compact one-line rows that show the controller type, controller name, and row actions. Detailed PV/SET/OUT, tuning, state, and alert information should no longer make every list entry tall.

## Scope
- Included:
  - Redesign the `ControllerWidget` list item template to a single-line row.
  - Show `Type` as a compact leading value, preferably `PID`.
  - Keep `Name`, `Edit`, and `Delete` directly visible.
  - Preserve existing add/edit/delete behavior and controller runtime synchronization.
  - Keep the no-controllers empty state.
  - Update widget documentation if it describes the visible list layout.
- Not included:
  - Changing PID runtime behavior.
  - Changing controller persistence, YAML shape, or registry paths.
  - Adding new dependencies.
  - Adding a new detail panel unless the implementation discovers an existing established widget pattern that already supports it with minimal change.
  - Changing the controller editor dialog fields or validation.

## Starting Point
- `ControllerControl.axaml` currently renders each controller row as a tall card:
  - name and type
  - edit/delete buttons
  - path summary
  - parameter summary
  - state/run text
  - optional alert text
- The visual result is too large when multiple controllers are configured.
- `ControllerRow` already exposes the current detail properties:
  - `Name`
  - `TypeText`
  - `PathSummary`
  - `ParameterSummary`
  - `StateText`
  - `AlertText`
  - `HasAlert`
- The requested direction is to limit each entry to one line and surface `Type`.

## Tasks
1. Review the existing `ControllerWidget` visual conventions and comparable compact list templates in nearby widgets.
2. Replace the tall controller row template with a single-line layout:
   - leading type column or small badge
   - controller name column
   - optional compact status indicator only if it fits without increasing row height
   - right-aligned `Edit` and `Delete` buttons
3. Ensure long controller names do not increase row height:
   - use trimming where supported by existing Avalonia patterns
   - avoid wrapping in the compact row
4. Adjust `ControllerRow` view-model text if needed:
   - expose `TypeDisplayText` or equivalent returning `PID` instead of `Type: PID`
   - keep existing detailed properties if still useful elsewhere or for future detail display
5. Decide how to handle hidden details:
   - preferred MVP: remove detail text from the list row entirely
   - optional minimal enhancement: put the details into tooltips if existing tooltip style is already used and this does not add complexity
6. Keep row alert visibility compact:
   - either show a short alert marker/text in the row
   - or omit alert details from the compact row and rely on editor/runtime feedback elsewhere
   - do not reintroduce multi-line alert wrapping in the list row
7. Update widget documentation only where it describes the list presentation.
8. Verify build and inspect the UI manually.

- `src/HornetStudio/docs/widgets/ControllerWidget.md` -> update only if the visible controller list behavior is described.

- `src/HornetStudio/docs/widgets/descriptions/ControllerWidget.md` -> likely unchanged unless it mentions list details directly.

## Implementation Order
1. Inspect compact list styles in existing widgets to match spacing, borders, and button sizing.
2. Modify `ControllerControl.axaml` so each item uses a one-line grid.
3. Add a compact `ControllerRow` type display property only if needed by the XAML.
4. Keep existing edit/delete handlers and command parameters unchanged.
5. Review whether any now-unused display properties should remain; avoid broad cleanup unless the compiler warns.
6. Update relevant widget documentation if the old tall-list behavior is documented.
7. Build the solution.
8. Manually inspect a `ControllerWidget` with multiple PID controllers.

## Technical Constraints
- Use existing Avalonia and widget patterns.
- Do not introduce a new component or service for this layout-only change.
- Do not change controller data model, serialization, runtime paths, or PID algorithm behavior.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- Keep the change local to the `ControllerWidget` UI unless documentation requires a small update.
- New public properties require XML documentation.

## Acceptance Criteria
- Each controller entry is visually one row high under normal content.
- The row shows controller type and name.
- `Edit` and `Delete` remain directly accessible per row.
- Long names are clipped or trimmed instead of wrapping and growing row height.
- PV/SET/OUT, tuning, state, run, and alert details are not permanently rendered as multi-line content inside every row.
- Empty state still appears when no controllers exist.
- Add, edit, delete, save, and reload behavior remains unchanged.
- Existing PID runtime tests are unaffected.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore has already completed.
- If restore is required, run `dotnet restore HornetStudio.sln` first.
- Manually open a screen containing a `ControllerWidget` with at least two PID controllers.
- Confirm that each list entry remains one line and the widget can show more controllers without excessive vertical space.
- Confirm that `Edit` opens the existing controller dialog and `Delete` still asks for confirmation.
- Confirm that saving and reloading the page keeps the controller definitions unchanged.

## Out of Scope
- PID algorithm changes.
- Runtime registry contract changes.
- YAML/persistence changes.
- A full selected-controller detail pane.
- Expand/collapse behavior per controller row.
- New icons, new dependencies, or broad widget theme refactoring.

## Risks / Watchouts
- Removing visible `StateText` and `AlertText` may hide useful runtime diagnostics. If this is a concern, prefer a compact status marker or tooltip rather than restoring multi-line text.
- Avalonia text trimming must be checked visually; if trimming is unsupported in a specific control setup, constrain width through grid columns and avoid wrapping.
- Button sizes should remain usable while not dominating the compact row.
- The row should not become dependent on live runtime synchronization for basic display; configured definitions must still render even before runtime sync.

## Notes
- Recommended row structure:
  - left: compact type text such as `PID`
  - center: controller name
  - right: `Edit` and `Delete`
- The main objective is scannability. Details should move out of the always-visible list entry instead of compressing several summaries into a dense but still noisy row.
