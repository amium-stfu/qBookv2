# IMPLEMENTATION HANDOFF

## Goal
Add a first Button `RunFunction` MVP so a Button can execute a declarative YAML function selected from the widget-independent `FunctionRegistry`. This proves that `FunctionRegistry` is a real shared lookup surface and not only a `Functions` widget implementation detail.

## Scope
- Included:
  - Add a Button/Interaction action option for `RunFunction`.
  - Let users select a declarative function from `FunctionRegistry`.
  - Persist the selected function reference in existing interaction rule/button configuration patterns.
  - Execute declarative YAML functions through the existing `FunctionExecutor`.
  - Resolve the function through `FunctionRegistry` at click time.
  - Provide clear logging/status on success, failure, missing function, invalid YAML, or unsupported function kind.
  - Keep existing Button actions and Interaction Rules behavior unchanged.
  - Add focused tests for parsing/serialization and execution dispatch where practical.
- Not included:
  - Python function execution through `RunFunction`.
  - Python provider implementation.
  - Monitor `RunFunction`.
  - Workflows widget.
  - Function parameters, return values, parallel execution, or orchestration.
  - Changing existing `InvokePythonFunction` behavior.
  - Replacing legacy `ButtonCommand` script behavior.

## Starting Point
- `Functions` and `FunctionRegistry` are already being implemented/stabilized.
- `ButtonControl` currently handles pointer release and executes:
  - interaction rules through `Item.TryExecuteInteraction(...)`
  - legacy `ButtonCommand` through `TryExecuteButtonCommand(...)`
- Interaction rules already support actions such as:
  - `OpenValueEditor`
  - `ToggleBool`
  - `SetValue`
  - `OpenDialog`
  - `CloseDialog`
  - `InvokePythonFunction`
- Existing editors already have target/function picker patterns in:
  - `InteractionRulesEditorDialogWindow`
  - `EditorDialogField`
- `FunctionExecutor` exists for declarative functions.
- `FunctionRegistry` should be widget-independent and queryable without a placed `Functions` widget.

## Tasks
1. Inspect current interaction rule model and serialization:
   - `ItemInteractionAction`
   - `ItemInteractionRule`
   - `ItemInteractionRuleCodec`
   - `EditorDialogField`
   - `InteractionRulesEditorDialogWindow`
2. Add `RunFunction` to `ItemInteractionAction`.
3. Decide persisted fields for the first MVP:
   - preferred: store stable function reference in `TargetPath` or a dedicated compatible field if the model already has one
   - acceptable MVP: store function reference in `FunctionName` and leave `TargetPath` empty/`this` if this matches existing editor wiring better
   - avoid display-name-only persistence when a stable reference is available
4. Add function option retrieval for declarative functions:
   - query `FunctionRegistry`
   - include only `Kind=Declarative` and runnable/valid entries for MVP
   - expose display text and stable reference consistently
5. Update Interaction Rules editor UI behavior:
   - when action is `RunFunction`, show a Function picker
   - hide irrelevant target picker fields if not needed
   - show `Argument` disabled or hidden for MVP
6. Update `EditorDialogField` helper methods:
   - function options for `RunFunction`
   - row option refresh
   - validation defaults
7. Execute `RunFunction` in `FolderItemModel.TryExecuteInteractionRule(...)`:
   - resolve function reference through `FunctionRegistry`
   - reject unsupported kinds such as future Python entries with a clear message
   - load the `FunctionDefinition`
   - execute it through `FunctionExecutor`
   - provide an execution environment that supports MVP step types as already implemented
8. Wire execution environment carefully:
   - `SetValue` should reuse existing write behavior/policies where possible
   - `Delay` must remain async-capable or be handled without blocking the UI
   - `Log` should reuse existing process log handling if available
   - `IfThenElse` behavior should match existing `FunctionExecutor` support
9. If existing interaction execution is synchronous, choose a safe MVP approach:
   - dispatch async execution fire-and-log from click path
   - avoid `.Wait()` / `.Result`
   - prevent unobserved exceptions
10. Add status/logging:
   - start log
   - success log
   - failed/missing/invalid log
11. Update Button and Interaction Rules documentation.
12. Add focused tests:
   - `RunFunction` enum/serialization round-trip
   - option filtering from registry if testable
   - missing function produces clear failure
   - execution dispatch invokes function executor with expected definition if test seams exist
13. Build and manually verify.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation/error messages must be English.
- Do not introduce new dependencies.
- Use `FunctionRegistry`; do not query the `Functions` widget.
- Do not require a placed `Functions` widget.
- Do not change Python function ownership or execution.
- Do not block UI with `.Result` or `.Wait()`.
- Keep the MVP declarative-only.
- Preserve all existing interaction rule behavior.

## Acceptance Criteria
- Interaction Rules offer `RunFunction` as an action.
- `RunFunction` can select a declarative function from `FunctionRegistry`.
- A Button can run the selected declarative function on click.
- The selected function still runs if no `Functions` widget is placed on the page, as long as the YAML function exists.
- Missing/invalid/unsupported function references fail with a clear English message and log entry.
- Existing Button interaction actions still work.
- Existing `InvokePythonFunction` still works unchanged.
- Build and focused tests pass.

## Out of Scope
- Python `RunFunction`.
- Function arguments.
- Function return values.
- Workflow orchestration.
- Monitor action integration.
- Visual function status UI on Buttons.
- Replacing legacy Python button scripts.

## Risks / Watchouts
- Interaction rule serialization is compact/string-based in places; preserve backward compatibility.
- Avoid conflating `RunFunction` with `InvokePythonFunction`.
- Async execution from a Button click must not block UI or swallow exceptions silently.
- Stable function references should not depend on localized display text.
- If duplicate function names exist between primary and legacy directories, follow `FunctionRegistry` precedence.
- `SetValue` execution should reuse existing conversion/write semantics to avoid different behavior from Monitor actions.
