# IMPLEMENTATION HANDOFF

## Goal
Replace the fragile free-text `SetValue` argument editing in Interaction Rules with a dialog-guided editor that validates the selected target and offers type-appropriate operations.

## Scope
- Improve `ItemInteractionAction.SetValue` editing in the Interaction Rules dialog.
- Show a readonly summary for the configured `SetValue` argument and edit it through a dedicated dialog opened by an icon button.
- Detect the selected target item/value shape before opening the editor and present suitable controls for numeric, string, bool, and unknown targets.
- Preserve existing persisted interaction rule format as much as possible by storing the structured operation in the existing `Argument` field.
- Keep existing actions such as `OpenValueEditor`, `ToggleBool`, `RunFunction`, `StopFunction`, `OpenDialog`, `CloseDialog`, and `InvokePythonFunction` behavior unchanged.

Explicitly not included:
- Full redesign of the Interaction Rules editor.
- Migration of Monitor action editing.
- Migration of Function/Workflow `SetValue` editing, except where a small shared codec/runtime helper is needed to avoid divergent semantics.
- New runtime dependencies.

## Starting Point
- Interaction rules are represented by `ItemInteractionRule` and persisted by `ItemInteractionRuleCodec`.
- `SetValue` currently writes `rule.Argument` directly through `PageItemModel.TryApplyInteractionWrite(...)`.
- `InteractionRulesEditorDialogWindow` currently renders `Argument` as an editable `TextBox` for all actions.
- `EditorDialogField` owns interaction row snapshots, target options, and sync back to the parameter value.
- The current model allows invalid combinations such as writing arbitrary text to numeric targets.
- Existing free-text values must remain loadable and executable.

## Tasks
1. Inspect target metadata and write conversion behavior.
2. Define a small `SetValueOperation` model with operations needed by the UI and runtime.
3. Add a codec that can parse both new structured arguments and legacy free-text arguments.
4. Add validation/type classification for interaction `SetValue` targets.
5. Add a dedicated `SetValue` editor dialog using existing Avalonia dialog/theme patterns.
6. Replace the `SetValue` argument text box in Interaction Rules with readonly summary plus edit button.
7. Execute structured `SetValue` operations at runtime while preserving legacy behavior.
8. Add focused tests for parsing, serialization, validation, summary generation, and execution value resolution where practical.
9. Update user-facing documentation for Interaction Rules/Button behavior if the action editor is documented there.
10. Build the solution and run the narrowest available test/self-test path that covers the changed logic.

- `src/HornetStudio.Editor/Models/SetValueOperation.cs` or nearby existing model file -> add the operation model, operation enum, codec, summary formatting, and validation helpers if no better existing location exists.
- `src/HornetStudio.Editor/ViewModels/ItemInteractionEditorRow.cs` -> expose `IsSetValueAction`, `SetValueSummary`, and edit-state notifications needed by the row UI.
- `src/HornetStudio.Editor/ViewModels/EditorDialogField.cs` -> provide target metadata/type information and compatible source item options for the `SetValue` dialog.
- `src/HornetStudio.Editor/Widgets/Common/InteractionRulesEditorDialogWindow.axaml` -> replace `SetValue` argument free-text editing with summary plus themed edit icon button.
- `src/HornetStudio.Editor/Widgets/Common/InteractionRulesEditorDialogWindow.axaml.cs` -> open the new dialog, update row/new-entry argument, refresh summaries, and handle unknown/invalid target cases.

- `src/HornetStudio.Editor.Tests/Program.cs` or existing relevant test location -> add focused tests for codec/validation/execution helpers if the project already covers editor model behavior there.
- `src/HornetStudio/docs/widgets/Button.md`, `src/HornetStudio/docs/widgets/help/Button.help.md`, or other relevant Interaction Rules documentation -> update only if these docs describe `SetValue` editing/action behavior.

## Implementation Order
1. Locate the current write-target metadata source and decide how to classify target types without adding a broad service layer.
2. Add the operation model/codec/summary helper.
3. Add tests for the operation codec before wiring UI.
4. Update runtime execution in `PageItemModel` to support structured operations while keeping legacy literal arguments.
5. Add row-level properties for `SetValue` summary and action visibility.
6. Add the `SetValue` operation editor dialog.
7. Wire the existing Interaction Rules dialog to open the editor for existing rows and the new-entry row.
8. Replace the free-text `Argument` editor only for `SetValue`; keep other actions unchanged.
9. Add/update documentation for user-visible editor behavior.
10. Run build/tests and fix only issues inside this scope.

## Technical Constraints
- Chat and planning notes are German, but code, comments, XML docs, UI text, file names, and identifiers must be English.
- Use existing Avalonia, MVVM, `EditorShellControl`, `ThemeSvgIcon`, and target picker patterns.
- Use existing icons such as `list.svg` or `cog.svg` only if present; otherwise reuse the closest existing themed icon without adding fixed-color assets.
- Do not introduce new dependencies.
- Keep the change localized to Interaction Rules `SetValue` unless a tiny shared helper is required for consistency.
- Public classes/methods added or behaviorally changed need XML documentation.
- Prefer named parameters where calls have multiple unclear parameters.
- Preserve existing persisted interaction rule lines and existing free-text arguments.

## Acceptance Criteria
- Existing legacy rule lines such as `BodyLeftClick|SetValue|this|123` still parse, display, save, and execute.
- `SetValue` rows no longer expose the raw argument as the primary editable control.
- `SetValue` rows show a readable summary and an edit icon button.
- Numeric targets offer at least:
  - set literal value
  - increment by (`+=`)
  - decrement by (`-=`)
  - increment one (`++`)
  - decrement one (`--`)
  - set from compatible item
- String targets offer at least:
  - set literal text
  - append text
  - set from compatible item
- Bool targets avoid duplicating `ToggleBool` as the primary path and offer clear set-true/set-false behavior if supported.
- Unknown targets do not silently allow invalid structured operations; they either show a clear warning or use a legacy literal fallback.
- Item picker options are constrained to compatible source items where target type is known.
- Invalid values are blocked in the dialog with English validation messages before saving.
- Runtime execution logs or returns useful errors when a structured operation cannot be applied.
- Existing non-`SetValue` interaction rule editing remains unchanged.

## Verification
- Run focused tests for the new `SetValueOperation` codec/validator.
- Run any existing editor model tests that cover `ItemInteractionRuleCodec`, interaction rows, or write execution.
- Run `dotnet build HornetStudio.sln --no-restore` after restore is available.
- Manually verify in the editor:
  - open Interaction Rules
  - choose `SetValue`
  - select numeric target and configure `++`, `+=`, `--`, `-=`
  - select string target and configure append
  - select source item through picker
  - save, reopen, and confirm summary/serialized value round-trip

## Out of Scope
- Reworking the storage format into additional persisted rule columns.
- Changing target path persistence.
- Changing function picker behavior.
- Monitor editor redesign.
- Function/Workflow editor redesign.
- Adding more operations such as multiply/divide/clamp unless explicitly requested later.

## Risks / Watchouts
- Target type detection may be incomplete for some item shapes; unknown types must remain safe and understandable.
- Legacy free-text arguments could look similar to structured payloads. Use a clear structured prefix/version to avoid accidental reinterpretation.
- Numeric conversion must match existing write semantics and culture expectations; avoid culture-sensitive parsing bugs.
- Source-item reads should use the same path normalization and resolution policy as current interaction writes.
- The UI should not autosave invalid intermediate dialog state into `ItemInteractionEditorRow.Argument`.
- Avoid a too-generic action editor abstraction until Monitor/Workflow reuse is explicitly planned.

## Notes
- Recommended persistence approach: keep `ItemInteractionRule.Argument` as the single storage field and encode structured operations with a versioned prefix, for example a compact key-value or JSON payload. Legacy strings without the prefix are treated as literal set values.
- Recommended first implementation should be conservative: support numeric/string/bool plus unknown fallback only.
- The existing `ToggleBool` action should remain the preferred toggle operation instead of adding a parallel toggle operation into `SetValue`.
