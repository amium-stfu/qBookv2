# IMPLEMENTATION HANDOFF

## Goal
Complete the Interaction Rules SetValue inline operation model so each target type exposes only the intended simple operations:

- Numeric: `=`, `+`, `-`, `Item`
- String/text: `=`, `+`, `Item`, with an optional separator field for `+`
- Boolean: `true`, `false`, `Item`

## Scope
Included:
- Inline Interaction Rules SetValue operation options.
- Boolean inline behavior using explicit `SetTrue`, `SetFalse`, and `SetFromItem`.
- String inline `+` behavior with optional separator support.
- Structured SetValue payload compatibility.
- Focused editor tests for operation options, row serialization, parsing, validation, and runtime behavior.

Not included:
- A broad SetValue redesign.
- New dependencies.
- Release/version changes.
- Changes to unrelated workflow execution unless required by shared SetValue codec compatibility.

## Starting Point
- `SetValueOperationKind.SetTrue`, `SetFalse`, and `SetFromItem` already exist.
- The full SetValue dialog already offers boolean `Set true`, `Set false`, `Set literal value`, and `Set from item`.
- Inline SetValue options currently expose `SetLiteral` and `SetFromItem` for boolean/string/unknown targets.
- String `AppendText` exists and is valid, but is not currently exposed in inline string options.
- Existing persisted `SetLiteral` boolean rules must continue to load and execute.

Required decisions:
- Inline boolean UI must not offer `=` for new edits. It should offer only `true`, `false`, and `Item`.
- Existing boolean `SetLiteral` values should be mapped into inline `true` or `false` when possible.
- If an existing boolean literal cannot be parsed as boolean-like, keep it loadable through the existing full dialog or preserve it without silently rewriting it until the user changes the operation.

## Tasks
1. Update inline operation options so boolean targets return `true`, `false`, and `Item`.
2. Update inline operation options so string targets return `=`, `+`, and `Item`.
3. Add optional separator support to string `+` if not already implemented from the previous handoff.
4. Map existing boolean literal operations into inline `SetTrue`/`SetFalse` when the literal is parseable.
5. Ensure `Item` for boolean targets uses the existing compatible source filtering.
6. Preserve backward compatibility for old `sv1:` payloads and legacy literal arguments.
7. Add focused tests for bool options, bool row serialization, bool legacy literal mapping, and string separator behavior.

## File-Level Changes
- `src/HornetStudio.Editor/Models/SetValueOperation.cs` -> Return bool inline options as `SetTrue`, `SetFalse`, `SetFromItem`; return string inline options as `SetLiteral`, `AppendText`, `SetFromItem`; extend or preserve conversion helpers for boolean literals.
- `src/HornetStudio.Editor/ViewModels/ItemInteractionEditorRow.cs` -> Ensure inline state can show bool true/false options without a literal editor; map parseable bool literals to the corresponding operation; preserve source item behavior.
- `src/HornetStudio.Editor/Widgets/Common/InteractionRulesEditorDialogWindow.axaml` -> Ensure bool inline rows do not show a literal text box for `true`/`false`; show source picker for `Item`; show string separator input only for string `+`.
- `src/HornetStudio.Editor/Widgets/Common/InteractionRulesEditorDialogWindow.axaml.cs` -> Keep new-row bool defaults and row refresh logic aligned with operation options.
- `src/HornetStudio.Editor/Widgets/Common/SetValueOperationEditorDialogWindow.axaml.cs` -> Keep full dialog compatible with existing bool literal editing; do not remove full-dialog literal support unless explicitly requested.
- `src/HornetStudio.Editor/Models/PageItemModel.cs` -> No bool runtime redesign expected; verify existing `SetTrue`, `SetFalse`, and `SetFromItem` execution remains correct. Add string separator runtime handling if part of this implementation.
- `src/HornetStudio.Editor.Tests/Program.cs` -> Add regression tests for bool inline options and serialization plus string separator behavior.

## Implementation Order
1. Update `SetValueOperationCodec.GetInlineOperationOptions` for boolean and string target kinds.
2. Update `SetValueOperationCodec.ToInlineEditorOperation` so boolean `SetLiteral` values parse to `SetTrue` or `SetFalse` where possible.
3. Adjust `ItemInteractionEditorRow` visibility/state properties so `SetTrue` and `SetFalse` do not show a literal editor.
4. Implement or preserve string separator state for `AppendText`.
5. Update Interaction Rules XAML/code-behind only where the inline UI needs bool or separator visibility changes.
6. Add tests for the operation matrix and row serialization.
7. Run focused editor tests and build verification.

## Technical Constraints
- User-visible UI text and validation messages must be English.
- Chat responses remain German.
- Do not introduce dependencies.
- Keep changes localized to SetValue Interaction Rules and shared SetValue codec/runtime paths.
- Existing serialized SetValue data must remain parseable.
- Do not remove full-dialog boolean literal support unless a later request explicitly asks for it.

## Acceptance Criteria
- Numeric inline SetValue shows `=`, `+`, `-`, `Item`.
- String inline SetValue shows `=`, `+`, `Item`.
- Boolean inline SetValue shows `true`, `false`, `Item`.
- Boolean inline `true` serializes to a structured `SetTrue` operation.
- Boolean inline `false` serializes to a structured `SetFalse` operation.
- Boolean inline `Item` serializes to `SetFromItem` and uses compatible source filtering.
- Existing boolean `SetLiteral` payloads such as `true`, `false`, `1`, and `0` still load without breaking.
- String `+` supports the planned optional separator field and remains backward-compatible when no separator is present.

## Verification
- Run `dotnet run --project src/HornetStudio.Editor.Tests/HornetStudio.Editor.Tests.csproj` or the repository's current focused editor test command if different.
- Run `dotnet build HornetStudio.sln --no-restore` after restore is available.
- Manually verify Interaction Rules SetValue rows for numeric, string, and boolean targets.

## Out of Scope
- Python integration changes.
- Target metadata redesign.
- New visual theme system.
- Release packaging or changelog updates unless implementation changes user-facing documentation requirements.

## Risks / Watchouts
- Inline boolean `SetLiteral` compatibility is the main edge case. Avoid rewriting invalid legacy literals silently.
- `SetTrue` and `SetFalse` do not need a literal editor; visibility logic must account for operation kind, not only target kind.
- Unknown target kind should not expose bool-only true/false operations.
- Full dialog and inline editor must not drop fields from structured payloads when reopening existing rules.
