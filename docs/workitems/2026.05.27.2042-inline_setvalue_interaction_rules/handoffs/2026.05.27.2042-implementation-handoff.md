# IMPLEMENTATION HANDOFF

## Goal
Replace the separate Edit SetValue dialog flow in Interaction Rules with an inline, type-aware SetValue editor directly in each Interaction Rule row.

## Scope
Included:
- Inline SetValue editing for configured rows and the New row.
- Use the existing `Function` column as the SetValue operation selector when `Action = SetValue`.
- Support only these SetValue operations in the first inline version:
  - `=` set literal value
  - `+` increase by literal value
  - `-` decrease by literal value
  - `Item` set value from another item
- Use the `Argument` column as the operation argument:
  - TextBox for literal operations.
  - Readonly target/source path plus item picker button for `Item`.
- Type-aware operation availability:
  - Numeric targets: `=`, `+`, `-`, `Item`
  - Boolean targets: `=`, `Item`
  - String targets: `=`, `Item`
  - Unknown targets: keep conservative, prefer `=`, optionally `Item` only if validation remains clear.
- Preserve structured SetValue operation serialization.
- Keep legacy literal loading compatible.

Not included:
- `++` / `--`.
- String append in this iteration.
- Full redesign of Interaction Rules.
- Full item metadata/type policy changes.
- Removing the existing dialog file immediately unless it becomes unused and removal is low risk.

## Starting Point
- The current Interaction Rules SetValue UI shows a readonly summary and opens `SetValueOperationEditorDialogWindow` via a cog button.
- The separate dialog can display stale or incorrect target text (`this`) even when the row target is `udl1.m001.set`.
- The target metadata path is already available on the row, and UDL demo channels now publish `type`.
- The desired behavior is a simpler inline flow where the visible row is also the editable state.

## Tasks
1. Review current SetValue UI bindings in `InteractionRulesEditorDialogWindow.axaml` for configured rows and New row.
2. Extend `ItemInteractionEditorRow` with inline SetValue state derived from `Argument`:
   - selected operation
   - literal argument
   - source item path
   - available operation options based on `SetValueTargetKind`
   - validation/summary refresh
3. Add equivalent inline state for the New row in `InteractionRulesEditorDialogWindow.axaml.cs`.
4. Replace the SetValue summary+cog UI with:
   - operation ComboBox in the existing Function column
   - adaptive Argument editor in the existing Argument column
5. Keep `RunFunction`, `StopFunction`, `InvokePythonFunction`, `OpenDialog`, and `CloseDialog` behavior unchanged.
6. For operation `Item`, wire the existing target/item picker pattern next to the Argument field and store the selected source item as a structured `SetFromItem` operation.
7. Ensure changing target or action refreshes available SetValue operation options and validation.
8. Keep structured serialization through `SetValueOperationCodec`; do not reintroduce free-text operation encoding.
9. Update tests for inline operation state and serialization roundtrip.
10. Run focused build/tests and record any unrelated existing failures.

- `src/HornetStudio.Editor/Widgets/Common/SetValueOperationEditorDialogWindow.axaml` and `.axaml.cs` -> leave unused or remove only if no references remain and removal is low risk.

## Implementation Order
1. Identify all references to `SetValueOperationEditorDialogWindow` and decide whether it remains temporarily unused or is removed.
2. Define a small row-level option model for SetValue operation display if no suitable existing type can be reused.
3. Implement configured-row state:
   - parse `Argument`
   - expose operation options
   - expose literal/source fields
   - write back serialized `Argument` on edits
4. Implement New-row state with the same behavior and no duplicate business logic where practical.
5. Update XAML:
   - show operation ComboBox in Function column for SetValue
   - show literal TextBox or source picker in Argument column
   - remove/hide cog button from SetValue path
6. Refresh operation options when row target changes or when target type metadata is refreshed.
7. Add tests around `=`, `+`, `-`, `Item`, target-kind filtering, and legacy literal loading.
8. Run validation.

## Technical Constraints
- Use existing Avalonia patterns in the dialog.
- Use existing `SetValueOperationCodec` for persistence.
- Keep UI text in English.
- Do not add dependencies.
- Do not change public runtime behavior for non-SetValue actions.
- Keep the first inline version deliberately small: no append, no increment/decrement shorthand, no broad rule editor redesign.
- Prefer localized changes over introducing a new service layer.

## Acceptance Criteria
- For `Action = SetValue`, the Function column shows operation choices instead of `-`.
- Numeric target with `type = float` offers `=`, `+`, `-`, and `Item`.
- Boolean and string targets do not offer numeric `+` or `-`.
- Selecting `+` with argument `18` stores a structured increase operation and displays/roundtrips correctly.
- Selecting `-` with argument `18` stores a structured decrease operation and displays/roundtrips correctly.
- Selecting `Item` shows an item/source picker next to the Argument field and stores a structured source-item operation.
- Legacy literal arguments still load as `=` with the literal value.
- The separate Edit SetValue cog is no longer required for normal SetValue editing.
- The visible row target is the only target used for type-aware operation options.

## Out of Scope
- Adding append for strings.
- Adding shorthand increment/decrement operations.
- Changing UDL/ItemClient type metadata policy.
- Reworking the full Interaction Rules layout beyond the SetValue row behavior.
- Release/version changes.

## Risks / Watchouts
- The current full console test run may already have unrelated failures; keep reporting separate.
- Two-way synchronization between UI fields and serialized `Argument` can easily cause loops. Keep setters guarded.
- The New row and configured rows should share behavior as much as practical, but avoid premature abstraction.
- Removing the dialog too early can create extra churn; first make the inline path work, then remove only if no references remain.
- The Function column already has different meanings for different actions. Keep visibility conditions explicit so RunFunction/Python/Dialog rows do not regress.
