# IMPLEMENTATION HANDOFF

## Goal
Extend Interaction Rules SetValue so string/text targets support a visible `+` operation with an optional free-text separator, while numeric targets keep their existing green function-style operation affordance and type-specific operation set.

## Scope
Included:
- Interaction Rules SetValue inline editor behavior for string/text targets.
- Structured SetValue payload support for an optional separator value.
- Runtime resolution for string append/concat behavior.
- Focused editor tests for codec, inline row state, validation, and runtime behavior.

Not included:
- New external dependencies.
- Broad Interaction Rules redesign.
- Workflow widget SetValue behavior unless it shares the same codec/runtime path and must be kept compatible.
- Release/versioning changes.

## Starting Point
- `SetValueOperationKind.AppendText` already exists and is valid for `SetValueTargetKind.String`.
- The full SetValue operation dialog already exposes `AppendText` for string targets.
- The inline Interaction Rules editor currently exposes only `=` and `Item` for string/unknown/boolean targets.
- Structured SetValue arguments use the `sv1:` JSON payload in `SetValueOperationCodec`.
- Runtime SetValue resolution is handled in `PageItemModel.TryResolveSetValueOperationValue`.

Required assumption:
- The separator is only meaningful for string `+` operations. It should be applied between the current target text and the appended value only when both sides are non-empty. Empty separator means direct append.

## Tasks
1. Add separator support to the SetValue operation model and structured codec.
2. Expose `+` for string targets in the inline Interaction Rules SetValue operation list.
3. Add inline row state for the separator field and serialize it into the structured SetValue payload.
4. Show the separator text field only when the selected SetValue operation is string `+`.
5. Update runtime resolution so string `+` writes `current + separator + appendedText`, with separator omitted when current text or appended text is empty.
6. Update summaries and validation messages so string `+` remains understandable.
7. Add focused tests for codec roundtrip, inline options, row serialization, and runtime string append with separator.

## File-Level Changes
- `src/HornetStudio.Editor/Models/SetValueOperation.cs` -> Add optional separator field, include it in structured serialization/parsing, expose `AppendText` in string inline options, update summaries and validation helpers.
- `src/HornetStudio.Editor/ViewModels/ItemInteractionEditorRow.cs` -> Add separator state, visibility property, synchronization from/to serialized SetValue argument, and reset behavior when changing operation kind.
- `src/HornetStudio.Editor/Widgets/Common/InteractionRulesEditorDialogWindow.axaml` -> Add a compact separator input bound to the row/new-rule separator state for string `+`.
- `src/HornetStudio.Editor/Widgets/Common/InteractionRulesEditorDialogWindow.axaml.cs` -> Populate, preserve, and validate separator state for existing and new rows.
- `src/HornetStudio.Editor/Widgets/Common/SetValueOperationEditorDialogWindow.axaml` -> Add separator input for full SetValue operation dialog if it edits `AppendText`.
- `src/HornetStudio.Editor/Widgets/Common/SetValueOperationEditorDialogWindow.axaml.cs` -> Include separator in dialog state and serialized operation.
- `src/HornetStudio.Editor/Models/PageItemModel.cs` -> Resolve `AppendText` with separator-aware concatenation.
- `src/HornetStudio.Editor.Tests/Program.cs` -> Add focused regression tests.

## Implementation Order
1. Extend `SetValueOperation` with a `Separator` property and keep parse/serialize backward-compatible when it is absent.
2. Update `SetValueOperationCodec.GetInlineOperationOptions` so `SetValueTargetKind.String` returns `=`, `+`, and `Item`.
3. Update summaries so `AppendText` can show separator context when present.
4. Update `ItemInteractionEditorRow` inline state and serialization for `AppendText`.
5. Update Interaction Rules XAML/code-behind to display the separator input only for string `+`.
6. Update full SetValue operation dialog only if existing dialog editing would otherwise drop the separator.
7. Update runtime append behavior in `PageItemModel`.
8. Add tests and run focused editor tests, then solution build if restore/build state allows.

## Technical Constraints
- Keep user-visible UI text and validation messages in English.
- Keep chat/docs explanations in German where applicable.
- Do not add dependencies.
- Keep changes localized to SetValue Interaction Rules behavior.
- Preserve legacy literal arguments and existing `sv1:` payloads without separator.
- Use existing Avalonia binding and view model patterns.
- Use named parameters for unclear method calls, especially where arguments have similar types.

## Acceptance Criteria
- String/text SetValue inline rules show `=` and `+` operations.
- Selecting string `+` provides an append text field and a free-text separator field.
- Numeric SetValue behavior remains unchanged.
- Existing persisted SetValue rules without separator continue to load and run.
- A string `+` operation with current value `A`, separator `, `, and appended value `B` writes `A, B`.
- A string `+` operation with an empty current value writes only the appended value.
- A string `+` operation with an empty appended value does not add a trailing separator.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore is available.
- Run `dotnet run --project src/HornetStudio.Editor.Tests/HornetStudio.Editor.Tests.csproj` or the repository's existing focused editor test command if different.
- Manually verify the Interaction Rules editor for a string target: operation list, separator visibility, save/reopen behavior.

## Out of Scope
- Changing Python integration.
- Changing target type metadata classification beyond what is needed for existing string detection.
- Adding new runtime dependencies.
- Redesigning all SetValue operation dialogs.

## Risks / Watchouts
- The inline Interaction Rules editor and full SetValue operation dialog can accidentally diverge. If the full dialog can open/edit the same operation, it must preserve the separator even if the inline editor created it.
- `AppendText` currently uses `LiteralValue`; separator support should not break existing payloads.
- Unknown targets currently behave like string/boolean fallback in some UI paths. Do not expose string-only separator behavior for unknown targets unless the target is explicitly classified as string.
- Avoid fixed green colors. If visual color changes are needed, bind through existing theme resources.
