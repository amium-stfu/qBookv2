# IMPLEMENTATION HANDOFF

## Goal
Add published `type` metadata to Enhanced Signals and Custom Signals so downstream target selection and value editing can resolve signal value types consistently.

## Scope
Implement `type` metadata for:

1. Enhanced Signal runtime items and relevant child channels.
2. Custom Signal runtime items based on the configured `CustomSignalDataType`.
3. Manual trigger items for Custom Signals.
4. Focused tests or self-test coverage for the new published metadata.
5. Relevant widget documentation updates if user-visible behavior or documented registry metadata changes.

Do not introduce a new type system or new dependencies.

## Starting Point
- `TargetValueType` and `TargetValueTypes` already define canonical metadata names such as `float`, `bool`, and `string`.
- `CustomSignalDefinition` already stores `DataType` as `Number`, `Boolean`, or `Text`.
- `CustomSignalsControl.PublishSignalSnapshot(...)` publishes custom signal registry items but does not currently set `item.Properties["type"]`.
- `CustomSignalsControl.PublishManualTriggerSnapshot(...)` publishes a writable manual trigger item and should expose `type = "bool"`.
- `EnhancedSignalRuntime` creates an `ExtendedSignalModule` and publishes numeric runtime values.
- `ExtendedSignalModule` creates child channels but currently does not assign explicit `type` metadata.
- Enhanced Signal processing is numeric, so the effective type for main numeric channels is `float`.

## Tasks
1. Add a small local mapping from `CustomSignalDataType` to registry `type` text in the Custom Signals widget code.
2. Publish `type` metadata for Custom Signal items:
   - `Number` -> `float`
   - `Boolean` -> `bool`
   - `Text` -> `string`
3. Publish `type = "bool"` for Custom Signal manual trigger items.
4. Add explicit type metadata to Enhanced Signal module channels:
   - main module/value-bearing numeric channels: `float`
   - `raw`, `read`, `set`, `out`: `float`
   - command-like boolean controls where applicable: `bool`
   - `state`, `alert`, and other textual diagnostics: `string`
   - integer timing/count channels where applicable: `int`
5. If adding an explicit `Type` property to `ExtendedSignalDefinition`, keep the default `float`, persist it through `ExtendedSignalDefinitionDocument`, and do not expose non-float options unless there is a concrete UI requirement.
6. Prefer constants or existing canonical strings over duplicated string literals when the existing code already provides a suitable convention.
7. Add or update focused tests/self-tests covering published metadata for both Custom Signals and Enhanced Signals.
8. Update documentation only where the published registry metadata is described.

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/CustomSignals/CustomSignalsControl.axaml.cs` -> publish `type` metadata for custom signal snapshots and manual trigger snapshots.
- `src/HornetStudio.Host/ExtendedSignalModule.cs` -> set default `type` metadata for Enhanced Signal module child channels at creation time.
- `src/HornetStudio.Host/EnhancedSignalRuntime.cs` -> verify runtime-created/configuration child items keep correct `type` metadata; add assignments here only for items not covered by `ExtendedSignalModule`.
- `src/HornetStudio.Contracts/ExtendedSignalDefinitions.cs` -> only if an explicit persisted Enhanced Signal `Type` property is required; default must be `float`.
- `src/HornetStudio.Editor/Models/ExtendedSignalDefinition.cs` -> only if `ExtendedSignalDefinitionDocument` must persist `Type`; default must be `float` and legacy definitions must remain compatible.
- `src/HornetStudio.Editor.Tests/Program.cs` or `src/HornetStudio.Host.Tests/Program.cs` -> add focused assertions for published `type` metadata using the existing console self-test style.
- `src/HornetStudio/docs/widgets/CustomSignals.md` and/or `src/HornetStudio/docs/widgets/help/CustomSignals.help.md` -> update only if registry metadata is documented there.
- `src/HornetStudio/docs/widgets/EnhancedSignals.md` and/or `src/HornetStudio/docs/widgets/help/EnhancedSignals.help.md` -> update only if registry metadata is documented there.

## Implementation Order
1. Inspect existing published item/type conventions in `UdlModule`, Python host value publishing, and target metadata handling to reuse names exactly.
2. Implement Custom Signal `type` publishing with a private mapping helper in `CustomSignalsControl`.
3. Implement Enhanced Signal `type` publishing in `ExtendedSignalModule` first, then fill any runtime-created gaps in `EnhancedSignalRuntime`.
4. Decide whether `ExtendedSignalDefinition.Type` is necessary. Prefer no persisted property unless callers need to read it from configuration rather than registry metadata.
5. Add focused tests/self-tests.
6. Update relevant Markdown documentation if it mentions signal registry metadata.
7. Run verification.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible error and validation messages must be English.
- Use existing project patterns and libraries.
- Do not add new dependencies.
- Keep changes scoped to signal `type` metadata.
- Do not reintroduce or depend on `FilteredSignals`.
- Keep public APIs stable unless a persisted Enhanced Signal `Type` property is explicitly needed.

## Acceptance Criteria
- Custom Signal published registry items include `type`.
- Custom Signal `Number` resolves to `float`.
- Custom Signal `Boolean` resolves to `bool`.
- Custom Signal `Text` resolves to `string`.
- Custom Signal manual trigger includes `type = "bool"`.
- Enhanced Signal numeric runtime channels include `type = "float"`.
- Enhanced Signal textual/status channels include `type = "string"` where applicable.
- Existing legacy definitions still load.
- Existing runtime values and write behavior remain unchanged.
- Focused tests or self-tests cover the new metadata.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore is already available.
- If tests are updated in `src/HornetStudio.Editor.Tests`, run the built console test DLL with `dotnet <path-to-dll>`.
- If tests are updated in `src/HornetStudio.Host.Tests`, run the built console test DLL with `dotnet <path-to-dll>`.
- Manually inspect a published Custom Signal and Enhanced Signal registry item, if practical, to confirm `type` metadata is visible.

## Out of Scope
- New UI for choosing Enhanced Signal types.
- A new global type abstraction.
- Broad refactoring of target metadata handling.
- Changing signal formulas, write semantics, paths, or value conversion behavior beyond metadata publication.
- Removing legacy `FilteredSignals` code unless directly touched by existing build failures.

## Risks / Watchouts
- Do not confuse persisted widget `Type` with registry value metadata `type`.
- Enhanced Signals are numerically processed; allowing non-float configuration would be misleading unless the runtime is redesigned.
- Some target selection logic may read `Properties["type"]` from child items rather than parent items, so set metadata on the actual published item that users select.
- Avoid breaking legacy JSON by requiring a new property without a default.
