# DEBUG REPORT

## Problem
`SetValue` remains active after the Monitor rule condition is no longer true. After adding both actions (`OnActivated -> SetValue(1000)` and `OnCleared -> SetValue(0)`), the target value jumps between `0` and `1000`.

## Expected Behavior
If the user expects `SetValue` to represent the rule state, the target should be set to `1000` when the rule activates and set to `0` when the rule clears, without oscillating.

## Actual Behavior
The current implementation executes transition actions correctly by edge:
- `false -> true` executes `OnActivated` actions.
- `true -> false` executes `OnCleared` actions.

If those actions write to an item that is also used by the rule condition, the action can change the next evaluation result and create a feedback loop:
- Rule active -> `OnActivated` writes `1000`.
- Written value clears the condition.
- Rule cleared -> `OnCleared` writes `0`.
- Written value activates the condition again.
- The loop repeats on each monitor evaluation.

  - `ExecuteTransitionActions(...)` runs `OnCleared` actions on `true -> false`.

  - `EvaluateState(...)` reads only the configured source/custom variables; there is no periodic action execution when state does not change.

## Current Hypothesis
- If only `OnActivated -> SetValue(true)` is configured, this is expected behavior and the rule needs a second action: `OnCleared -> SetValue(false)`.
- If `OnCleared -> SetValue(false)` is already configured and the target still remains true, the next debug step should add a focused regression test for the clear transition and inspect whether the rule actually transitions from active to inactive.
- If both actions are configured and the value jumps between both arguments, the most likely root cause is a feedback loop where the SetValue target is also the monitored source or one of the custom formula variables.
- Recommended fix is configuration-level separation: monitor an input/source item and write to a separate output/status item. A later product improvement could add validation or warnings when an action target equals a source/variable path.
- The focused independent-target test disproves a basic transition-state bug in `MonitorRuleRow`: an independent `{C}` remains stable at `0` after `{A}` becomes false and a second evaluation runs.
- Remaining likely causes are configuration/runtime-specific:
  - More than one Monitor rule writes to the same `{C}`.
  - Another widget/process writes to `{A}` or `{C}`.
  - The configured `{A}` path resolves to a different runtime item than expected.
  - `{A}` is indirectly affected by the write to `{C}` through broker/write-back/application logic.
