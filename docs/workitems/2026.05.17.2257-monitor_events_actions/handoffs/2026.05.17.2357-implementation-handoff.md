# IMPLEMENTATION HANDOFF

## Goal
Extend Monitor transition actions beyond `WriteLog` by adding `SetValue` and `InvokeFunction`, using an editor experience similar to Button/Signal interaction rules while keeping Monitor-specific triggers (`OnActivated`, `OnCleared`).

## Scope
Included:
- Add `SetValue` as a Monitor action type.
- Add `InvokeFunction` as a Monitor action type.
- Keep existing `WriteLog` action behavior.
- Extend Monitor action persistence with fields needed by all action types:
  - `TargetPath`
  - `FunctionName`
  - `Argument`
- Update the Monitor Actions tab to show action-specific fields:
  - `WriteLog`: log target picker.
  - `SetValue`: item target picker and argument.
  - `InvokeFunction`: function target picker, function selector, and argument.
- Reuse existing target path normalization and function lookup patterns from Button/Signal interaction rules where practical.
- Execute `SetValue` and `InvokeFunction` on configured transitions.
- Update YAML, documentation, and tests.

Explicitly not included:
- A full generic action-rule framework shared by all widgets.
- Migrating Button/Signal interaction rules.
- New action types beyond `WriteLog`, `SetValue`, and `InvokeFunction`.
- Signal widget blinking or visual monitor-state binding.
- Alarm acknowledgement, history, or dashboard.
- New dependencies.

## Starting Point
- Monitor already has `Rule` and `Actions` tabs.
- Current Monitor action editing supports `WriteLog` only.
- Current Monitor action model already has transition triggers such as `OnActivated` and `OnCleared`.
- Button/Signal interaction rules already support action-like configuration with:
  - target path
  - function name
  - argument
  - Python function option lookup
- Existing interaction code uses `ItemInteractionAction.InvokePythonFunction`; Monitor should use the requested action name `InvokeFunction`.

Assumptions:
- `SetValue` writes the configured `Argument` to the configured `TargetPath`.
- `InvokeFunction` calls a function on the configured function-capable target using `FunctionName` and `Argument`.
- `InvokeFunction` should use the same function discovery path as existing interaction rules, including `PythonClientRuntimeRegistry.GetFunctionNames(...)` and `ApplicationExplorerRuntime.ResolveInteractionTargetPath(...)` if those are the current patterns.
- `WriteLog` should continue to use rule-level `EventId`, `EventText`, and log level/severity.
- Monitor action execution should remain transition-based and ordered by configured action list order.
- User-visible labels should be English: `SetValue`, `InvokeFunction`, `Target`, `Function`, `Argument`.

## Tasks
1. Inspect the current Monitor action model and action-row view model.
2. Add `SetValue` and `InvokeFunction` to the Monitor action type enum.
3. Add action fields to the persisted action model if missing:
   - `TargetPath`
   - `FunctionName`
   - `Argument`
4. Preserve backward compatibility for existing `WriteLog` actions.
5. Update action normalization:
   - `WriteLog` target uses persisted layout target path/log target handling.
   - `SetValue` target uses normal item target path normalization.
   - `InvokeFunction` target uses the same interaction-function target normalization as Button/Signal.
6. Update the Actions tab UI:
   - Show action selector with `WriteLog`, `SetValue`, `InvokeFunction`.
   - Show target picker for all action types.
   - Show argument field for `SetValue` and `InvokeFunction`.
   - Show function selector only for `InvokeFunction`.
   - Refresh available function options when the target changes.
7. Add validation:
   - `WriteLog` requires a log target.
   - `SetValue` requires a target path.
   - `InvokeFunction` requires target path and function name.
8. Implement `SetValue` execution.
9. Implement `InvokeFunction` execution by reusing existing interaction/function invocation infrastructure where possible.
10. Add focused tests for persistence and execution.
11. Update YAML examples and Monitor docs/help.

## Technical Constraints
- Chat and implementation summaries must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation and error messages must be English.
- Do not add dependencies.
- Keep Monitor triggers separate from Button/Signal interaction events.
- Do not migrate Button/Signal interaction rules in this work.
- Reuse existing target/function resolution patterns instead of inventing new ones.
- Runtime path segments and persisted technical identifiers must remain valid lowercase snake_case where they represent paths.
- Action execution must remain transition-based, not continuous during active state.

## Acceptance Criteria
- Monitor Actions tab offers `WriteLog`, `SetValue`, and `InvokeFunction`.
- `SetValue` action shows target picker and argument field.
- `InvokeFunction` action shows target picker, function selector, and argument field.
- Function options refresh when the `InvokeFunction` target changes.
- Saving and reopening a rule preserves all action fields.
- `SetValue` executes on `OnActivated` and/or `OnCleared` according to configuration.
- `InvokeFunction` executes on `OnActivated` and/or `OnCleared` according to configuration.
- Existing `WriteLog` actions continue to work.
- Existing Monitor rules without the new fields still load safely.
- Documentation and YAML examples describe all three action types.

  - Confirm `WriteLog` still writes using rule event metadata.
  - Save/reopen and confirm all actions persist.

## Out of Scope
- Generic shared action editor extraction.
- Button/Signal interaction rule migration.
- Additional action types such as `PublishEvent`, `SendInput`, or notification actions.
- Repeating action execution while a rule remains active.
- Alert acknowledgement/history/dashboard.

## Risks / Watchouts
- `SetValue` argument conversion must match existing item write semantics; avoid ad hoc conversion if an existing write helper is available.
- `InvokeFunction` may depend on Python/ApplicationExplorer runtime target resolution; reuse existing interaction logic to avoid diverging behavior.
- Function target persistence may require different normalization than normal item target paths, as existing interaction rules already distinguish `InvokePythonFunction`.
- Action rows need action-type-aware validation so hidden fields do not accidentally block save.
- Execution failures should be logged or surfaced without breaking the monitor evaluation loop.
