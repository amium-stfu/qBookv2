# IMPLEMENTATION HANDOFF

## Goal
Refine the already implemented `Monitor` widget event model so monitor rules produce event state, transition triggers can execute multiple actions, and active events can be published as aggregate lists such as `fatal_active`.

## Scope
Included:
- Keep the existing `Monitor` widget and rule evaluation model.
- Treat string-literal comparison in custom formulas as already implemented and preserve it.
- Move the current direct logging behavior into transition-based actions.
- Keep `EventId`, `EventText`, and severity/log-level semantics on the monitor rule because they describe the event state itself.
- Add transition triggers for monitor rule state changes:
  - `OnActivated`
  - `OnCleared`
- Support multiple actions per trigger.
- Implement or preserve `WriteLog` as the first concrete action type.
- Add a structure that can later support additional actions such as function calls without implementing all future action types now.
- Publish per-rule event state and monitor-level active event aggregates.
- Update persistence, editor UI, YAML, documentation, and tests.

Explicitly not included:
- Signal widget blinking or visual monitor-state binding.
- Full alert manager, acknowledgement, history, dashboard, or notification system.
- Implementing every future action type such as function calls unless a minimal placeholder enum/config is needed.
- Repeating log entries while a rule remains active.
- Release/version/tag work.
- New dependencies.

## Starting Point
- `Monitor` is already implemented as a widget.
- `MonitorDefinition` currently contains rule-local settings including:
  - `RefreshRateMs`
  - `TimeoutMs`
  - `Mode`
  - `LowerLimit`
  - `UpperLimit`
  - `InhibitMs`
  - `CustomFormula`
  - `CustomVariables`
  - `EventId`
  - `EventText`
  - `TargetLog`
  - `LogLevel`
- `MonitorControl` currently publishes per-rule runtime items and writes to `TargetLog` directly from the rule when the rule becomes active.
- `MonitorLogLevel` currently supports `Debug`, `Info`, `Warning`, `Error`, and `Fatal`.
- Custom formula string comparisons with quoted string literals have been implemented and should remain supported.
- Current direct logging fields are too narrow because multiple actions per transition are required.

Assumptions:
- The persisted widget `Type` remains `Monitor`.
- Runtime paths remain under `studio.<page>.monitor.<monitor_widget>`.
- Rule event metadata should remain rule-level:
  - `EventId`
  - `EventText`
  - `Severity` or the existing `LogLevel`
- Prefer renaming `LogLevel` to `Severity` only if the implementation can provide compatibility for old persisted `LogLevel` values. Otherwise keep `LogLevel` and document that it also acts as event severity.
- `WriteLog` should use the rule event metadata by default.
- Logging should occur on configured transitions only, not automatically for every active state.
- Multiple actions with the same trigger must execute in configured order.

## Tasks
1. Inspect the current Monitor implementation and identify the direct logging path.
2. Introduce monitor transition and action models:
   - `MonitorActionTrigger` with `OnActivated` and `OnCleared`.
   - `MonitorActionType` with at least `WriteLog`.
   - `MonitorActionDefinition` with trigger, action type, and action-specific fields.
3. Move `TargetLog` from rule-level direct logging into `WriteLog` action configuration.
4. Keep `EventId`, `EventText`, and severity/log-level on the rule.
5. Add action list persistence to each monitor rule.
6. Add backward-compatible loading:
   - If an old rule has `TargetLog`, create an equivalent `WriteLog` action with `Trigger = OnActivated`.
   - Preserve old `LogLevel` as rule severity/log-level.
7. Update runtime evaluation to detect state transitions:
   - inactive to active -> execute `OnActivated` actions.
   - active to inactive -> execute `OnCleared` actions.
8. Execute all matching actions in configured order.
9. Implement `WriteLog` action using the rule `EventId`, `EventText`, and severity/log-level.
10. Publish per-rule runtime state including active status and event metadata.
11. Publish monitor-level aggregate active lists grouped by severity/log-level.
12. Update the Monitor editor UI to manage multiple actions per rule.
13. Update YAML examples and documentation.
14. Add focused tests for actions, compatibility, aggregates, and string comparison preservation.

## Implementation Order
1. Add `MonitorActionTrigger`, `MonitorActionType`, and `MonitorActionDefinition` to the Monitor model.
2. Add `Actions` to `MonitorDefinition` and `MonitorDefinitionDocument`.
3. Implement backward-compatible deserialization from old direct `TargetLog` rules.
4. Update serialization to write only the new `Actions` model for logging targets.
5. Refactor `MonitorControl` so transition detection dispatches actions instead of direct rule-level logging.
6. Implement `WriteLog` action execution.
7. Add aggregate runtime publishing for active event ids grouped by severity/log-level.
8. Update the rule editor UI with an actions table/list.
9. Update YAML and documentation.
10. Add focused tests.
11. Run verification commands.

## Technical Constraints
- Chat and implementation summaries must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation and error messages must be English.
- Do not add dependencies.
- Follow existing Monitor, CustomSignals, EnhancedSignals, and ProcessLog patterns.
- Runtime path segments must be lowercase snake_case.
- Preserve compatibility for layouts produced by the already implemented Monitor version.
- Avoid turning this into a full alert manager.
- Actions must be explicit and transition-based.
- Do not log repeatedly while a rule remains active unless a future repeat action setting is explicitly added.

## Acceptance Criteria
- Existing Monitor rules with direct `TargetLog` still load and behave as an `OnActivated -> WriteLog` action.
- New Monitor rules can configure zero, one, or multiple actions.
- Multiple actions can share the same trigger.
- `OnActivated` actions run only when a rule transitions from inactive to active.
- `OnCleared` actions run only when a rule transitions from active to inactive.
- `WriteLog` action writes using the rule `EventId`, `EventText`, and severity/log-level.
- Rule event metadata is published independently of whether actions exist.
- Monitor publishes aggregate active event id lists grouped by severity/log-level.
- Aggregates use lowercase snake_case runtime item names.
- Custom formulas with string comparisons such as `({A} == true) && ({B} == "ABC")` continue to work.
- Documentation and YAML examples show the new action model.

## Verification
- `dotnet run --project src\HornetStudio.Editor.Tests\HornetStudio.Editor.Tests.csproj --no-restore`
- `dotnet build HornetStudio.sln --no-restore`
- Manual verification:
  - Load a layout with an existing Monitor rule that has direct `TargetLog` and confirm it migrates to a `WriteLog` action.
  - Add a rule with two `OnActivated` actions and confirm both execute once on activation.
  - Add an `OnCleared` `WriteLog` action and confirm it executes once when the condition clears.
  - Trigger active rules with `Fatal`, `Error`, and `Warning` severity/log-level and inspect aggregate runtime outputs.
  - Test a custom formula with a string literal comparison.

## Out of Scope
- New action implementations beyond `WriteLog`, except placeholder model values if explicitly useful.
- Signal widget visual state binding.
- Alarm acknowledgement and lifecycle management beyond active/cleared transitions.
- Persistent alarm history.
- Notification integrations.
- Release versioning.

## Risks / Watchouts
- If `LogLevel` is renamed to `Severity`, compatibility must map old persisted `LogLevel` safely.
- Aggregated string outputs such as `fatal_active = "1,3,7"` are simple but require stable ordering to avoid flicker and noisy downstream changes.
- Duplicate `EventId` values should be either allowed and documented or validated per Monitor widget. Decide before implementation.
- Direct logging migration must avoid creating duplicate actions on repeated load/save cycles.
- Action execution should not block the monitor evaluation loop for long-running future actions.
- Do not remove existing string-literal formula support while touching the formula tests.
