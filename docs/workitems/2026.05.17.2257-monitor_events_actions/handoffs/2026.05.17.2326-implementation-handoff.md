# IMPLEMENTATION HANDOFF

## Goal
Improve the `Monitor` rule editor structure by splitting rule configuration and transition actions into tabs, and align the action editing experience with the existing Button/Signal interaction rules without directly coupling Monitor actions to the current `InteractionRulesEditorDialogWindow`.

## Scope
Included:
- Refactor the Monitor rule dialog into a tabbed layout:
  - `Rule`
  - `Actions`
- Keep rule condition, timing, event metadata, variables, and formula editing on the `Rule` tab.
- Move all transition action editing to the `Actions` tab.
- Make Monitor actions visually and behaviorally similar to Button/Signal interaction rules where practical.
- Support multiple Monitor actions per trigger.
- Preserve existing Monitor action model from the previous `monitor_events_actions` work.
- Evaluate whether small reusable action-editor pieces can be extracted without forcing Button/Signal migration.
- Update documentation, YAML examples, and tests if UI/persistence behavior changes.

Explicitly not included:
- Migrating Button/Signal `InteractionRulesEditorDialogWindow` to a new generic action editor in the same change.
- Replacing `ItemInteractionRule` with Monitor action models.
- Adding new action types beyond the current Monitor action scope unless already planned.
- Signal widget blinking or visual monitor-state binding.
- Full alert manager, acknowledgement, history, or dashboard.
- New dependencies.

## Starting Point
- Monitor currently has a dedicated `MonitorEditorDialogWindow`.
- The dialog has become dense because rule settings, mode-specific fields, variables/formula, event metadata, and actions share one vertical layout.
- Button/Signal interaction rules already use `InteractionRulesEditorDialogWindow`, but that dialog is tied to `EditorDialogField`, `ItemInteractionEditorRow`, and `ItemInteractionRuleCodec`.
- Monitor actions use different triggers (`OnActivated`, `OnCleared`) and different action fields than Button/Signal interactions.
- The existing Monitor action editor currently supports adding a log action, but the desired model must support multiple actions per trigger and future action types.

Assumptions:
- The Monitor dialog window title remains `Monitor Rule`.
- Tab labels should be user-visible English: `Rule` and `Actions`.
- `Rule` tab remains the default selected tab.
- The action model remains monitor-specific for now.
- Any reusable action editor should be extracted only as a generic UI helper/control if it reduces duplication immediately.
- Existing Button/Signal behavior must remain unchanged.

## Tasks
1. Inspect the current `MonitorEditorDialogWindow.axaml` and identify which controls belong on `Rule` vs `Actions`.
2. Add a `TabControl` or existing project tab pattern to the Monitor rule dialog.
3. Move general rule fields to the `Rule` tab:
   - `Name`
   - `Mode`
   - `RefreshRateMs`
   - `TimeoutMs`
   - `InhibitMs`
   - `EventId`
   - event severity/log-level
   - `EventText`
4. Keep mode-specific condition sections on the `Rule` tab:
   - `Default`: source path, lower limit, upper limit.
   - `Custom`: variables and formula editor.
5. Move all action rows and action add/remove controls to the `Actions` tab.
6. Rework the `Actions` tab layout to match the interaction-rule style:
   - configured actions list
   - new action row or add/edit affordance
   - trigger selector
   - action selector
   - action-specific target/value fields
7. Support multiple configured actions with the same trigger.
8. Keep `WriteLog` target selection working through the existing target picker.
9. Add validation so saving a rule with invalid action configuration shows an English validation message.
10. Decide whether to extract a small reusable action-row control:
    - Extract only if it can be used by Monitor immediately without changing Button/Signal.
    - Do not generalize codecs or persistence in this work.
11. Update Monitor docs/help and screenshots/text descriptions if they mention the old single-page dialog structure.
12. Add/update focused tests for action serialization and multiple actions if not already covered.

## Implementation Order
1. Read the current Monitor rule dialog XAML and code-behind.
2. Split the existing UI into `Rule` and `Actions` tabs without changing persistence first.
3. Verify that existing field bindings still populate and save correctly.
4. Rework the Actions tab layout to support multiple configured action rows clearly.
5. Ensure add/remove/pick target behavior works for each action row.
6. Add validation for action rows.
7. Add or update tests for multiple actions and action persistence.
8. Update Monitor help/docs.
9. Run focused tests and build.

## Technical Constraints
- Chat and implementation summaries must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation and error messages must be English.
- Do not add dependencies.
- Keep Button/Signal interaction rules unchanged unless a small shared UI helper is clearly safe.
- Do not force a generic action framework across widgets in this step.
- Follow existing Avalonia styling and the existing property dialog tab visual pattern.
- Keep Monitor persistence compatible with layouts already produced by the current Monitor implementation.

## Acceptance Criteria
- Monitor rule dialog has two tabs: `Rule` and `Actions`.
- `Rule` tab contains timing, mode, condition, event metadata, variables, and formula fields.
- `Actions` tab contains only transition action editing.
- Multiple actions can be configured for the same trigger.
- `WriteLog` action target picking still works.
- Existing Monitor rules load and save without losing fields.
- Button and Signal interaction rules behave unchanged.
- The dialog is easier to scan and no longer places actions between rule metadata and custom variables/formula.
- Tests cover multiple action persistence or execution where practical.
- Monitor help documentation reflects the tabbed editor.

## Verification
- `dotnet run --project src\HornetStudio.Editor.Tests\HornetStudio.Editor.Tests.csproj --no-restore`
- `dotnet build HornetStudio.sln --no-restore`
- Manual verification:
  - Open an existing Monitor rule and confirm fields appear on the correct tabs.
  - Add two `OnActivated` actions and save/reopen the rule.
  - Pick a `WriteLog` target from the Actions tab.
  - Configure a `Custom` rule and confirm variables/formula remain on the Rule tab.
  - Confirm Button/Signal interaction rule editing still opens and behaves as before.

## Out of Scope
- A complete reusable action-rule framework shared by all widgets.
- Migration of Button/Signal `InteractionRulesEditorDialogWindow`.
- New Monitor action execution types beyond the existing action model.
- Alarm acknowledgement, history, dashboard, or visual Signal blinking.

## Risks / Watchouts
- A too-generic action editor would expand the scope and risk breaking Button/Signal interaction rules.
- The current InteractionRules editor is coupled to `EditorDialogField`; direct reuse for Monitor would likely create awkward adapters.
- Moving controls between tabs can break bindings if names or data contexts change.
- Action validation must remain action-type aware so future action types can add fields cleanly.
- Keep hidden mode-specific values stable unless the existing dialog already clears them intentionally.
