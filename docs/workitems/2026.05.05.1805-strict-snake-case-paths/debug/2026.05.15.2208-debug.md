# DEBUG REPORT

## Problem
The Select Widget Type dialog shows an ItemPath snake_case exception when CircleDisplay is selected or filtered, and the CircleDisplay preview is not available.

## Expected Behavior
Selecting CircleDisplay should create the preview model, publish its internal display runtime values, render the preview, and allow the widget type to be selected.

## Actual Behavior
CircleDisplay preview creation calls `FolderItemModel.SetHierarchy(...)`, which refreshes paths and publishes CircleDisplay runtime values. Runtime child items are created with PascalCase item names, so the strict `Amium.Items.ItemPath` validator rejects them.

## Error Messages / Logs
`System.ArgumentException: Item path segment 'SignalColor' must use lowercase snake_case and start with a letter.`

The screenshot stack points to:

- `HornetStudio.Editor.Models.FolderItemModel.EnsureCircleDisplayRuntimeSignals()` at `PageItemModel.cs:4202`
- `HornetStudio.Editor.Models.FolderItemModel.RefreshPathRecursive()` at `PageItemModel.cs:4046`
- `HornetStudio.Editor.Widgets.Common.WidgetSelectionDialogWindow.UpdatePreview()`

## Relevant Code
- `src/HornetStudio.Editor/Models/PageItemModel.cs`
  - CircleDisplay runtime child constants used PascalCase names such as `SignalColor`.
  - `GetDisplayRuntimeBasePath()` used a mixed-case `Displayruntime` path segment.
- `src/HornetStudio.Editor/Widgets/CircleDisplay/EditorCircleDisplayControl.axaml.cs`
  - The control used the same PascalCase runtime child item names when publishing live runtime values.

## Attempted Fixes
- Changed CircleDisplay runtime child item names to strict snake_case: `signal_color`, `signal_run`, `progress_bar`, `progress_state`, and `progress_bar_color` -> the preview/runtime item creation no longer throws.
- Changed the CircleDisplay runtime root to `studio.<folder_name>.display_runtime.<widget_name>` -> the internal runtime branch is now canonical snake_case.
- Kept display text/title values human-readable, so only technical item path segments changed.
- Added `CircleDisplayRuntimePathUsesSnakeCaseSegments` to `src/HornetStudio.Editor.Tests/Program.cs` -> verifies the runtime root and child items are snake_case and resolvable.
- Updated CircleDisplay widget documentation and short description with the runtime path names.

## Current Hypothesis (optional)
- The widget picker symptom and the CircleDisplay selection/preview symptom had the same root cause: preview creation executes the same runtime publication path as a real CircleDisplay instance, and that path still contained PascalCase technical item segments after the strict snake_case migration.
