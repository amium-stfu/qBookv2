# IMPLEMENTATION HANDOFF

## Goal
Make `RunFunction` the unified interaction action for all runnable entries from the central `FunctionRegistry`, including Python functions. `RunFunction` must support an optional `Argument` field and dispatch to the correct existing executor path based on the selected registry entry.

## Scope
- Included:
  - Let `RunFunction` pick runnable entries from `FunctionRegistry`, including Python entries.
  - Keep `InvokePythonFunction` working for backward compatibility.
  - Re-enable/show `Argument` for `RunFunction`.
  - Pass `Argument` to Python functions through the existing Python invocation payload behavior.
  - Keep declarative YAML functions runnable through the existing declarative executor.
  - Keep `Functions` as visualizer/editor only; no dependency on placing the widget.
  - Add a central dispatch boundary if needed so callers do not branch directly into Python/declarative execution.
  - Update docs and focused tests.
- Not included:
  - Removing `InvokePythonFunction`.
  - Migrating existing YAML/layout data from `InvokePythonFunction` to `RunFunction`.
  - Changing Python files, Python templates, Python bridge commands, or runtime startup.
  - Adding C# function support.
  - Implementing Workflows.
  - Adding typed argument schemas or return values.

## Starting Point
- `FunctionRegistry` already exists and is widget-independent.
- Declarative entries use stable references like `declarative:<name>`.
- Python entries are now shown in `Functions` from runtime metadata.
- `RunFunction` currently runs only declarative functions and rejects unsupported kinds.
- `InvokePythonFunction` currently invokes Python functions through `PythonClientRuntimeRegistry`.
- Interaction rule persistence already has `FunctionName` and `Argument`.
- Current UI logic hides/clears `Argument` for `RunFunction` in:
  - `ItemInteractionEditorRow.ShowsArgumentEditor`
  - `InteractionRulesEditorDialogWindow.ShowsArgumentEditor`
  - related refresh logic in `EditorDialogField`

## Tasks
1. Inspect current Python registry entry shape:
   - confirm `FunctionCatalogKind.Python`
   - confirm `FunctionCatalogSource.PythonApplication`
   - confirm `SourceIdentifier` contains the Python runtime target path
   - confirm `Reference` format is stable enough for persisted `RunFunction`
2. Decide the central dispatch boundary:
   - preferred: introduce a small `FunctionRunner` / `FunctionDispatcher` under `src/HornetStudio.Editor/Functions/`
   - acceptable: keep dispatch in `PageItemModel` only if extracting would create unnecessary churn
3. Update registry entry capabilities:
   - Python entries should expose `CanRun = true` when they have target path and function name metadata.
   - Invalid/incomplete Python entries should stay `CanRun = false`.
4. Update `RunFunction` option filtering:
   - query `FunctionRegistry`
   - include all entries with `CanRun = true`
   - no longer filter to declarative-only
   - persist the stable `Reference` in `FunctionName`
5. Re-enable `Argument` for `RunFunction`:
   - configured rows should show the `Argument` editor for `RunFunction`
   - the new-rule row should show the `Argument` editor for `RunFunction`
   - refresh logic must not clear `Argument` when action is `RunFunction`
6. Dispatch declarative entries:
   - preserve the existing declarative load/validation path
   - keep current `FunctionExecutor` behavior
   - ignore `Argument` for declarative functions for now, unless an existing execution context already has a safe place to carry it
7. Dispatch Python entries:
   - resolve target path and function name from the registry entry/reference
   - use the existing Python client lookup and invocation behavior from `InvokePythonFunction`
   - pass `Argument` through the existing JSON/plain-text payload helper
   - log start, success, failure, and exception paths with `RunFunction` context
8. Keep legacy `InvokePythonFunction` behavior unchanged:
   - existing target/function pickers still work
   - existing persisted rules still run
   - no automatic migration in this step
9. Add clear user-visible errors in English:
   - missing function reference
   - registry entry not found
   - entry not runnable
   - Python target/client not available
   - Python function failed
   - unsupported future kind
10. Update docs:
   - Button docs mention `RunFunction` can run registry functions, including Python when registered.
   - Functions docs clarify registry visibility and execution through `RunFunction`.
   - Help docs clarify `Argument`: passed to Python, currently not used by declarative functions.
11. Add focused tests where practical:
   - `RunFunction` keeps/persists `Argument`.
   - option filtering includes runnable Python entries.
   - declarative dispatch still works.
   - Python dispatch delegates to existing Python invocation path if test seams exist.
   - unsupported kinds fail clearly.

## Implementation Order
1. Inspect current Python registry entry properties and existing `InvokePythonFunction` execution helper.
2. Ensure Python registry entries are runnable only when they can be invoked.
3. Update `RunFunction` picker filtering to include all `CanRun` registry entries.
4. Re-enable and preserve `Argument` for `RunFunction` in both interaction rule editors.
5. Extract or centralize dispatch enough that `RunFunction` can route declarative and Python entries cleanly.
6. Implement Python dispatch by reusing the existing Python client invocation path.
7. Preserve legacy `InvokePythonFunction` without behavior changes.
8. Add/adjust tests.
9. Update documentation.
10. Build and manually verify.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation/error messages must be English.
- Do not introduce new dependencies.
- Do not change Python storage, templates, generated scripts, bridge commands, or runtime startup.
- Do not remove or break `InvokePythonFunction`.
- Do not require a placed `Functions` widget.
- Do not block the UI with `.Result` or `.Wait()`.
- Keep dispatch small and focused; avoid broad service architecture unless current code already has a matching pattern.

## Acceptance Criteria
- `RunFunction` picker lists runnable declarative and runnable Python registry entries.
- A user can select a Python registry entry in an Action Rule via `RunFunction`.
- `RunFunction` shows and persists `Argument`.
- For Python entries, `RunFunction` passes `Argument` through the same payload semantics as existing `InvokePythonFunction`.
- For declarative entries, existing function execution still works and does not fail because an argument is present.
- Existing `InvokePythonFunction` rules still load, edit, and execute as before.
- `RunFunction` still works without placing a `Functions` widget.
- Missing/unavailable Python clients produce clear English errors and log entries.
- Build and focused tests pass.

## Verification
- Build:
  - `dotnet build HornetStudio.sln --no-restore`
  - If a running app locks output, use the established project-level or separate `OutDir` fallback.
- Tests:
  - Run focused editor tests if available.
  - Add/run interaction rule serialization and registry option tests if feasible.
- Manual:
  - Register/start a Python function source as before.
  - Configure a Button Action Rule: `BodyLeftClick -> RunFunction`.
  - Select a Python registry reference.
  - Enter an `Argument`.
  - Click the Button and confirm the Python function receives the argument.
  - Configure a declarative function with `RunFunction` and confirm it still executes.
  - Confirm old `InvokePythonFunction` configuration still works.

## Out of Scope
- Removing or hiding `InvokePythonFunction`.
- Automatic migration of old action rules.
- Typed argument schemas.
- Function return values.
- C# function provider.
- Workflow provider/execution.
- Monitor conversion to unified `RunFunction`.

## Risks / Watchouts
- Python registry entries may exist only while a Python client is registered. `RunFunction` must fail clearly if the persisted reference points to a target that is no longer registered.
- Do not persist display names; persist stable registry references.
- If Python references encode target path and function name, parsing must tolerate delimiters in paths. Prefer using `SourceIdentifier` from the resolved registry entry instead of reparsing when possible.
- Keep `Argument` behavior simple: passed to Python, ignored by declarative functions for now.
- Avoid changing existing Python argument payload semantics; reuse the current helper.
- If a central `FunctionRunner` extraction becomes invasive, keep the implementation in `PageItemModel` and document the follow-up.
