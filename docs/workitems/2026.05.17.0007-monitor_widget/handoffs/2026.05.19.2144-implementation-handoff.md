# IMPLEMENTATION HANDOFF

## Goal
Extend MonitorWidget aggregate status items so active event IDs remain published per log level and the same aggregate items also expose the matching active event texts in `meta`. Enforce valid, unique `EventId` values per MonitorWidget and auto-assign the next free event ID for new rules.

## Scope
- Include MonitorWidget runtime aggregate publishing for `fatal_active`, `error_active`, `warning_active`, `info_active`, and `debug_active`.
- Include Monitor rule editor behavior for required, unique `EventId` values.
- Include focused tests and documentation updates for MonitorWidget runtime paths and EventId validation.
- Do not change Monitor rule action behavior, ProcessLog behavior, or public log level names.
- Do not introduce new dependencies.

## Starting Point
- `MonitorControl.BuildActiveEventIdAggregates(...)` currently returns only a dictionary of aggregate item names to comma-separated active `EventId` strings.
- `UpdateAggregateRuntime()` writes those values to `snapshot[aggregate.Key].Value` and labels each aggregate via `Properties["text"]`.
- `MonitorDefinition.EventId` is an `int` and currently defaults to `0`.
- `MonitorEditorDialogViewModel.TryBuildDefinition(...)` currently accepts blank `EventIdText` as `0` and only validates integer parsing.
- Existing event metadata fields already exist on `MonitorDefinition`: `EventId`, `EventText`, and `LogLevel`.
- Existing aggregate path documentation mentions only the aggregate item names, not their metadata payload.

## Tasks
1. Replace the string-only aggregate result with a small internal aggregate data shape that carries:
   - aggregate item name, for example `warning_active`
   - comma-separated active event IDs for the item value/read payload
   - JSON metadata containing active event IDs and event texts
2. Preserve the existing aggregate value format exactly:
   - `warning_active` value remains `0,123` or `123,124`
   - ordering remains by rule name, case-insensitive, matching current behavior
3. Add aggregate metadata on the same item:
   - `snapshot["warning_active"].Properties["meta"].Value`
   - use valid JSON, not a pseudo-set literal
   - preferred JSON shape:
     `{"events":[{"event_id":123,"text":"RangeError"},{"event_id":124,"text":"DI5 high"}]}`
   - for no active events, use `{"events":[]}`
4. Update or add tests for aggregate metadata:
   - active rules of the same `LogLevel` publish all matching event IDs in the value
   - metadata contains matching event IDs and event texts
   - inactive rules and other log levels are excluded
5. Add EventId auto-assignment for new Monitor rules:
   - when the dialog is opened for a new definition, assign the smallest free positive integer among existing rules of the same MonitorWidget
   - use `1` as the first candidate unless existing project conventions indicate another base
6. Enforce EventId on save:
   - reject blank or whitespace EventId
   - reject non-integer EventId
   - reject `EventId <= 0`
   - reject duplicate EventId within the same MonitorWidget
   - when editing an existing rule, ignore that rule's own current definition for duplicate detection
7. Add or update focused tests for EventId behavior:
   - new rule receives next free positive EventId
   - duplicate EventId is rejected
   - blank and zero EventId are rejected
8. Update Monitor documentation:
   - `src/HornetStudio/docs/widgets/Monitor.md`
   - `src/HornetStudio/docs/widgets/help/Monitor.help.md`
   - `src/HornetStudio/docs/widgets/descriptions/Monitor.md` only if the concise picker text mentions event metadata
   - `docs/data-flow-and-signals.md` and mirrored `src/HornetStudio/docs/data-flow-and-signals.md` if both are maintained in parallel

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/Monitor/MonitorControl.axaml.cs` -> build and publish aggregate event metadata alongside existing aggregate values.
- `src/HornetStudio.Editor/Widgets/Monitor/MonitorEditorDialogWindow.axaml.cs` -> auto-assign next free EventId and validate required/unique EventId values.
- `src/HornetStudio.Editor.Tests/Program.cs` -> add focused self-tests for aggregate metadata and EventId validation/auto-assignment.
- `src/HornetStudio/docs/widgets/Monitor.md` -> document aggregate `meta` payload and EventId uniqueness requirement.
- `src/HornetStudio/docs/widgets/help/Monitor.help.md` -> update end-user help for EventId and aggregate metadata.
- `docs/data-flow-and-signals.md` -> document aggregate value and metadata shape.
- `src/HornetStudio/docs/data-flow-and-signals.md` -> mirror the same runtime path documentation if the file remains duplicated.

## Implementation Order
1. Inspect the current `MonitorEditorDialogViewModel` constructor path to determine how new versus existing definitions are represented.
2. Add private helper methods in `MonitorEditorDialogWindow.axaml.cs` for:
   - collecting existing EventIds from `_ownerItem.MonitorDefinitions`
   - generating the next free positive EventId
   - validating uniqueness while excluding the current rule name when editing
3. Set `EventIdText` for new rules using the next free positive EventId.
4. Strengthen `TryBuildDefinition(...)` EventId validation before creating `MonitorDefinition`.
5. Refactor `BuildActiveEventIdAggregates(...)` to return aggregate records instead of raw strings, keeping the public method only if tests or other code require it.
6. Publish both aggregate value and `meta` property in `UpdateAggregateRuntime()`.
7. Add/update tests.
8. Update documentation.
9. Run focused tests/build.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and technical identifiers must remain English.
- Use existing project patterns and libraries.
- Do not introduce new dependencies for JSON generation; use existing BCL JSON APIs if needed.
- Keep the change scoped to MonitorWidget behavior.
- Public methods/classes require XML documentation if new public API is introduced; prefer private/internal helpers unless public API is necessary.
- Use named parameters where calls have unclear positional arguments or more than two parameters.
- User-visible validation messages must be English.

## Acceptance Criteria
- Existing aggregate values still expose comma-separated active EventIds per log level.
- Each aggregate item exposes `Properties["meta"].Value` as valid JSON with active event IDs and EventText values.
- Empty aggregate levels expose an empty value string and metadata with an empty `events` array.
- New Monitor rules automatically receive the next free positive EventId.
- Saving a rule with blank, non-integer, zero, negative, or duplicate EventId fails with a clear English validation message.
- Editing an existing rule without changing its EventId does not falsely report a duplicate.
- Documentation describes the aggregate value and metadata shape.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore has already completed.
- Run the focused editor self-test executable or the repository's established editor test command after build.
- If restore is required, run `dotnet restore HornetStudio.sln` first.
- Manually verify in the Monitor rule dialog that a new rule shows an automatically assigned EventId and duplicate IDs are rejected.

## Out of Scope
- Renaming `LogLevel` to `Severity`.
- Changing ProcessLog entry formatting.
- Changing action triggers or Monitor action serialization.
- Migrating old saved `EventId = 0` definitions automatically during project load.
- Adding global EventId uniqueness across different MonitorWidgets.

## Risks / Watchouts
- `meta` is a protected/default item property in other parts of the registry; this runtime write is internal and should use the existing snapshot/property pattern rather than user write APIs.
- Duplicate detection by rule name can be ambiguous if the user also renames a rule during edit. Prefer excluding the originally edited definition by captured original name or object state, not the newly typed name only.
- Use valid JSON. The example `{"RangeError","DI5 high"}` is not valid JSON and should not be implemented literally.
- Keep aggregate ordering stable to avoid unnecessary broker/topic churn.
- Existing projects may contain `EventId = 0`; avoid breaking project load, but enforce valid IDs when the user saves a rule.
