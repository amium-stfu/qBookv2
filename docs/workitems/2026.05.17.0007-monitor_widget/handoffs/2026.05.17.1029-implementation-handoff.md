# IMPLEMENTATION HANDOFF

## Goal
Refine the implemented `Monitor` widget so monitoring settings are owned by each monitor rule instead of the widget. Each rule should have its own evaluation interval, timeout, mode-specific condition editor, event metadata, logging target, and log level.

## Scope
Included:
- Remove global Monitor widget monitoring settings such as widget-level `RefreshRateMs` and default conditions.
- Move `RefreshRateMs`, `TimeoutMs`, `InhibitMs`, event metadata, logging target, and log level to each monitor rule.
- Keep `TimeoutMs` available for every monitor rule independent of mode.
- Add rule mode selection: `Default` or `Custom`.
- In `Default` mode, show and evaluate direct limit fields such as `LowerLimit` and `UpperLimit`.
- In `Custom` mode, hide direct limit fields and use a CustomSignals-computed-style boolean formula editor.
- Add `LogLevel` per monitor rule with values `Debug`, `Info`, `Warning`, `Error`, and `Fatal`.
- Preserve existing Monitor runtime path shape and Signal timeout removal from the previous implementation.
- Update persistence, YAML, documentation, and tests for the new per-rule model.

Explicitly not included:
- Signal widget blinking or visual monitor binding.
- Global alert manager, acknowledgement, history, repeat interval, severity model, or dashboard.
- New external dependencies.
- Broad redesign of CustomSignals or ProcessLog.
- Release/version/tag work.

## Starting Point
- The Monitor widget has already been implemented from the previous plan.
- The previous plan allowed widget-level `RefreshRateMs` and default conditions.
- The desired structure is now rule-local: the widget is a container only and should not own monitoring behavior settings.
- CustomSignals already has a `Computed` editor pattern with variables and formula controls that should be used as the UI reference for `Custom` mode.
- ProcessLog integration already exists, and Monitor logging should continue to write only on inactive-to-active transitions to avoid log spam.

Assumptions:
- The persisted widget `Type` remains `Monitor`.
- Existing Monitor runtime paths remain under `studio.<page>.monitor.<monitor_widget>.<rule>`.
- Each monitor rule has its own effective evaluation interval.
- A dedicated timer per rule is acceptable for clarity, but the behavioral contract is "per-rule evaluation interval"; implementation may internally batch equal intervals later if needed.
- `TimeoutMs` is always present in the rule editor, but empty/null means no timeout check.
- `RefreshRateMs` belongs to each rule and controls how often that rule is evaluated.
- `InhibitMs` belongs to each rule and delays activation for any active condition in that rule.
- `LogLevel` default should be `Warning`.
- `LogLevel` is only operationally relevant when `TargetLog` is configured.

7. Ensure `Default` mode evaluates lower/upper limits only when configured.
8. Ensure `Custom` mode evaluates only the boolean formula condition and fails closed for invalid or non-boolean results.
9. Update logging so `LogLevel`, `EventId`, and `EventText` are used when writing to `TargetLog`.
10. Update tests, YAML examples, and Monitor documentation.

## Technical Constraints
- Chat and implementation summaries must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation and error messages must be English.
- Do not add dependencies.
- Follow existing CustomSignals/EnhancedSignals widget patterns.
- Keep runtime path segments lowercase snake_case.
- Avoid broad formula-engine refactoring unless required to share the existing computed editor safely.
- Logging must happen only on inactive-to-active transition unless a repeat interval is explicitly planned later.
- Keep Signal widget behavior out of scope.

## Acceptance Criteria
- Monitor widget no longer exposes global monitoring settings.
- Each Monitor rule has its own `RefreshRateMs`.
- Each Monitor rule has its own `TimeoutMs`, visible in both `Default` and `Custom` modes.
- Each Monitor rule has its own `InhibitMs`.
- Rule mode can be switched between `Default` and `Custom`.
- `Default` mode shows lower/upper limit fields and evaluates configured limits.
- `Custom` mode hides lower/upper limit fields and evaluates a boolean formula with variables.
- Invalid or non-boolean custom formulas do not crash evaluation and do not produce a false active alarm.
- Each rule can configure `EventId`, `EventText`, `TargetLog`, and `LogLevel`.
- `LogLevel` supports `Debug`, `Info`, `Warning`, `Error`, and `Fatal`, defaulting to `Warning`.
- Monitor logging uses the configured log level and writes only when a rule transitions from inactive to active.
- Existing Monitor layouts with old widget-level settings still load safely.
- YAML and documentation describe the rule-local settings model.

## Verification

- `dotnet build HornetStudio.sln --no-restore`
- Manual verification:
  - Add a Monitor widget and confirm there are no global monitoring settings.
  - Add two rules with different `RefreshRateMs` values and confirm they evaluate independently.
  - Configure a `Default` rule with lower/upper limits and verify active/inactive transitions.
  - Switch a rule to `Custom` and confirm lower/upper fields are hidden while variables/formula are shown.
  - Configure a custom formula that returns bool and verify active/inactive transitions.
  - Configure `TargetLog` and different `LogLevel` values and verify log entries use the selected level.
  - Save and reload the layout and confirm all per-rule settings persist.

## Out of Scope
- Signal widget blinking or visual state consumption.
- Alarm acknowledgement, history, severity beyond log level, dashboards, and global alert management.
- Repeating log entries while a condition remains active.
- Cross-widget batching optimization for rule timers.
- Release versioning.

## Risks / Watchouts
- Multiple per-rule timers are simple but can become expensive with many rules; keep the implementation clear and consider future batching only if real pressure appears.
- Loading layouts from the first Monitor implementation may require fallback migration from widget-level settings to rule-level defaults.
- `TimeoutMs`, `RefreshRateMs`, and `InhibitMs` have different meanings and should not be conflated in UI text or code.
- Custom formulas must be constrained to boolean monitor conditions.
- Avoid log spam by preserving transition-based logging.
- Mode switching should not accidentally delete hidden values unless the user explicitly clears or saves changed fields by design.
