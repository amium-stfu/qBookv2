# IMPLEMENTATION HANDOFF

## Goal
Remove the Signal widget timeout monitoring again and plan a dedicated `Monitor` widget for item and state monitoring. The Monitor widget should follow the existing CustomSignals/EnhancedSignals pattern with a header `Add` action, persisted rule definitions, runtime publishing, optional ProcessLog integration, and optional computed boolean conditions.

## Scope
Included:
- Remove Signal widget `TimeoutMs` configuration, runtime timer logic, and `signal_runtime` publishing.
- Remove Signal widget documentation that describes timeout monitoring.
- Add a new persisted widget type `Monitor`.
- Add a Monitor widget UI with an `Add` action in the header and a list of monitor rules in the body.
- Add monitor settings for `RefreshRateMs` and default condition values.
- Add monitor rule definitions with source path, timeout, lower limit, upper limit, inhibit, custom boolean expression, event id, event text, and optional target ProcessLog.
- Publish monitor state items into the runtime registry using canonical lowercase snake_case paths.
- Add or update focused tests for persistence, runtime paths, condition evaluation, and Signal timeout removal.
- Update widget documentation, descriptions, and help files for Signal and Monitor.

Explicitly not included:
- Signal widget blinking or visual alarm state binding.
- A global alert manager, acknowledgement workflow, alarm history, severity model, or dashboard.
- External notification integrations.
- Release/version/tag work.
- New dependencies.
- Broad refactoring of CustomSignals, EnhancedSignals, or ProcessLog.

## Starting Point
- Signal currently has `TimeoutMs` as a flat property and publishes `studio.<page>.signal_runtime.<widget>.status.timeout`.
- Timeout monitoring is implemented inside `PageItemModel`.
- Signal docs and descriptions mention timeout monitoring.
- CustomSignals and EnhancedSignals already provide the closest UI and persistence patterns for editable lists of rule-like definitions.
- ProcessLog selection already exists through `TargetLog` and `GetProcessLogTargetOptions()`.
- CustomSignal computed evaluation exists in `CustomSignalFormulaEngine` and can be reused or generalized only if the change remains local and low-risk.

Assumptions:
- The new persisted widget `Type` should be `Monitor`.
- The runtime root should be `studio.<page>.monitor.<monitor_widget>.<rule>`.
- `RefreshRateMs` belongs to the Monitor widget settings.
- `TimeoutMs`, `LowerLimit`, `UpperLimit`, and `InhibitMs` can be provided as defaults and overridden per monitor rule.
- Empty numeric condition values mean "not configured".
- Custom expressions must evaluate to a boolean; non-boolean results are invalid and should fail closed.
- `EventId` is an integer used for simple error or event codes.
- `EventText` is the message used for published state and optional log entries.
- `TargetLog` is optional and should use the existing ProcessLog picker/options.

1. Remove Signal timeout monitoring from model, persistence, YAML, editor settings, tests, and documentation.

9. Add focused tests for removed Signal timeout behavior and new Monitor behavior.

## Technical Constraints
- Chat and implementation summaries must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible error and validation messages must be English.
- Do not add dependencies.
- Follow existing widget patterns and avoid unnecessary abstraction layers.
- Runtime path segments must be lowercase snake_case.
- Do not rename existing persisted widget types except adding the new `Monitor` type.
- Keep Signal widget flat after removing `TimeoutMs`; do not add blinking, alarm visuals, or monitor binding in this work.
- Use existing HostRegistry and TargetPathHelper patterns.
- ProcessLog integration must use existing ProcessLog runtime and target selection patterns.

## Acceptance Criteria
- Signal widget no longer exposes or persists `TimeoutMs`.
- Signal widget no longer publishes `studio.<page>.signal_runtime...` timeout items.
- Existing layouts that contain old `TimeoutMs` do not crash on load; the value is ignored or safely dropped on save.
- Widget picker can add a new `Monitor` widget.
- Monitor widget displays configured rules and supports adding/editing rules.
- Monitor settings include `RefreshRateMs` and default condition values.
- Monitor rules can evaluate timeout, lower limit, upper limit, and inhibit.
- Custom monitor expressions can be configured and must evaluate to bool.
- Monitor publishes runtime state under `studio.<page>.monitor.<monitor_widget>.<rule>`.
- Monitor can optionally write `EventId` and `EventText` to a selected ProcessLog when a rule becomes active.
- Non-numeric source values do not crash lower/upper limit evaluation.
- Runtime paths and rule names are valid lowercase snake_case identities.
- Widget documentation describes Monitor behavior and no longer claims Signal timeout monitoring.

## Out of Scope
- Signal widget blinking or styling based on monitor state.
- Alarm acknowledgement, reset, history, grouping, or severity.
- Central alert dashboard.
- Cross-page global monitor service.
- Release version updates.
- External notifications.

## Risks / Watchouts
- Removing `TimeoutMs` from Signal must preserve compatibility with older layout files that still contain it.
- Custom expression support can easily become too broad; keep it constrained to boolean results.
- Logging must avoid repeated log spam while a rule remains active; write on inactive-to-active transition only unless a repeat interval is explicitly added later.
- Inhibit timing must be based on condition activation time, not the last source update time.
- Timeout evaluation needs a reliable source update timestamp; fail closed and document behavior when a source cannot provide one.
- Rule names must be stable, unique within one Monitor widget, and valid for runtime paths.
- Reusing CustomSignal formula code may require careful extraction to avoid breaking CustomSignals.
