# IMPLEMENTATION HANDOFF

## Goal
Replace the permanent `Edit` and `Delete` buttons in each compact MonitorWidget rule row with one touch-friendly row action menu button. The menu opens a dropdown/context menu containing `Edit` and `Delete`.

## Scope
- Change only MonitorWidget row actions in the compact body layout.
- Keep `Add` in the header unchanged.
- Keep existing edit/delete handlers and behavior.
- Keep compact one-line rule rows with LED, EventId, and EventText.
- Do not introduce hover-only behavior as the primary access path.
- Do not change MonitorDefinition, runtime publishing, rule evaluation, actions, or persistence.

## Starting Point
- The compact MonitorWidget body has already been implemented.
- Each row currently shows permanent `Edit` and `Delete` buttons at the right side.
- Existing handlers are available in `MonitorControl.axaml.cs`:
  - `OnEditRuleClicked`
  - `OnDeleteRuleClicked`
- Existing project patterns use `Button.ContextMenu` and explicit `ContextMenu.Open()` for dropdown-like button menus, for example in RealtimeChart/Table/CircleDisplay widgets.
- No dedicated ellipsis icon pattern was found. `cog.svg` exists, but it semantically suggests settings rather than row actions.

## Tasks
1. Replace the two permanent row buttons with a single compact action button at the row end.
2. Prefer a simple visible text glyph such as `...` or `⋯` only if the file already accepts Unicode.
   - Because repository edits default to ASCII, prefer `...` unless an existing icon asset is available and clearly appropriate.
3. Attach a `ContextMenu` to the action button with:
   - `MenuItem Header="Edit"`
   - `MenuItem Header="Delete"`
4. Ensure the clicked menu items can identify the correct `MonitorRuleRow`.
   - Use `CommandParameter`, `Tag`, or `DataContext` consistently with existing Avalonia patterns.
   - If existing `OnEditRuleClicked`/`OnDeleteRuleClicked` expect a `Button.CommandParameter`, either adapt them to also handle `MenuItem` context or add small forwarding handlers.
5. Add an action button click handler if needed to open the context menu on normal left-click/touch:
   - mirror existing pattern from `RealtimeChartControl.axaml.cs` where a button opens its `ContextMenu`.
6. Keep the action button visible for touch usability.
   - Do not hide it behind hover.
   - Keep it narrow so it does not dominate the row.
7. Add tooltip text if existing project conventions support it:
   - `Rule actions`
8. Ensure the row still fits small widget widths:
   - LED and EventId columns stay fixed/minimal.
   - EventText column gets remaining width and trims/wraps according to the current compact row behavior.
   - action button remains fixed width.
9. Update Monitor widget documentation/help if the body interaction is described.

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/Monitor/MonitorControl.axaml` -> replace row `Edit`/`Delete` buttons with one menu button and context menu.
- `src/HornetStudio.Editor/Widgets/Monitor/MonitorControl.axaml.cs` -> add/open row action menu handler and adapt edit/delete handlers if needed.
- `src/HornetStudio/docs/widgets/Monitor.md` -> update row action description if present.
- `src/HornetStudio/docs/widgets/help/Monitor.help.md` -> update help text if it describes row buttons.

## Implementation Order
1. Inspect current compact row XAML to confirm how the implemented row passes `MonitorRuleRow` to `Edit` and `Delete`.
2. Update XAML to use one compact row action button with `Button.ContextMenu`.
3. Add `OnRuleActionsClicked` or similarly named handler that opens the attached context menu on left-click/touch.
4. Make `Edit` and `Delete` menu item handlers resolve the row reliably.
5. Remove the old permanent row buttons from XAML.
6. Validate layout at narrow widths.
7. Update docs only where they mention row-level buttons.
8. Run focused build/tests.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and technical identifiers must remain English.
- Use existing Avalonia `ContextMenu`/`MenuItem` patterns.
- Do not introduce new dependencies.
- Avoid new icon assets unless an appropriate existing asset is already available.
- Keep row actions touch-accessible; do not rely only on mouse hover.
- Keep UI text English.

## Acceptance Criteria
- Each Monitor rule row shows one compact action menu button instead of permanent `Edit` and `Delete` buttons.
- Opening the menu by mouse click or touch shows `Edit` and `Delete`.
- `Edit` opens the existing rule editor for the correct row.
- `Delete` deletes the correct row using existing behavior.
- The row remains compact and does not visually prioritize actions over EventId/EventText.
- No Monitor runtime behavior, serialization, actions, or aggregate publishing changes.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore has already completed.
- Manually verify:
  - action menu opens on click/touch-equivalent interaction
  - `Edit` edits the selected row
  - `Delete` deletes the selected row
  - compact row layout remains readable at the current widget size from the screenshot

## Out of Scope
- Hover-only actions.
- Keyboard shortcut design.
- Delete confirmation changes.
- New icons or a global row-action component.
- Any EventId or aggregate metadata behavior.

## Risks / Watchouts
- `MenuItem.DataContext` may not automatically be the row when nested inside `Button.ContextMenu`; explicitly pass the row via `Tag` or `CommandParameter` if needed.
- `ContextMenu` may need `PlacementTarget` set before opening, following the RealtimeChart pattern.
- If Unicode ellipsis is used, ensure the file already contains Unicode or there is a clear reason. ASCII `...` is acceptable and safer.
