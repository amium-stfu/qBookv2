# IMPLEMENTATION HANDOFF

## Goal
Simplify the MonitorWidget body from multi-line rule cards to a compact event list. Each rule should render as one row with a log-level colored LED, `EventId`, and `EventText`, sorted from highest to lowest severity: `Fatal`, `Error`, `Warning`, `Info`, `Debug`.

## Scope
- Change only the MonitorWidget body presentation.
- Keep existing rule editing, deletion, runtime evaluation, serialization, actions, and aggregate publishing behavior unchanged.
- Keep the existing `Add`, `Edit`, and `Delete` capabilities available in editor mode.
- Reuse existing theme/log severity colors so the LED colors match LogWidget log entry colors.
- Update Monitor widget documentation/help to describe the compact severity-sorted rule body.

## Starting Point
- `src/HornetStudio.Editor/Widgets/Monitor/MonitorControl.axaml` currently renders each rule as a bordered card with:
  - rule name
  - `Edit` button
  - `Delete` button
  - source/mode/refresh text
  - status text such as `Active: Formula matched`
- `src/HornetStudio.Editor/Widgets/Monitor/MonitorControl.axaml.cs` exposes `Rules` as an observable list of `MonitorRuleRow`.
- `MonitorRuleRow` already has access to:
  - `Definition.EventId`
  - `Definition.EventText`
  - `Definition.LogLevel`
  - `IsActive`
  - owner item/theme properties
- `EditorLogControl` maps log levels to theme colors via `ViewModel.ActiveTheme.LogDebugForeground`, `LogInfoForeground`, `LogWarningForeground`, `LogErrorForeground`, and `LogFatalForeground`.

## Tasks
1. Add display properties to `MonitorRuleRow` for the compact row:
   - `EventIdText`
   - `EventDisplayText`
   - `SeveritySortOrder`
   - `SeverityForegroundBrush` or `LedBrush`
   - `InactiveLedBrush`
   - `EffectiveLedBrush`
2. Use `EventText` as the primary displayed text.
   - Fallback to `Name` when `EventText` is empty.
   - Keep UI text English where new text is needed.
3. Sort displayed rows by severity:
   - `Fatal` first
   - `Error`
   - `Warning`
   - `Info`
   - `Debug` last
   - within the same severity, sort by `EventId`, then by `Name` for stable ordering
4. Decide whether to sort the backing `Rules` collection or expose a separate sorted view.
   - Prefer a separate sorted view if it avoids changing persistence/order semantics.
   - Name it clearly, for example `DisplayRules`.
5. Replace the card body template in `MonitorControl.axaml` with one-line rows:
   - left LED/status dot
   - EventId column
   - EventText column
   - optional compact edit/delete controls only where the current editor interaction requires them
6. Preserve editing capability:
   - `Add` remains in the header.
   - `Edit` and `Delete` should remain reachable.
   - Preferred UI: compact icon/text buttons at the row end, or row click/double-click for edit plus a small delete action if that pattern already exists elsewhere.
7. Keep inactive rules visible.
   - Active rows use log-level LED color.
   - Inactive rows use a muted/neutral LED color.
   - Text may remain normal foreground or be slightly muted, but must stay readable.
8. Remove Source/Mode/Refresh/Status text from the normal body row.
   - Do not delete the underlying properties if they are used elsewhere.
   - Optionally keep detailed text as a tooltip if Avalonia binding is straightforward and does not clutter implementation.
9. Update row refresh notifications so changes to active state, theme, or definition update LED color and displayed text.
10. Update docs:
   - `src/HornetStudio/docs/widgets/Monitor.md`
   - `src/HornetStudio/docs/widgets/help/Monitor.help.md`
   - `src/HornetStudio/docs/widgets/descriptions/Monitor.md` only if the short description mentions the body layout

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/Monitor/MonitorControl.axaml` -> replace multi-line rule cards with compact one-line event rows.
- `src/HornetStudio.Editor/Widgets/Monitor/MonitorControl.axaml.cs` -> add display/sort properties and expose severity-sorted row collection or view.
- `src/HornetStudio/docs/widgets/Monitor.md` -> document compact row layout and severity sort order.
- `src/HornetStudio/docs/widgets/help/Monitor.help.md` -> update help text for the simplified body.
- `src/HornetStudio/docs/widgets/descriptions/Monitor.md` -> update only if needed.

## Implementation Order
1. Inspect how `Rules` are added, removed, and refreshed in `MonitorControl.axaml.cs`.
2. Add a non-persisted sorted display collection/view for rules.
3. Update add/edit/delete paths to refresh the display ordering after rule list changes.
4. Add LED/log-level color properties to `MonitorRuleRow`, reusing the same theme colors as `EditorLogControl`.
5. Update `RaiseThemeChanged()` and active-state notifications to raise new display property changes.
6. Replace XAML body template with compact rows.
7. Verify row text fits in small widget sizes using trimming/wrapping rules:
   - EventId should not wrap.
   - EventText may trim with ellipsis or wrap to a second visual line only if unavoidable; the target requirement is one row.
8. Update documentation.
9. Run focused build/tests.

## Technical Constraints
- Chat responses must be German; code, comments, XML documentation, UI text, file names, and technical identifiers must remain English.
- Do not add dependencies.
- Reuse existing theme values and widget patterns.
- Keep source/mode/status data available for runtime publishing and dialogs; this is only a visual body simplification.
- Avoid changing saved MonitorDefinition order unless explicitly required.
- Do not introduce a broad UI framework refactor.

## Acceptance Criteria
- MonitorWidget body renders each configured rule as a compact row containing:
  - LED/status dot
  - EventId
  - EventText or rule-name fallback
- Rows are sorted top-down by log level severity:
  - `Fatal`, `Error`, `Warning`, `Info`, `Debug`
- Within the same severity, rows have stable deterministic order.
- Active rows show the LED in the same color family used by LogWidget entries for that log level.
- Inactive rows show a muted/neutral LED.
- Source, mode, refresh rate, and active reason are no longer shown as normal row text.
- Add/edit/delete remain usable in editor mode.
- Existing Monitor evaluation, runtime publishing, actions, and serialization behavior remain unchanged.
- Monitor documentation reflects the new body layout.

## Verification
- Run `dotnet build HornetStudio.sln --no-restore` after restore has already completed.
- Run focused editor tests if available after the build.
- Manually open a MonitorWidget with mixed `Fatal`, `Error`, `Warning`, `Info`, and `Debug` rules and verify row order.
- Manually toggle rule active states and verify LED colors update without changing row order.
- Verify small widget sizes do not produce overlapping text or unusable row actions.

## Out of Scope
- EventId validation or automatic EventId assignment. That is covered by a separate handoff.
- Aggregate `meta` publishing. That is covered by a separate handoff.
- Changing rule editor dialog layout.
- Changing runtime item shape.
- Adding filters, grouping headers, badges, counters, or additional body columns.

## Risks / Watchouts
- Sorting the actual `Rules` collection may accidentally change persisted order or user expectations. Prefer a display-only sorted view.
- LogWidget uses `Information` while Monitor uses `Info`; map `MonitorLogLevel.Info` to the theme's info color explicitly.
- Permanent row-end `Edit`/`Delete` buttons can consume too much width. Keep them compact and ensure `EventText` does not overlap.
- If row click is repurposed for edit, ensure it does not conflict with normal editor selection/drag interactions.
- Theme changes must update LED brushes, not only text foregrounds.
