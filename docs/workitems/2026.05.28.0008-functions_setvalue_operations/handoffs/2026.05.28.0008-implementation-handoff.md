# IMPLEMENTATION HANDOFF

## Goal
Bring Functions `SetValue` steps to the same structured SetValue behavior as Interaction Rules, using the existing `SetValueOperationCodec` and the same typed operation semantics.

## Scope
Included:
- Update the Functions editor UI so `SetValue` rows use typed inline operations equivalent to Interaction Rules.
- Keep existing YAML `value` and `valueFrom` definitions loadable.
- Persist newly edited Functions `SetValue` steps through the structured `value` payload where appropriate.
- Execute structured Functions `SetValue` operations at runtime with the same semantics as Interaction Rules.
- Update relevant widget documentation and focused tests.

Not included:
- New SetValue operation kinds.
- New dependencies.
- Release/version changes.
- Broad workflow editor refactoring outside the SetValue path.

## Starting Point
- Interaction Rules already use `SetValueOperationCodec`, `SetValueInlineOperationOption`, `SetValueTargetKind`, source compatibility validation, summaries, and structured persisted arguments with the `sv1:` prefix.
- Functions currently use `FunctionSetValueStepDefinition.Target`, `Value`, and `ValueFrom`.
- Functions runtime currently resolves `ValueFrom` manually in `PageItemModel.ExecuteRunFunctionSetValueAsync` and writes the resulting value directly.
- Workflow YAML currently supports `type: SetValue`, `target`, `value`, and `valueFrom`.
- Existing YAML files must remain compatible.

## Tasks
1. Extend the Functions SetValue editor row state to mirror the Interaction Rules inline SetValue state.
2. Update the Functions editor XAML row template so SetValue rows show target selection plus operation picker, literal/separator/source editors, and validation text.
3. Add Functions SetValue metadata refresh logic to resolve target kind, compatible source options, operation summary, and validation.
4. Convert legacy `valueFrom` rows into structured `SetFromItem` editor state while preserving load compatibility.
5. Save edited Functions SetValue rows using `FunctionSetValueStepDefinition.Value` with the serialized structured SetValue operation and avoid writing `valueFrom` for newly edited structured rows.
6. Update runtime execution so Functions structured SetValue operations use the same operation resolution as Interaction Rules.
7. Add or update tests for codec/editor/runtime roundtrips and structured operation execution.
8. Update Functions/Workflow widget documentation.

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowStepEditorRow.cs` -> Add SetValue inline editor state, source options, target kind, summary, validation message, and conversion between legacy fields and structured `SetValueOperation`.
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowEditorDialogWindow.axaml` -> Replace old SetValue value/source UI with typed inline SetValue controls matching Interaction Rules layout patterns.
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowEditorDialogWindow.axaml.cs` -> Refresh SetValue metadata for rows, browse compatible SetValue sources, validate rows before save, and pass target/source data into rows.
- `src/HornetStudio.Editor/Widgets/Workflow/WorkflowDefinitionCodec.cs` -> Preserve loading of `valueFrom`; serialize structured `value` for edited rows; keep legacy literal `value` valid.
- `src/HornetStudio.Editor/Models/WorkflowDefinition.cs` -> Update XML documentation for `FunctionSetValueStepDefinition` to describe structured `Value` and legacy `ValueFrom`.
- `src/HornetStudio.Editor/Models/PageItemModel.cs` -> Share or reuse the existing SetValue operation resolution path for Functions and Interaction Rules.
- `src/HornetStudio.Editor.Tests/Program.cs` -> Add focused tests for structured Functions SetValue editor serialization, legacy `valueFrom` compatibility, and runtime execution.
- `src/HornetStudio/docs/widgets/WorkflowWidget.md` -> Document structured SetValue behavior and legacy compatibility.
- `src/HornetStudio/docs/widgets/help/WorkflowWidget.help.md` -> Update detailed help for Functions SetValue operations.
- `src/HornetStudio/docs/widgets/Functions.md` and `src/HornetStudio/docs/widgets/help/Functions.help.md` -> Update only if these pages describe SetValue step editing or execution behavior.

## Implementation Order
1. Add row-level SetValue operation state and helper methods in `WorkflowStepEditorRow.cs`.
2. Update `FunctionEditorDefinitionConverter` conversion so legacy `valueFrom` maps to `SetFromItem` and structured `value` remains intact.
3. Add metadata refresh and compatible source selection in `WorkflowEditorDialogWindow.axaml.cs`.
4. Update `WorkflowEditorDialogWindow.axaml` controls.
5. Refactor `PageItemModel` runtime SetValue execution so declarative Functions call the same structured operation logic as Interaction Rules.
6. Update YAML serialization/deserialization only as needed to preserve backward compatibility and new structured persistence.
7. Add tests.
8. Update docs.
9. Run verification.

## Technical Constraints
- Use the existing `SetValueOperationCodec`; do not introduce a second codec or new dependency.
- Keep UI text, code comments, XML documentation, and technical identifiers in English.
- Keep chat communication German.
- Preserve existing workflow YAML compatibility.
- Use existing Avalonia patterns from `InteractionRulesEditorDialogWindow` and `ItemInteractionEditorRow`.
- Avoid broad refactoring outside SetValue.
- Public classes and public members touched by the change need accurate XML documentation.
- User-visible validation and error messages must be English.

## Acceptance Criteria
- Existing Functions YAML with `value` still loads and executes.
- Existing Functions YAML with `valueFrom` still loads and executes.
- Newly edited SetValue rows can configure numeric increment/decrement, string append, boolean true/false, literal set, and set-from-item where target type allows it.
- SetValue source picker offers compatible source paths for the selected target.
- Invalid structured operations are blocked before saving or report a clear validation message.
- Runtime Functions SetValue behavior matches Interaction Rules for supported structured operations.
- Documentation reflects the new behavior and legacy compatibility.

## Verification
- `dotnet build HornetStudio.sln --no-restore`
- Run focused editor tests through the repository's existing test command or direct executable pattern for the relevant test project.
- If restore is needed first, use `dotnet restore HornetStudio.sln`.
- Manually inspect at least one Function editor SetValue row path for layout consistency if UI can be launched locally.

## Out of Scope
- Changing Monitor SetValue behavior.
- Changing Interaction Rules behavior except for extracting reusable private helpers in `PageItemModel`.
- Changing Python function invocation.
- Adding new UI themes or icons.
- Release tagging or version updates.

## Risks / Watchouts
- `valueFrom` currently takes precedence over `value`; conversion must not accidentally invert behavior for existing YAML.
- Structured `sv1:` payloads stored inside YAML scalar values must serialize safely and parse after reload.
- Numeric operations depend on current target value being readable and numeric; error messages must make failed preconditions clear.
- Avoid recursive property-change loops when syncing inline editor state back into the row's persisted `Value`.
- Nested workflow rows in `IfThenElse` and `While` must receive the same SetValue metadata refresh behavior as top-level rows.
