# IMPLEMENTATION HANDOFF

## Goal
Restore selectable Signal widget variants in the `Select Widget Type` dialog. The picker should expose practical Signal presets such as numeric, boolean, bitmask, text, and hex while keeping all created widgets as the existing `ControlKind.Signal` runtime widget.

## Scope
- Add picker entries for Signal variants.
- Keep the persisted widget `Type` value as `Signal`.
- Apply variant-specific defaults only during widget creation and preview creation.
- Keep list insertion support aligned with the canvas picker where practical.
- Update concise widget picker documentation only if the user-facing picker behavior or Signal description changes.

Do not add new runtime widget kinds, new dependencies, or broad widget architecture changes.

## Starting Point
- The dialog currently shows only one generic `Signal` entry.
- `CreateSignalPreviewHost` already demonstrates numeric, boolean, bitmask, text, and hex Signal modes, but these are only preview children and not selectable picker entries.
- `WidgetSelectionItem` currently identifies selections only by `ControlKind`, so multiple entries with the same `ControlKind.Signal` need an additional preset identity to avoid selection preservation and creation ambiguity.
- Signal defaults are currently applied through `MainWindowViewModel.CreateItem(...)` and preview presets in `FolderEditorControl.axaml.cs`.

## Tasks
1. Introduce a lightweight Signal picker preset concept that can represent the five selectable Signal variants without adding new `ControlKind` values.
2. Extend widget selection items so multiple entries can share `ControlKind.Signal` while preserving selection by a stable item identity instead of only by kind.
3. Add Signal variant entries to the canvas picker.
4. Add matching Signal variant entries to the list picker if list insertion is expected to offer the same Signal choices.
5. Apply the selected preset after creating a new `ControlKind.Signal` item so the initial caption, value example, unit, and `TargetPropertyFormat` match the selected variant.
6. Update preview creation so each Signal variant previews itself instead of relying only on the combined Signal variants preview.
7. Add focused tests or extend existing editor self-tests if there is suitable coverage for picker items or preset application.
8. Run focused build/test verification for the editor scope, then run broader solution build if practical.

## File-Level Changes
- `src/HornetStudio.Editor/Widgets/Common/WidgetSelectionDialogWindow.axaml.cs` -> add a stable selection identity or preset field to `WidgetSelectionItem`; update filter/selection preservation to handle duplicate `ControlKind` values.
- `src/HornetStudio.Editor/Widgets/FolderEditor/FolderEditorControl.axaml.cs` -> define Signal picker variants, add entries to picker lists, generate variant-specific previews, and pass the chosen preset into creation flow.
- `src/HornetStudio.Editor/ViewModels/MainWindowViewModel.cs` -> add or reuse a narrow method for applying Signal preset defaults after item creation if this cannot stay localized in `FolderEditorControl`.
- `src/HornetStudio.Editor.Tests/Program.cs` -> add focused checks if existing test structure can validate preset defaults without requiring UI automation.
- `src/HornetStudio/docs/widgets/descriptions/Signal.md` -> update only if the picker description should explicitly mention selectable presets.
- `src/HornetStudio/docs/widgets/help/Signal.help.md` -> update only if the user-facing behavior documentation changes beyond picker entry names.

## Implementation Order
1. Add a small internal model for picker presets, preferably local to the picker/folder editor area unless reuse in tests requires wider visibility.
2. Update `WidgetSelectionItem` so it has a unique selection key independent of `ControlKind`.
3. Replace the single generic Signal picker entry with concrete Signal variant entries or keep the generic entry plus variants if the generic default remains useful.
4. Ensure `RefreshWidgetList` preserves the selected item by the new key, not by `Kind` alone.
5. Route the selected preset through the canvas and list insertion flows.
6. Apply variant defaults after creating the Signal item.
7. Update previews and descriptions.
8. Add focused tests where feasible.
9. Build and verify the picker manually.

## Technical Constraints
- All Signal variants must still create `ControlKind.Signal`.
- Persisted YAML/layout output must still use `Type: Signal`.
- Do not introduce new dependencies.
- Keep changes localized to picker and Signal defaulting behavior.
- Follow existing Avalonia and view model patterns.
- Chat responses are German; code, documentation text, UI text, and technical identifiers are English.
- User-visible error or validation messages must be English.

## Acceptance Criteria
- The `Select Widget Type` dialog shows selectable Signal variants for numeric, boolean, bitmask, text, and hex.
- Selecting any Signal variant creates a normal Signal widget.
- Created Signal widgets receive the expected default `TargetPropertyFormat`.
- Existing layouts with `Type: Signal` continue to load unchanged.
- The picker preview for each Signal variant reflects the selected format.
- Filtering can find the variants by display name and summary.
- Selecting/filtering does not collapse all Signal variants back to the first `ControlKind.Signal` entry.

## Verification
- Build: `dotnet build HornetStudio.sln --no-restore`
- Focused test: run `src/HornetStudio.Editor.Tests` according to the existing project pattern if tests are added there.
- Manual UI check: open the editor, open `Select Widget Type`, confirm Signal variants are listed, select each variant, and verify the inserted widget defaults.
- Manual persistence check: save a layout containing at least one variant and confirm the persisted type remains `Signal`.

## Out of Scope
- New runtime widget kinds such as `SignalNumeric` or `SignalHex`.
- Changes to Signal rendering behavior beyond initial default configuration.
- Changes to Signal runtime item binding, write routing, monitoring, or logging.
- Broad refactoring of widget catalog, YAML loaders, or editor dialog infrastructure.

## Risks / Watchouts
- Preserving selection by `ControlKind` currently cannot distinguish multiple Signal entries; this must be fixed or the dialog will keep selecting the wrong variant.
- Creation flow currently returns only `selected.Kind`; variant data must not be lost before item creation.
- If the generic `Signal` entry is removed, existing user muscle memory changes. Keeping it as a default numeric Signal may be less disruptive.
- Documentation files are named by persisted widget type, so all variants should continue to use `Signal.md` / `Signal.help.md` unless a separate picker-only description mechanism is introduced.

## Notes
- The simplest stable design is a preset descriptor with fields such as key, display name, summary, body caption, sample value, unit, and format.
- Avoid spreading preset constants across multiple files unless tests or creation flow need a shared type.
- Prefer preserving the existing generic `Signal` entry as the default numeric-style Signal if product direction is uncertain.
