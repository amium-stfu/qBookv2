# IMPLEMENTATION HANDOFF

## Goal
Improve function picker readability by separating persisted function references from user-facing display labels. The Action Rules picker should show concise labels such as `YAML / new_workflow` and `Python / application_explorer_1 / raw / start_loop`, while saving stable registry references internally.

## Scope
- Included:
  - Add a display-label layer for `RunFunction` options.
  - Keep persisted `FunctionName` values as stable registry references.
  - Keep old `declarative:<name>` references resolvable.
  - Prefer `yaml:<name>` as the new user-facing/persisted YAML prefix if the change can be made with safe alias support.
  - Shorten Python picker labels by removing technical duplicated segments such as `python:python-env`.
  - Keep Python execution and `InvokePythonFunction` behavior unchanged.
  - Update docs/help to describe reference vs display label.
  - Add focused tests for reference resolution and picker display mapping where practical.
- Not included:
  - Removing `InvokePythonFunction`.
  - Migrating existing persisted files in bulk.
  - Changing Python runtime target registration.
  - Changing Python file/template structure.
  - Adding Workflows.
  - Adding typed argument schemas.

## Starting Point
- `FunctionRegistry` returns `FunctionCatalogEntry` objects with `Reference`, `Name`, `Kind`, `SourceIdentifier`, and `DisplaySource`.
- `RunFunction` picker currently uses `ObservableCollection<string>` / `IReadOnlyList<string>` and binds `SelectedItem` directly to `FunctionName`.
- Therefore the UI shows the persisted technical reference directly:
  - `declarative:new_workflow`
  - `python:python-env:main.application_explorer_1:raw:start_loop`
- `FunctionRegistry.TryGetEntry(...)` currently matches `candidate.Reference` directly.
- Python references are built in `PythonFunctionRegistryProvider` as `python:{normalizedTargetPath}:{functionName}`.
- Declarative references are built in `DeclarativeFunctionRegistryProvider` as `declarative:{fileName}`.

## Tasks
1. Introduce a small picker option model for interaction rule function pickers:
   - suggested name: `FunctionPickerOption`
   - fields:
     - `Reference`
     - `DisplayText`
   - `ToString()` should return `DisplayText` if existing Avalonia ComboBox binding cannot easily use `DisplayMemberBinding`.
2. Update `RunFunction` option loading:
   - get runnable entries from `FunctionRegistry`
   - create options with readable `DisplayText`
   - persist `Reference` into the rule `FunctionName`
   - display `DisplayText` in the ComboBox
3. Keep existing Python-specific `InvokePythonFunction` picker as string-based unless changing it is required for shared UI code.
4. Update configured-row picker binding:
   - existing row stores `FunctionName` as reference
   - ComboBox selection must map between selected option and row `FunctionName`
   - if the saved reference is missing from options, show a fallback option using the raw reference so users do not lose data
5. Update new-rule picker binding:
   - selecting a display option should write the option `Reference` to `NewFunctionName`
   - default selection should still choose the first available runnable function
6. Add display label generation:
   - YAML/declarative:
     - display: `YAML / <name>`
     - avoid `declarative:` in UI
   - Python:
     - display: `Python / <owner-or-widget> / <script-or-channel> / <function>`
     - acceptable compact fallback: `Python / <relative-target-path> / <function>`
     - remove duplicated technical prefix like `python-env` when it does not add meaning
   - missing/unknown:
     - display raw reference with a clear marker only when needed, e.g. `Missing / <reference>`
7. Add alias support for YAML references:
   - `yaml:<name>` should resolve to the same entry as legacy `declarative:<name>`.
   - existing `declarative:<name>` persisted values must continue to run.
   - If new persistence is changed to `yaml:<name>`, ensure `FunctionRegistry.TryGetEntry(...)` resolves both forms.
8. Decide whether to change declarative provider references now:
   - preferred if low-risk: new declarative entries use `yaml:<name>`.
   - required compatibility: old `declarative:<name>` still resolves.
   - acceptable if lower risk: keep internal reference `declarative:<name>` for now and only change display labels; document `yaml:` as a follow-up.
9. Improve Python display names without weakening stable references:
   - keep persisted reference stable enough for dispatch.
   - derive display text from `SourceIdentifier`, `Name`, and target path segments.
   - do not depend on mutable display labels for execution.
10. Update docs:
   - `RunFunction` saves stable references but shows friendly labels.
   - YAML/declarative compatibility.
   - Python labels are shortened for display only.
11. Add focused tests:
   - `TryGetEntry("yaml:name")` resolves YAML function.
   - `TryGetEntry("declarative:name")` still resolves YAML function.
   - picker option display text differs from persisted reference.
   - missing saved reference remains visible as fallback option.

## Implementation Order
1. Inspect current ComboBox binding limits and decide whether to use `SelectedItem` with option objects or `SelectedValue`.
2. Add the function picker option model in the smallest existing view-model location.
3. Implement display label generation from `FunctionCatalogEntry`.
4. Update `RunFunction` option loading for configured rows.
5. Update `RunFunction` option loading for the new-entry row.
6. Add fallback option behavior for saved references not currently in the registry.
7. Add YAML alias resolution in `FunctionRegistry`.
8. Optionally change new YAML references from `declarative:` to `yaml:` only after alias tests are in place.
9. Update docs.
10. Build and manually verify the picker UI.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation/error messages must be English.
- Do not introduce new dependencies.
- Do not change Python execution behavior.
- Do not break existing persisted `declarative:` or Python references.
- Do not persist user-facing display text as the executable reference.
- Keep the change local to registry/picker presentation unless a minimal binding helper is required.

## Acceptance Criteria
- The `RunFunction` picker no longer shows raw references like `python:python-env:...` as the normal display text.
- YAML functions display with a clear label such as `YAML / new_workflow`.
- Python functions display with a concise relative label such as `Python / application_explorer_1 / raw / start_loop`.
- Selecting a friendly option still persists a stable registry reference in `FunctionName`.
- Existing saved `declarative:<name>` rules still resolve and execute.
- If `yaml:<name>` is introduced, it resolves and executes the same YAML function.
- Missing saved references remain visible in the picker and are not silently cleared.
- Existing `InvokePythonFunction` picker behavior is unchanged.
- Build and focused tests pass.

## Verification
- Build:
  - `dotnet build HornetStudio.sln --no-restore`
  - If a running app locks output, use the established project-level or separate `OutDir` fallback.
- Tests:
  - Run focused editor tests if available.
  - Add/run alias and picker-display tests if feasible.
- Manual:
  - Open Interaction Rules.
  - Choose `RunFunction`.
  - Confirm function list shows friendly labels.
  - Select a YAML function and save/reopen; confirm selection remains readable and executable.
  - Select a Python function and save/reopen; confirm selection remains readable and executable.
  - Open an existing rule saved with `declarative:<name>`; confirm it still resolves.

## Out of Scope
- Bulk migration of existing YAML files.
- Removing `declarative:` support.
- Removing `InvokePythonFunction`.
- Changing Python target registration format.
- Changing `Functions` widget layout beyond any optional display-label reuse.
- Workflow implementation.

## Risks / Watchouts
- Avalonia ComboBox selection can become brittle if object equality is not handled. Use stable reference matching when rebuilding option lists.
- Do not rely on `ToString()` alone if saving the selected item might write the display text instead of the reference.
- Python target paths may contain delimiters. Prefer resolved registry entry metadata for display; avoid parsing references when possible.
- Widget names can change. Display labels may use friendly names, but persisted references must stay stable.
- If switching new YAML references to `yaml:`, make alias behavior bidirectional before changing persistence.
