# IMPLEMENTATION HANDOFF

## Goal

Convert `LogControl` into a widget-owned log control with no public log configuration properties. Every `LogControl` must create and display its own `ProcessLog` and publish writable level input items (`debug`, `info`, `warning`, `error`, `fatal`) that external controls can write to.

## Scope

Included:

1. Remove `TargetLog` and `AutoCreateLog` from the public `LogControl` configuration surface.
2. Generate the owned log path internally from the widget identity and folder/page context.
3. Keep legacy layouts loadable even when they contain `TargetLog` or `AutoCreateLog`.
4. Ensure the owned `ProcessLog` and its level input items are published through the existing `ProcessLogRuntime`.
5. Update matching widget documentation.
6. Add or adjust focused tests for load/save and runtime log item behavior.

Not included:

1. Redesigning the visual log UI.
2. Changing `ProcessLog` storage format or Serilog configuration unless required by this task.
3. Changing Monitor action semantics except where they target the owned log input items.
4. Refactoring unrelated widget properties.

## Starting Point

- `EditorLogControl` currently resolves a log through `LogItem?.GetAutoCreatedLogPath()` and still falls back to `Logs.Host`.
- `FolderItemModel` currently exposes public `TargetLog` and `AutoCreateLog` properties.
- `LogViewWidget.yaml` currently contains `Properties.TargetLog` and `Properties.AutoCreateLog`.
- `MainWindowViewModel` still saves `TargetLog` for `LogControl` and exposes a `TargetLog` choice in the editor dialog.
- `ProcessLogRuntime` already creates a `ProcessLog` and level input items, so the core runtime mechanism exists.
- `LogWindow` currently uses `EditorLogControl` with `TargetLog = "Logs.Host"` and must be treated as a special host-log viewer case, not as the page `LogControl` ownership model.

## Tasks

1. Define the owned `LogControl` runtime path rule.
2. Remove `LogControl` public log configuration from editor dialogs, YAML templates, and save output.
3. Keep legacy YAML loading tolerant of `TargetLog` and `AutoCreateLog`, but do not let those values control new `LogControl` behavior.
4. Update `EditorLogControl` to always ensure and resolve the owned log for page widgets.
5. Preserve or isolate the host-log window behavior so it can still display `Logs.Host`.
6. Update widget docs and descriptions.
7. Add focused tests for persistence and runtime publishing.
8. Run build/tests according to solution rules.

## File-Level Changes

- `src/HornetStudio.Editor/ViewModels/MainWindowViewModel.cs` -> remove `TargetLog` from `LogControl` save output and property dialog bindings. Keep legacy load tolerant.
- `src/HornetStudio.Editor/Persistence/LayoutDocument.cs` -> review `TargetLog` and `AutoCreateLog` usage. Keep backward-compatible document members only if required, but ensure new save output does not emit them for `LogControl`.

- `src/HornetStudio/LogWindow.axaml.cs` and `src/HornetStudio/LogWindow.axaml` -> verify host-log display still has an explicit path mechanism or use a dedicated host-log viewer path that does not reintroduce public `LogControl` properties.
- `src/HornetStudio.Host/Logging/ProcessLogRuntime.cs` -> keep existing runtime item publishing; adjust only if the owned path contract requires it.
- `src/HornetStudio.Host/HostRegistries.cs` -> keep write forwarding to `ProcessLogRuntime` and add tests if behavior changes.

- `src/HornetStudio.Host.Tests/Program.cs` -> add or update tests for level input item writes producing `ProcessLog` entries.
- `src/HornetStudio/docs/widgets/LogControl.md` -> update summary documentation.

## Implementation Order

1. Add or identify an internal owned-log path helper on `FolderItemModel`.
2. Update `EditorLogControl` to call the owned-log helper and `ProcessLogRuntime.EnsurePublished(...)`.
3. Remove `TargetLog` binding from the `LogControl` editor dialog.
4. Remove `TargetLog` and `AutoCreateLog` from new `LogControl` YAML/template output.
5. Keep legacy load paths tolerant without making legacy values active for new behavior.
6. Verify `LogWindow` separately and prevent its host-log needs from leaking back into page `LogControl`.
7. Update docs.
8. Add tests.
9. Build and run focused tests.

## Technical Constraints

- Follow existing Avalonia, MVVM, and registry patterns.
- Do not add new dependencies.
- Use existing path normalization helpers such as `TargetPathHelper` and `HostPathSegmentNormalizer`.
- Keep code, comments, XML documentation, UI text, file names, and technical identifiers in English.
- Chat responses remain German.
- Preserve backward compatibility for loading existing layouts.
- Avoid broad refactoring outside the `LogControl` ownership problem.

## Acceptance Criteria

1. New `LogControl` instances expose no editable `TargetLog` or `AutoCreateLog` properties.
2. New saved layouts do not emit `TargetLog` or `AutoCreateLog` for `LogControl`.
3. `LogControl` always displays its own generated `ProcessLog`.
4. The owned log publishes writable input items:
   - `debug`
   - `info`
   - `warning`
   - `error`
   - `fatal`
5. Writing a string to a level input item creates a matching log entry.
6. Existing layouts containing legacy `TargetLog` or `AutoCreateLog` still load without errors.
7. The host log window still displays `Logs.Host` or an equivalent explicit host log without changing page `LogControl` semantics.
8. Documentation describes the owned-log behavior and no public log properties.

## Verification

Run, when restore is already available:

```powershell
dotnet build HornetStudio.sln --no-restore
```

Run focused self-tests if affected:

```powershell
dotnet src/HornetStudio.Host.Tests/bin/Debug/net9.0/HornetStudio.Host.Tests.dll
```

Run editor tests if the touched persistence/editor behavior is covered there:

```powershell
dotnet src/HornetStudio.Editor.Tests/bin/Debug/net9.0-windows/HornetStudio.Editor.Tests.dll
```

Manual verification:

1. Add a `LogControl` in the editor.
2. Confirm no `TargetLog` property is offered.
3. Save the layout and confirm no `TargetLog` or `AutoCreateLog` appears for that `LogControl`.
4. Write to the generated `debug/info/warning/error/fatal` items and confirm entries appear in the widget.
5. Open the host log window and confirm host logs still display.

## Out of Scope

- Monitor widget redesign.
- Global log routing redesign.
- Replacing Serilog.
- Changing visual styling beyond what is necessary to keep current UI behavior intact.
- Removing legacy document members if doing so would break existing layouts.

## Risks / Watchouts

- `LogWindow` currently relies on the same `EditorLogControl` type and may need an explicit host-log path path that is separate from page `LogControl`.
- Runtime path generation must avoid collisions between widgets with the same name in different folders/pages.
- Renaming a widget may create a new owned log path; decide whether this is acceptable or whether `Id` should be part of the internal path.
- If `TargetLog` is removed too aggressively from document models, old layouts may fail to deserialize.
- `ProcessLogRuntime.TryWriteInputEntry(...)` clears written values; tests should account for that side effect.
