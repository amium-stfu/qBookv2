# IMPLEMENTATION HANDOFF

## Goal
Fix widget runtime registry paths so CsvLogger, SqlLogger, and related widget runtime items publish only valid lowercase snake_case item paths.

## Scope
- Fix CsvLogger and SqlLogger runtime root paths and runtime child item names.
- Align logger runtime filtering and lookup code with the new canonical path.
- Tighten path identity validation so editor-side validation matches the Amium item path rule.
- Add focused regression tests for logger runtime paths.
- Review nearby widget runtime path publishers for obvious remaining PascalCase path segments.

Do not rename persisted widget `Type` values, C# classes, namespaces, public APIs, or user-facing labels solely for snake_case.

## Starting Point
- CsvLogger currently throws:
  `System.ArgumentException: Item path segment 'Loggerruntime' must use lowercase snake_case and start with a letter.`
- `FolderItemModel.GetLoggerRuntimeBasePath()` builds `studio.<folder>.Loggerruntime.<item>`.
- `MainWindowViewModel.GetAllowedLoggerRuntimePrefixes()` and `IsAllowedLoggerRuntimePath()` also use `Loggerruntime`.
- CsvLogger and SqlLogger runtime controls use PascalCase item names such as `Record`, `OutputPath`, `IsRecording`, and `LastFile`.
- Similar snake_case fixes already exist for CircleDisplay and Python application runtime paths.

## Tasks
1. Change the canonical logger runtime root to `studio.<folder>.logger_runtime.<logger_widget>`.
2. Change CsvLogger runtime child item names to lowercase snake_case:
   - `record`
   - `output_path`
   - `is_recording`
   - `last_file`
   - `status`
3. Apply the same runtime child item names to SqlLogger.
4. Update logger runtime target filtering to detect and allow `.logger_runtime.`.
5. Preserve compatibility only where needed for reading or cleanup of legacy `Loggerruntime` paths.
6. Tighten editor path identity validation so a valid identity must start with a lowercase letter.
7. Add regression tests for CsvLogger and SqlLogger runtime paths.
8. Review other widget runtime publishers for direct PascalCase path construction and either fix or document out-of-scope risks.
9. Update matching widget documentation for CsvLogger and SqlLogger if widget code changes.

## Implementation Order
1. Update the shared logger runtime path builder in `FolderItemModel`.
2. Update CsvLogger and SqlLogger constants to snake_case runtime item names.
3. Update logger runtime filtering in `MainWindowViewModel`.
4. Add tests proving CsvLogger and SqlLogger paths resolve and contain no PascalCase child names.
5. Tighten identity validation and update existing tests accordingly.
6. Run targeted editor tests.
7. Update widget documentation if runtime paths are mentioned.
8. Run the solution build.

## Technical Constraints
- Chat and summaries must be in German.
- Code, comments, XML documentation, UI text, filenames, and technical identifiers must be English.
- Do not introduce new dependencies.
- Keep changes scoped to widget runtime path correctness.
- Preserve existing public/persisted widget `Type` values such as `CsvLoggerControl` and `SqlLoggerControl`.
- Use existing `TargetPathHelper` and host registry patterns.
- Prefer backward-compatible resolution or cleanup for legacy paths, but publish only canonical snake_case paths.

## Acceptance Criteria
- Creating or previewing CsvLogger no longer throws a `Loggerruntime` path exception.
- Creating or previewing SqlLogger no longer throws the same class of path exception.
- Logger runtime paths use `logger_runtime`.
- Runtime child items use lowercase snake_case names.
- Editor identity validation rejects names that start with a digit or underscore.
- Existing CircleDisplay, Python application, UDL, and broker runtime path tests still pass.

## Verification
- `dotnet run --project src\HornetStudio.Editor.Tests\HornetStudio.Editor.Tests.csproj --no-restore`
- `dotnet build HornetStudio.sln --no-restore`
- Manual smoke test: open widget picker or page editor, create CsvLogger and SqlLogger, confirm no snake_case exception.

## Out of Scope
- Renaming persisted widget types or documentation files derived from persisted `Type` values.
- Broad refactoring of the widget system.
- Changing user-visible labels such as `CsvLogger`, `SqlLogger`, or `Start`.
- Reworking broker, UDL, Python, or log publishing beyond obvious path-safety regressions found during review.

## Risks / Watchouts
- Existing layouts or scripts may reference old logger runtime paths. If required, support legacy lookup/cleanup, but do not keep publishing legacy paths.
- `TargetPathHelper.NormalizeConfiguredTargetPath(...)` can convert mixed-case input to snake_case, but direct `new ItemModel(...)` calls with PascalCase names can still throw before normalization helps.
- `UiPublisher.Publish(...)` and log paths such as `Logs.Host` may still be structurally risky; only include them if the implementation review confirms they affect this widget issue.
- Tightening identity validation may expose older layouts with invalid names. Do not silently rename existing persisted names unless a migration strategy is explicitly approved.
