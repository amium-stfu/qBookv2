# IMPLEMENTATION HANDOFF

## Goal
Extend declarative `SetValue` function steps so the argument can either be a literal value or be read from another item path at execution time.

## Scope
Included:
- Add a persisted `SetValue` value-source path alongside the existing literal value.
- Support YAML like `value: 10` for literals and `valueFrom: some.item.path` for runtime item-source reads.
- Wire the second `SetValue` picker in the Function Editor to select the source item path for the argument.
- Execute `SetValue` by reading the current value from `valueFrom` when it is configured, then writing that resolved value to `target`.
- Preserve existing literal `value` behavior and existing YAML compatibility.
- Update function/widget documentation and focused tests.

Explicitly not included:
- No broad workflow schema redesign.
- No UI mode selector unless required to make `value` versus `valueFrom` unambiguous.
- No release/version/tag work.
- No Delay unit feature.
- No IfThenElse expand/collapse feature.

## Starting Point
- `FunctionSetValueStepDefinition` currently has `Target` and `Value` only.
- YAML codec currently parses and serializes only `target` and `value` for `SetValue`.
- `FunctionStepEditorRow` currently exposes `Target` and `Value`.
- The Function Editor row already has two list buttons in the `SetValue` layout:
  - first picker: target item path
  - second picker: intended source item path for the argument
- `FunctionExecutionEnvironment` has:
  - `SetValueAsync`
  - `ResolveConditionSourceValueAsync`
- `PageItemModel` builds the execution environment and already has condition source resolution behavior that can likely be reused or factored locally for `valueFrom`.

## Tasks
1. Extend `FunctionSetValueStepDefinition` with a nullable/empty `ValueFrom` or `ValueSourcePath` property.
2. Update clone, editor-row conversion, and definition-building logic to preserve both literal values and source paths.
3. Update `FunctionDefinitionCodec`:
   - parse optional `valueFrom`
   - keep existing `value` parsing
   - serialize `valueFrom` when configured
   - avoid emitting ambiguous combinations where both `value` and `valueFrom` are meaningful unless the chosen model explicitly allows one to be inactive.
4. Update `WorkflowEditorDialogWindow.axaml.cs` so the second `SetValue` picker selects the argument source path, not the target path.
5. Update `WorkflowEditorDialogWindow.axaml` binding if a separate editor-row property is added for the source path.
6. Update execution:
   - if `valueFrom` is configured, resolve the current item value before writing
   - write the resolved value to the `target`
   - keep literal `value` behavior unchanged when `valueFrom` is empty
7. Add or update focused tests in `src/HornetStudio.Editor.Tests/Program.cs` for codec, editor conversion, and execution behavior.
8. Update `Functions` and `WorkflowWidget` Markdown documentation files that describe `SetValue`.

## Implementation Order
1. Add the new SetValue source-path model property.
2. Update codec parsing/serialization and tests for literal-only and `valueFrom` YAML.
3. Update editor row conversion/building so existing literal steps still round-trip.
4. Wire the second picker to set the source-path property.
5. Update execution environment and runtime execution to resolve `valueFrom`.
6. Update `PageItemModel` environment construction and source value resolution.
7. Add execution tests for resolved source values and unresolved source errors.
8. Update documentation.
9. Run focused tests/build.

## Technical Constraints
- Chat responses must be German.
- Code, comments, XML documentation, UI text, file names, and technical identifiers must be English.
- User-visible validation and error messages must be English.
- Use existing Avalonia, workflow, and item-resolution patterns.
- Do not introduce new dependencies.
- Keep the change localized to declarative function SetValue handling.
- Public classes and public properties need XML documentation.
- Prefer named parameters for new C# calls where argument meaning is not obvious.

## Acceptance Criteria
- Existing YAML with `value` continues to load, edit, save, and execute as before.
- YAML with `valueFrom` loads and executes by reading the current source item value.
- Saving a source-backed SetValue preserves `valueFrom`.
- The second SetValue picker writes/selects the argument source path, not the target path.
- Runtime errors are explicit when `valueFrom` cannot be resolved.
- Ambiguous `SetValue` state is handled predictably and documented.
- Tests cover literal SetValue, valueFrom SetValue, codec round-trip, and unresolved valueFrom behavior.

- If the complete solution is not locked by a running app/debugger, run `dotnet build HornetStudio.sln --no-restore`.
- Manually verify in the Function Editor that:
  - first SetValue picker selects the target
  - second SetValue picker selects the argument source path

## Out of Scope
- Delay unit selector.
- IfThenElse collapse/expand.
- Changes to monitor interaction rules unless implementation reveals shared code that must be updated for compilation.
- Release versioning, tags, NuGet metadata, and notice files.

## Risks / Watchouts
- If both `value` and `valueFrom` are stored, the executor must have a deterministic precedence. Prefer `valueFrom` when non-empty and document it, or validate that only one is active.
- The current execution callback `SetValueAsync` receives the whole step. If value resolution happens inside `FunctionExecutor`, the environment may need a new callback such as `ResolveValueSourceAsync`.
- If value resolution happens inside `PageItemModel.ExecuteRunFunctionSetValueAsync`, tests should still prove the behavior at the executor/environment boundary.
- The UI needs to make the selected source-path state distinguishable from a literal string; otherwise a path typed manually could be misinterpreted.
- Existing preserved rows and clone paths must not drop `valueFrom`.
