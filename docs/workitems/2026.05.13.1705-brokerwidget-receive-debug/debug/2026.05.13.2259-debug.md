# DEBUG REPORT

## Problem
ItemClient write-back diagnostics flooded logging and could partially block the UI during fast broker write-back traffic. UI writes could also appear to have no effect after broker write-back traffic.

## Expected Behavior
UI writes should remain responsive, and external write requests should be applied even when the MQTT source client id is missing.

## Actual Behavior
The log repeatedly showed `ItemWriteRequestMessage` entries with `sourceClientId=<none>` being ignored as `selfEcho`.

Later UI-write diagnostics showed the UI path successfully writing the local `write` parameter, but no external reaction followed:

```text
[SignalWrite] result=ok item=main.signal_2_set writeTarget=studio.main.enhanced_signals.filtered_1.set writeParam=write value=1000000 (Double) registryUpdated=True
[SignalWrite] state item=main.signal_2_set writeTarget=studio.main.enhanced_signals.filtered_1.set writeParam=write current=1000000 (Double) targetSameAsWriteTarget=True
```

## Error Messages / Logs
Relevant pattern:

```text
[ItemClientWriteBack] messageType=ItemWriteRequestMessage brokerPath=studio.main.enhanced_signals.filtered_1.set parameter=write value=10 (Single) sourceClientId=<none> ... action=ignored reason=selfEcho
```

## Relevant Code
`HostItemBrokerWriteBackClient.IsEcho(...)` accepted `_lastAppliedValues` matches before checking whether the message was actually produced by the same client. That allowed a missing source client id to still be treated as a self echo when the cached value matched.

`HostRegistries.Data.UpdateProperty(...)` returns `true` when the converted value already equals the current value, but it does not raise `ItemChanged` in that case. For normal state this is correct. For `write`, repeated equal values are commands and still need to notify the ItemClient publisher.

## Attempted Fixes
- Fix 1 -> Added broad information-level decision diagnostics. Result: wrong direction; log volume was too high on retained/live ItemClient write-back traffic.
- Fix 2 -> Excluded `ItemWriteRequestMessage` from echo suppression and moved decision diagnostics to Debug. Result: write-request behavior fixed, but Debug-level logging still ran too often and could affect UI responsiveness.
- Fix 3 -> Guarded write-back decision diagnostics behind `HornetStudio.ItemClient.WriteBackDiagnostics`. Result: diagnostics are fully disabled by default and only run when explicitly enabled.
- Fix 4 -> Added targeted forced property change notification for UI writes to the `write` parameter, including repeated equal values. Result: implemented and validated with a regression test.

## Current Hypothesis
- `read`, metadata, and retained state messages can remain echo-suppressed.
- `ItemWriteRequestMessage` is a command and must not be dropped by value-cache echo suppression, especially when `sourceClientId` is missing.
- UI writes to `write` must emit a registry change event even when the value equals the previous `write` value, otherwise the ItemClient publisher does not publish a repeated command.
