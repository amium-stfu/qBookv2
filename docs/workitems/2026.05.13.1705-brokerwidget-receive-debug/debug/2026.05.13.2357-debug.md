# DEBUG REPORT

## Problem
UI writes to an enhanced signal still start a feedback loop where the chart alternates between the enhanced/UI-domain value and the inverse-adjusted source-domain value.

## Expected Behavior
Writing through the UI should publish the local command once, forward one inverse-adjusted source command, and then stop. External live writes should still be accepted, including repeated equal command values.

## Actual Behavior
External manipulation is stable, but UI manipulation starts repeated oscillation. The attached chart shows rapid alternating values after the UI write begins.

## Error Messages / Logs
No new log excerpt was provided with this report. The chart shows the same symptom as the previous `100` / `1000` write-domain oscillation.

## Relevant Code
`ItemClientControl.HostItemBrokerPublisher` publishes local `write` property changes as non-retained live MQTT property updates.

`HostItemBrokerWriteBackClient` receives non-retained `.../write` topics as `ItemWriteRequestMessage`. Because shared MQTT topics do not carry `SourceClientId`, the write-back path could not identify its own published live `write` echo.

`HostRegistries.Data.UpdateProperty(..., forceChangeNotification: true)` is required for real repeated `write` commands, but it also makes a self-published live echo dangerous: the echo writes the same local value, forces a registry notification, and the publisher sends the same command again.

## Attempted Fixes
- Fix 1 -> Ignored retained/property-style `write` state, omitted `write` from retained snapshots, and forced notifications for broker write requests. Result: retained/state replay was fixed, but the UI-triggered live command echo loop remained.
- Fix 2 -> Added a pending own-write echo tracker shared by the ItemClient publisher and write-back client. The publisher records each self-published live `write` command; write-back consumes exactly the next matching `ItemWriteRequestMessage` echo before applying it locally. Result: focused regression passes and external repeated same-value commands still notify.

## Current Hypothesis
- The remaining loop was a live self-echo loop, not a retained-state loop.
- UI writes trigger it because the enhanced signal first writes the UI-domain `write` value and then forwards the inverse-adjusted source-domain `write` value. Both are published to shared MQTT topics and both were previously re-applied by the same ItemClient as fresh commands.
