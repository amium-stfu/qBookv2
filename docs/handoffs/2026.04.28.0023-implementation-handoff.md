# IMPLEMENTATION HANDOFF

## Goal
Prevent physical camera devices from being activated during application startup when no `CameraControl` widget needs a camera feed.

## Scope
- Included:
  - Make camera device activation lazy.
  - Keep camera discovery available for camera selection UI.
  - Start the physical camera only when a `CameraControl` with a configured `CameraName` subscribes or otherwise explicitly needs frames.
  - Stop the camera when no widget still uses it, if the existing design can support that with a small scoped change.
- Not included:
  - Redesigning the widget system.
  - Adding new camera libraries or dependencies.
  - Changing persisted layout format.
  - Adding new user-facing features beyond preventing unwanted startup activation.

## Tasks
1. Inspect `src/HornetStudio.Host/HostRegistries.cs` and remove eager device start from default camera registration.
2. Refactor `src/HornetStudio.Host/Helpers/WindowsCameraFrameSource.cs` so its constructor stores camera metadata only and does not call `StartDevice()`.
3. Add a small explicit activation path to `WindowsCameraFrameSource`, for example starting on first `FrameAvailable` subscription or through a focused method on `ICameraFrameSource` if the interface already supports a suitable pattern.
4. Preserve supported resolution discovery without starting the camera feed if possible. If AForge requires a `VideoCaptureDevice` instance for capabilities, create the instance but do not call `Start()` during discovery.
5. Ensure `SetResolution(...)` does not start an unused camera. It should apply the requested resolution only after the source is active, or store it for the next start.
6. Update `EditorCameraControl` subscription cleanup so the camera can stop when the widget unsubscribes, if reference counting or an equivalent scoped ownership mechanism is added.
7. Replace silent catch blocks touched by this work with minimal meaningful logging where appropriate, following existing `HostLogger` patterns.
8. Verify that a project without any `CameraControl` no longer turns on the camera at startup.
9. Verify that adding/configuring a `CameraControl` still starts the selected camera and receives frames.
10. Run `dotnet build HornetStudio.sln --no-restore`.

## Technical Constraints
- Follow existing C# and Avalonia patterns.
- Do not introduce new dependencies.
- Keep public API changes minimal; if `ICameraFrameSource` must change, update all implementations/usages in the same scoped change.
- Chat and planning are German, but code, comments, XML docs, UI text, file names, and technical identifiers remain English.
- Public classes and public methods touched by the implementation need XML documentation if missing or outdated.
- Avoid broad refactoring of `HostRegistries` or widget rendering.

## Notes
- Current root cause: `HostRegistries` calls `TryInitializeDefaultCamera()` in its static constructor. That method registers each Windows camera by constructing `WindowsCameraFrameSource`. The `WindowsCameraFrameSource` constructor immediately calls `StartDevice()`, which calls `VideoCaptureDevice.Start()`. This activates the physical camera during startup even when no `CameraControl` is present.
- `FolderEditorControl.axaml` creates an `EditorCameraControl` inside every item template, but it is hidden unless `IsCameraControl` is true. That is probably not the direct startup activation cause because `EditorCameraControl` subscribes only when `CameraName` is non-empty.
- `GetCameraOptions()` in `MainWindowViewModel` uses `HostRegistries.Cameras.GetAll()` for the property UI. This should continue to list available cameras after the change.
- Be careful with resolution options: `GetCameraResolutionOptions(...)` reads `source.SupportedResolutions`. The implementation should not require an active streaming device just to populate this list.
